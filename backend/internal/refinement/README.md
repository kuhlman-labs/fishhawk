# backend/internal/refinement

Intake refinement flow (ADR-052): natural-language brief → structured, validated epic/children draft → preview + approval gate → filing executor that turns the approved draft into real tracker items.

## Drafting (ADR-052 option A, E34.1 / #1592)

This package is the **drafting half** of the intake refinement flow — it turns a natural-language brief into a structured, validated epic/children draft and persists it, but **NEVER files** (ADR-052 decision 1: no provider write — the E34.3 filing executor is what turns an approved draft into provider work items).

- **Domain model** (`refinement.go`): `EpicDraft{Epic EpicSpec, Children []ChildDraft}` where each child carries summary/proposal/done-means, `AcceptanceCriteria []string`, `Labels []string`, and `DependsOn []int` (1-based sibling ordinals — children have no issue numbers at draft time; the E34.3 filing executor remaps ordinals to real #numbers).
- **Validation:** `EpicDraft.Validate()` enforces the structural rules (non-empty epic summary/scope, ≥1 child, per-child non-empty summary+proposal and ≥1 acceptance criterion) then reuses `campaign.Assemble` over a synthetic 1..N ordinal numbering, so a dangling edge surfaces wrapped `campaign.ErrDanglingDependency` and a cycle/self-edge `campaign.ErrCycle` — the SAME rules the campaign assembler applies at filing time.
- **Agent** (`agent.go`): the `Drafter` (an `Inferencer` whose signature matches `(*claudecode.Client).Inference` structurally + the repo's `workmgmt.Conventions`) builds a closed-field-set prompt (literal JSON Schema + prose enumeration of every inner-element shape, #1543/#1567); `Draft(ctx, sessionID, brief)` invokes → decodes → validates.
- **Decoding** (`decode.go`): `DecodeDraft` is the prose-prefix/fence-tolerant strict decoder — strip a markdown fence, extract the first balanced top-level object via a string-literal-aware brace scan (E32.9), then strict-decode with `DisallowUnknownFields` (closed field set; nested children too).
- **Rendering** (`render.go`): `RenderChild`/`RenderEpic`/`RenderDraft` route every item through the SAME `workmgmt.Apply` conventions pipeline the filing path uses (byte-compatible by construction — children as the v0 `feature` type), appending the `Depends on: #N` marker (provider format, draft ordinals) ONLY for children with edges. The epic folds out-of-scope into its Scope section (the epic skeleton has no out-of-scope section) and seeds `ExistingNumbers` (sentinel `[0]`→1) so numbering fails closed (#1265).
- **Persistence:** `Repository`/`postgres.go` store the draft as a JSONB row keyed by a refinement session id in `refinement_drafts` (migration `0046`, sqlc `db/`) — NOT the artifact table, which is stage-keyed and no run/stage exists at drafting time.

## Preview + approval gate (E34.2 / #1593)

E34.2 adds the HTTP surface + the persistence and gate semantics that make approval load-bearing.

- `gate.go` is the pure, repository-agnostic seam:
  - `ContentHash(EpicDraft)` = sha256 over `json.Marshal` of the DECODED struct (deterministic field order sidesteps Postgres jsonb re-serialization).
  - `ResolveState(drafts, decisions)` derives `SessionState` (`awaiting_approval`/`approved`/`rejected`) — a decision counts ONLY when it targets the session's LATEST revision AND its pinned hash equals the recomputed hash, so an edit-after-approval structurally re-gates and a hash mismatch fails closed (`drifted`).
  - `ApprovedDraft(drafts, decisions)` is the ONE place the E34.3 filing executor resolves a fileable draft, returning the pinned revision or the typed sentinels `ErrNotApproved` / `ErrDraftDrifted`.
- Every edit is a NEW immutable `refinement_drafts` revision (append-only; a decision never mutates a draft) and every decision is an append-only `refinement_decisions` row (migration `0047`) pinning `draft_id` + `draft_content_hash`. `refinement_drafts.origin` (`brief`/`amendment`/`edit`) makes the brief-amendment budget countable from rows.
- HTTP handlers in `backend/internal/server/refinement.go` (all behind `write:approvals`, the intake analogue of the plan-approval gate): `POST /v0/refinement/sessions` (draft from a brief), `GET …/{session_id}` (derived state + full filing preview via `RenderDraft` + the wave DAG via `EpicDraft.Waves()` + decision history), `PATCH …/draft` (brief-amendment re-draft bounded like `revise_plan`, or a direct strict-decoded field edit), `POST …/decision` (approve/reject with a required reason).
- **Audit ordering (gpt-5.5's #1593 HIGH concern, binding):** every edit/decision appends its `refinement_draft_edited` / `refinement_draft_approved` / `refinement_draft_rejected` entry on the GLOBAL audit chain (session_id/draft_id/content_hash, +reason for decisions) via `AppendGlobalChained` BEFORE persisting (durable-before-state-change) — an audit-append failure is a `500` and NOTHING persists, so no gate action is ever unaudited.

## Filing executor (E34.3 / #1594, ADR-052 filing half)

`filing.go` is the FILING half `refinement.go`'s package doc reserves: it turns an approved, hash-pinned draft into real tracker items over the EXISTING conventions/provider pipeline.

- `render.go` exports the byte-compat builders `FilingRequestForEpic`/`FilingRequestForChild` — the ONE seam the E34.2 preview and the executor share (preview passes the sentinel epic/parent refs + nil depends_on; the executor passes the real epic number, `#N` parent ref, and `#N`-resolved depends_on refs), so "what was previewed is what files" by construction.
- `FilingOrder(draft)` flattens `EpicDraft.Waves()` into wave-by-wave, ascending-within-wave filing order so every `depends_on` target (including a legal forward sibling edge) is filed — its real `#number` known — before any dependent.
- `ExecuteFiling(ctx, draft, repo, repository, fileItem)` files the epic first (ordinal 0), then each child in wave order via a `FileItemFunc` the server wires to `applyAndFileWorkItem` (so draft children ride exactly the hand-filed pipeline — #1616 label completeness, #1269 epic-number discovery, board placement, sub-issue linking — and `deriveEpicTitleVar` short-circuits because `{epic}` is passed explicitly), recording each success in `refinement_filed_items` (unique `(draft_id, ordinal)`) IMMEDIATELY after the provider create so a re-invoke resumes at the first unfiled ordinal.
- **Idempotency ledger** (migration `0048`): `refinement_filing_sessions` (PK `draft_id`) pins the target repo at first invoke (a different repo on re-invoke → typed `ErrFilingRepoMismatch`) and its `completed_at` flip is the session-closing state change; a completed session replays the recorded result with no writes.
- **Concurrent-duplication guard (gpt-5.5 binding condition):** `Repository.WithFilingLock` holds a session-level `pg_advisory_lock` keyed on the draft UUID on a dedicated pooled connection (the pgtest bootstrap precedent) for the executor's whole body, so two concurrent `POST …/file` calls cannot both observe an ordinal as unfiled — the second blocks, then observes the first's recorded progress.
  Record-after-file + the unique constraint remain the residual at-least-once backstop (the external create is not transactional with Postgres; a crash between create and record re-files exactly one item on resume).
- `VerifyFiledEpic` asserts the filed epic round-trips: the provider's `EpicChildren` child set matches the recorded numbers, then `campaign.Assemble` (which already fails closed on `DroppedEdges`/dangling/cycle) over the result.
- HTTP handler `backend/internal/server/refinement_file.go`: `POST /v0/refinement/sessions/{session_id}/file` (behind `write:approvals`, no new scope) — `ApprovedDraft` gate → resolve installation (run-absent operator path, like `handleFileWorkItem`) → `ExecuteFiling` → verify → the `refinement_filing_completed` global-chain audit entry BEFORE `CompleteFilingSession` (durable-before-state-change; audit failure → `500`, session stays open for a retry).

## Criteria quality pre-check (ADR-052, E34.5 / #1596)

`precheck.go::EvaluateDraftCriteria(EpicDraft) CriteriaPrecheck` runs the acceptance-criteria rule set over EVERY drafted child at intake.

- It is the intake sibling of the plan-gate acceptance pre-check: both dispatch through the ONE exported `plan.EvaluateAcceptanceCriteria` in `backend/internal/plan/acceptance_check.go` (the deterministic rule set extracted there from `backend/internal/server/acceptance_precheck.go`, which now delegates via `plan.EvaluateAcceptanceCriteria` in `runAcceptancePrecheck` — `AcceptanceFinding` is a type alias of `plan.AcceptanceFinding`), so a rule added to the shared set applies to intake AND the plan gate at once with no second copy to drift.
- Each child's `[]string acceptance_criteria` maps into a synthetic `plan.Verification`: criterion text becomes both the `AcceptanceCriterion.ID` and `.Statement`, so an empty/whitespace criterion fires `empty_id` and two identical fire `duplicate_id`; `Blocking` nil = the schema default `true`; `Source` empty so the provenance rules cannot fire on the intake shape.
- The epic's `out_of_scope` prose is the only out-of-scope justification at draft time, so a non-empty epic `out_of_scope` supplies `Verification.OutOfScope` for EVERY child — the justified escape hatch that suppresses `no_blocking_criterion` across the draft.
- **Advisory-only:** `CriteriaPrecheck{NeedsAttention, Children []ChildCriteriaCheck}` where a child's `NeedsAttention` is set ONLY by `no_blocking_criterion` (other findings render but do not flag), and the draft-level `NeedsAttention` is the OR over children.
- Surfaces as the `criteria_precheck` block on `GET /v0/refinement/sessions/{id}` (`backend/internal/server/refinement.go`, computed fresh per read via `refinement.EvaluateDraftCriteria(latest.Draft)`) and rides the `RefinementSession` mirror onto the `fishhawk_draft_epic` open/preview/edit/decide results.
- It INFORMS the operator's decide verdict but **never blocks approval or filing** (`EpicDraft.Validate` no longer hard-rejects a criteria-less child precisely so the child can REACH the preview for the gate to be informational rather than a 422).
