# backend/internal/budget

Budget decision cores (ADR-030): the advisory periodic workflow budget (`budget_alert`), the whole-run tripwire (`run_budget_exceeded`), and the decomposition concurrency cap — all pure evaluators; wiring lives in `backend/internal/server` and `serve.go`.

## Advisory periodic budget (`budget_alert`, #690)

Detector: `budget.go` — pure `PeriodRange(period, now, loc)` + `Evaluate(spent, limit, warnAt, period, now, loc) Decision`, no repository dependency: the caller supplies an already-summed period spend and gets back `{PeriodStart, Spent, Limit, Fraction, Over, WarnCrossed}`. Period boundaries are timezone-aware (weekly resets Monday 00:00 local, monthly the 1st 00:00) in a `*time.Location`.

- **Persistence:** `run.queries.sql::SumWorkflowCostInRange` (sqlc) + thin `postgresRepo.SumWorkflowCostInRange(ctx, repo, workflowID, from, to)` wrapper — `COALESCE(SUM(cost_usd_total),0)` over `runs` matching `(repo, workflow_id)` with `created_at` in `[from, to)`. Like `AddRunCost` it is **not** on `run.Repository`; the trace handler asserts the optional `runCostSummer` capability at runtime so non-cost test fakes need no stub.
- **Wiring:** trace ingest at `trace.go::checkBudgetAlerts` (called from `recordCost` **after** `AddRunCost` so the period sum already includes the current bundle — a check before the increment would fire the crossing one bundle late).
  It parses the run's cached `workflow_spec`, and for each `Workflow.Budgets` entry with `enforcement: advisory` (the empty value defaults to advisory; only `blocking` is excluded — blocking enforcement is an admission-time gate, a separate scope item) sums the current calendar period and evaluates.
  On a ladder-rung crossing it calls `emitBudgetAlert` → a `budget_alert` audit entry (`{workflow_id, repo, period, period_start, spent, limit, fraction, warn_at, tier, enforcement}`) **plus** an advisory issue comment via `issuecomment.Notifier.NotifyBudgetAlert`.
- **Escalating tier ladder (#1371):** the tier is the single source of truth `budget.Tier(d, warnAt, ackMultiple, pageMultiple)` — `ok < warn < over < ack_required < page` — where `over` is `fraction >= 1`, `ack_required` is `fraction >= ackMultiple` (default 2x), and `page` is `fraction >= pageMultiple` (default 3x); an over-budget signal escalates by configured multiples instead of saturating at `over` forever.
  A non-positive or inverted (`page <= ack`) multiple falls back to `budget.DefaultAckMultiple`/`DefaultPageMultiple` so a zero-value Config never collapses every fraction into `page`.
- **De-dup** is keyed on `(workflow_id, period_start, tier)` (tier ∈ `warn`/`over`/`ack_required`/`page`) via an in-memory `ListAll(category="budget_alert")` scan, so each tier fires once per period.
- **Calibration override (#1371):** `effectiveBudgetLimit(b)` returns `FISHHAWKD_BUDGET_LIMIT_OVERRIDE_USD` (flag `--budget-limit-override-usd`, default `0` = use the spec `limit_usd`) when `>0`, applied centrally on a copy of the budget in BOTH the alert and display paths so the dogfood weekly limit can be calibrated WITHOUT editing `.fishhawk/workflows.yaml` (a forbidden implement-stage commit path).
  The ack/page multiples are `server.Config.BudgetAckMultiple`/`BudgetPageMultiple` (flags `--budget-ack-multiple`/`--budget-page-multiple`, env `FISHHAWKD_BUDGET_ACK_MULTIPLE`/`FISHHAWKD_BUDGET_PAGE_MULTIPLE`, defaults `2.0`/`3.0`).
- **Plan-approval acknowledgment gate (#1371):** `approvals.go::checkPeriodicBudgetTier` (a pre-Submit sibling of `checkPlanBudget`) refuses a plan-stage approve with `422 periodic_budget_requires_ack` once the advisory budget escalates to `ack_required`/`page`, unless the comment carries `--ack-budget` — recording `plan_violates_periodic_budget` on refusal, `plan_periodic_budget_tier_acknowledged` on the ack; fail-open on every read failure.
- **Timezone:** `FISHHAWKD_BUDGET_TIMEZONE` (flag `--budget-timezone`, default `UTC`; an unresolvable IANA name falls back to `time.UTC` with a WARN via `serve.go::resolveBudgetLocation` so a missing zoneinfo never crashes startup), wired onto `server.Config.BudgetLocation`.
- `checkBudgetAlerts` is warn-only and best-effort throughout — nil repo/spec-parse/empty-budgets/sum-failure/notifier-failure all WARN-log and never unwind the upload. Period spend is summed from `runs.cost_usd_total`, which undercounts `known_usage=false` invocations (#685), so an advisory budget can under-warn (the documented ADR-030 estimate caveat).

### Display read endpoint (#693)

`GET /v0/runs/{run_id}/budget` (`backend/internal/server/budget_status.go::handleGetRunBudget` → `runBudgetStatus`) reuses the **same** period-range → `SumWorkflowCostInRange` → `Evaluate` chain (extracted into the shared `trace.go::evaluateWorkflowBudget` helper that `checkBudgetAlerts` also calls) to return the FIRST workflow budget's current-period status.
The response is `{period, period_start, limit_usd (effective), spent_usd, fraction, warn_at, tier (ok|warn|over|ack_required|page), ack_required (omitempty bool), enforcement}`, with the empty enforcement value normalized to `advisory` on the wire.

**Display-only** — it never gates or blocks (the admission/alert/emit-dedup paths are untouched); a workflow with no budget returns 200 with an empty object (no `period` key). The MCP server's `fishhawk_start_run` / `fishhawk_run_stage` / `fishhawk_get_run_status` tools fetch it best-effort (`apiClient.GetRunBudget`) and surface it as an omitempty `budget` block so the operator sees spend-vs-limit every stage instead of relying on the easily-missed advisory issue comment.

### Cache-efficiency display endpoint (ADR-044 slice 3 / #1352)

`GET /v0/runs/{run_id}/cache-efficiency` (`backend/internal/server/cache_efficiency.go::handleGetRunCacheEfficiency` → `runCacheEfficiency`) reads the run's `cost_recorded` ledger (same `AuditRepo.ListForRunByCategory` call `sumRunTokens` uses) and folds it via the pure `backend/internal/cost/efficiency.go::AggregateCacheEfficiency` into a per-run metric.
The metric is `{fresh_input_tokens, cache_read_tokens, cache_write_tokens, output_tokens, cache_read_ratio (cache_read/(cache_read+fresh), 0-guarded), reuse_factor (cache_read/max(cache_write,1)), gross_read_savings_usd, write_penalty_usd, net_savings_usd, stages[]}` with a per-source breakdown keyed on the entry's `source` (`plan_review` / `implement_review`; the runner stage-agent path carries no `source` and buckets as `agent`).

Dollar figures reuse slice 1's `pricing` package (`Cost` / `CostWithCache`) per entry — an unknown model contributes 0 savings while its tokens still count toward the ratios. **Display-only** — no write path, no migration; derived purely from existing audit data, returns 200 with an empty object when the run has no cost data. `fishhawk_get_run_status` fetches it best-effort (`apiClient.GetRunCacheEfficiency`) and surfaces it as an omitempty `cache_efficiency` block so the operator can see and maximize cache-hit usage.

## Per-run budget tripwire (`run_budget_exceeded`, #653 / ADR-030)

Evaluator: `runbudget.go` — pure `EvaluateRun(costUSD, tokens, maxUSD, maxTokens) RunDecision`, no repository dependency: the caller supplies the run's rolled cost + cumulative tokens and gets back `{Over, Dimension (usd|tokens), CostUSD, MaxUSD, Tokens, MaxTokens}`. A non-positive ceiling disables that dimension (default 0 = off); US$ is checked first so it wins when both breach.

This is ADR-030's third budget axis — a **whole-run safety rail** ("the system can stop itself") distinct from the per-workflow periodic budgets (`budget_alert` above / `run_rejected_budget`) and the rate-anomaly `spend_alert`.

- **Config:** `server.Config.MaxRunUSD` / `MaxRunTokens` — a global operator backstop, NOT a workflow-v0 schema field (so no schema soak/version-advertising discipline).
- **Wiring:** trace ingest at `trace.go::checkRunBudget`, called from `handleShipTrace` right after `recordCost` rolls the bundle's cost into `runs.cost_usd_total` and under the same raw-variant guard (so it evaluates once per stage bundle). US$ is enforced against the rolled `cost_usd_total`; tokens against `sumRunTokens` (input+output summed from the run's `cost_recorded` ledger — there is no dedicated runs column).
- **Decomposition-family aggregation (E24.6 / #1146):** before evaluating, `checkRunBudget` sums `cost_usd_total` and tokens across the whole decomposition **family** (`trace.go::familyRuns` — the family root, i.e. the run itself when `DecomposedFrom` is nil else `*DecomposedFrom`, plus every child via `ListRuns(DecomposedFrom)`), so a wide fan-out can't blow the run budget even when no single child is over its own ceiling.
  A non-decomposed run's family is just itself, so its figure is byte-identical to the pre-#1146 single-run behavior. `familyRuns` is best-effort: a parent `GetRun` or children `ListRuns` failure degrades to the single run (WARN) rather than unwinding the upload.
- **On breach** it HALTS the run via the cancel transition (terminal state `cancelled`, SYSTEM actor — a protective stop, non-retryable, so a runaway run is deliberately not auto-redriven, unlike a failed-A/C), appends a `run_budget_exceeded` audit entry (`{dimension, cost_usd_total, max_run_usd, tokens_total, max_run_tokens, terminal_state}`), and **short-circuits before the stage-advancement block** so no further stage is dispatched.
- Best-effort and audit-only (no Notifier surface — see `docs/issue-comment-surfaces.md`): the fast-path returns immediately when both ceilings are 0, and a get-run / transition / audit-write failure WARN-logs without unwinding the upload.

## Decomposition concurrency cap (resolution; `effective_max_parallel`, E24.6 / #1146)

Decision core: `parallel.go` — pure `ParallelDecision(requested, cap) ParallelResult` with `0 = unlimited` semantics (cap≤0 → no throttle; cap<requested → `Allowed=cap, Capped=true`), the CONTRACT the E24.3 (#1143) dispatch throttle consumes — `orchestrator.go::DispatchDecomposedChildren` calls `ParallelDecision(len(pending)+len(in-flight), cap)` and dispatches `Allowed - in-flight` headroom children.

- **Resolver:** `spec.Workflow.EffectiveMaxParallel(globalDefault)` — the per-workflow `decomposition.max_parallel` knob (`backend/internal/spec/spec.go`, `Workflow.Decomposition`) wins when `>0`, else the global default; `0 = unlimited`, kept consistent with `ParallelDecision`.
- **Global default:** `server.Config.MaxParallelChildren`, wired from `FISHHAWKD_MAX_PARALLEL_CHILDREN` (flag `--max-parallel-children`, default `0`) in `serve.go` onto both the Config and `orchestrator.Orchestrator.MaxParallelChildren`.
- **Schema:** `workflow.decomposition.max_parallel` (version `0.6`, additive).
- **Resolution + surfacing** live in `orchestrator.go::fanoutIfDecomposed` → `resolveEffectiveMaxParallel` (parse the run's cached `workflow_spec`, look up the workflow, resolve the cap; best-effort — absent/unparseable spec or workflow-not-in-spec degrades to the global default with a WARN): the resolved value is logged and recorded as `effective_max_parallel` in the `plan_decomposed` audit payload.
- **Mechanism vs. enforcement:** #1146 RESOLVES and SURFACES the cap; E24.3 (#1143) ENFORCES it — `DispatchDecomposedChildren` dispatches pending children (ascending `slice_index`) up to the cap via the runner-kind-aware `Advance` path, wired at fanout (initial), event-driven refill in `maybeAdvanceDecomposedParent`, and the `childcompletion` sweeper backstop (see `backend/internal/orchestrator`).
