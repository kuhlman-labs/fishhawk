# backend/internal/planreview

Verdict/authority types and audit payloads for the plan-review and implement-review agent invocations (ADR-027).

## Plan-review-agent invocation + authority resolution (ADR-027 / #560)

`review.go` — `Verdict` type (approve / approve_with_concerns / reject), `AuthorityMode` enum, `ResolveAuthority(ReviewersConfig) AuthorityMode`, `PlanReviewedPayload` audit struct.

- Prompt: `backend/internal/prompt/prompt.go::buildPlanReview` — review-agent prompt template (structured verdict JSON, re-planning forbidden).
- Invocation: `backend/internal/server/plan.go::handleShipPlan` invokes review agents after artifact validation when `reviewers.agent>0`; appends `plan_reviewed` audit entries; gates stage advancement when authority is gating and any verdict is reject.
- MCP surface: `fishhawk_get_plan` surfaces `Reviews[]` in `GetPlanOutput` decoded from those audit entries (including synthesized `verdict: "skipped"` entries from `plan_review_skipped`), plus a `plan_review_status` lifecycle summary (#600) backed by the `plan_review_started` dispatch proxy.
- `runPlanReviews` emits one `plan_review_started` audit entry at dispatch (only when `agent>0` AND a reviewer is wired), ordered before every `plan_reviewed` entry, so the MCP `review_status` can report `pending` for a configured-but-not-yet-complete review and `fishhawk_await_review` can block on it.
- **Nil-reviewer hardening (#574)**: `handleCreateRun` (`backend/internal/server/runs.go`) rejects agent-gated plan stages (`agent>0, human==0`) with `400 plan_reviewer_unconfigured` + a `run_rejected_misconfigured` global-chain audit entry when no `PlanReviewer` is wired; advisory-mode (`agent>0, human>0`) instead emits a `plan_review_skipped` entry in `runPlanReviews` and continues.

Full flow documented in `docs/ARCHITECTURE.md` §4.2.

## Implement-review-agent invocation (ADR-027 impl 2/2 / #561)

- Prompt: `backend/internal/prompt/prompt.go::buildImplementReview` — diff-review prompt variant (`prompt.Build("implement_review", …)`, split at `ImplementReviewSplitMarker`).
- Invocation: `backend/internal/server/trace.go::runImplementReviews` + the hook in `advanceStageAfterTrace` — runs after `dispatched→running` and before the terminal transition, guarded by `stage.Type == implement`; extracts the diff via `bundle.ExtractDiff` (works in local `--no-pr` mode); appends `implement_reviewed` audit entries (payload `ImplementReviewedPayload` in this package's `review.go`).
- A gating reject (`agent>0, human==0` + any `reject`) fails the stage to category-B before the terminal advance.
- **scope.files drift is flag-only** (ADR-027 Q6): the prompt emits a `{category:"scope"}` concern, never an auto-reject.
- Reviewer resolution is shared with plan-review via `resolveStageReviewers(runRow, stageType)` (`backend/internal/server/plan.go`). Nil-reviewer + per-invocation-error behavior mirrors `runPlanReviews` exactly (`implement_review_skipped` with `reviewer_not_configured`).
- MCP surface: `fishhawk_get_run_status` surfaces `ImplementReviews[]` in `GetRunStatusOutput`, plus `plan_review_status` + `implement_review_status` lifecycle summaries (#600).
- `runImplementReviews` emits an `implement_review_started` audit entry at dispatch (agent>0-AND-wired, ordered before every `implement_reviewed` entry) backing the `pending` state for `review_status` / `fishhawk_await_review`.

Full flow documented in `docs/ARCHITECTURE.md` §4.2.1.

## Review-agent invocation — full server-side contract (ADR-027)

Relocated verbatim from `docs/ARCHITECTURE.md` §4.2/§4.2.1 (which now carries the distilled invariants).
This is the authoritative long-form contract for plan- and implement-review invocation: adapter wiring and
precedence, env vars, review budgets, subprocess hardening, capability degradation, prompt caching, reviewer
context surfaces, and per-audit-category payload shapes.

### Plan-review-agent invocation

When a stage's `reviewers.agent > 0` is configured in the workflow spec, the backend invokes that many review agents inline inside `handleShipPlan` (in `backend/internal/server/plan.go`), after the plan artifact is stored and validated but before the stage transitions to `awaiting_approval`. Each review agent receives the plan artifact and the originating issue body via the
`plan_review` prompt (`buildPlanReview` in `backend/internal/prompt/prompt.go`) and is instructed to return only a structured verdict JSON — re-planning and proposing edits are explicitly forbidden.

**Invocation flow:**

1. Plan artifact POSTed to `POST /v0/runs/{id}/plan?stage_id=…`.
2. Backend verifies signature, validates against `standard_v1`, stores in `artifacts`.
3. If `stage.reviewers.agent > 0`: spawn N review-agent invocations synchronously, each receiving the plan artifact + issue body via `prompt.Build("plan_review", …)`.
4. Self-review guard: if the review agent's model matches `plan.GeneratedBy.Model`, log WARN (ADR-027 §5 — no hard block in v0).
5. For each review response, decode the `ReviewVerdict` JSON, append a `plan_reviewed` audit entry (category string `"plan_reviewed"`, payload `PlanReviewedPayload` from `backend/internal/planreview/review.go`).
6. Authority check: if `ResolveAuthority(reviewers) == AuthorityGating` (i.e. `agent>0 && human==0`) and any verdict is `reject`, the stage stays in its current state rather than advancing to `awaiting_approval`.
7. Otherwise `handleShipPlan` drives the plan stage's terminal advance (`advancePlanStageTerminal`): `running → awaiting_approval`, or `running → succeeded` + an orchestrator `Advance` for a gateless plan stage.

**Plan-stage advancement invariant** (#603): a plan stage's terminal transition is driven by a valid `standard_v1` plan artifact landing via the plan-upload handler (`handleShipPlan`), **not** by trace upload alone. The runner ships both trace variants before the plan, so `advanceStageAfterTrace` (`backend/internal/server/trace.go`) leaves a plan stage in `running` until
`planArtifactExists` confirms a plan is stored — guarding against a gated plan stage reaching `awaiting_approval` with nothing to review when the later plan-ship fails `standard_v1` validation. On an invalid plan, `handleShipPlan` fails the stage category-B (`running → failed`) and calls `advanceAfterFailure` so the run walks to terminal `failed` rather than stranding. The
transition is idempotent (same-state re-application is a repo-layer no-op), so a future plan-first upload ordering — where the trace handler finds the artifact and advances first — does not double-fault. When `ArtifactRepo` is unwired (minimal config), `planArtifactExists` returns true so the gate is a no-op and the prior trace-driven advance is preserved.

**Push-and-open-pr implement advancement invariant** (#742): mirroring the plan-stage gate, an implement stage that will open a PR has its terminal transition driven by the **`/pull-request` upload** (`handleShipPullRequest`), **not** by trace upload alone. The runner ships the trace *before* it commits/pushes/opens the PR, so advancing to `awaiting_approval` on trace upload
would strand the run at the review gate with a null `pull_request_url` if the PR-open step then fails (the b6811dc9 zombie). The runner stamps `push_and_open_pr` in the bundle manifest (`bundle.ManifestData` / `bundle.Manifest`, both `omitempty`, lockstep across the two modules — see the trace-bundle manifest row in §10) for every standalone implement stage (not `--no-pr`, not
a decomposed child). `advanceStageAfterTrace` (`trace.go::pushAndOpenPRGated`) then leaves the stage in `running` when the flag is set **and** the bundle carries a non-empty `git_diff` (a commit + PR-open will follow) — emitting the `runtime_observed` calibration row before the early return, since the bundle timing is only available at trace time. The `/pull-request` upload
becomes the authoritative terminal driver: the **success** body advances `running → awaiting_approval`/`succeeded` (+ orchestrator `Advance` for a gateless stage) after stamping `run.pull_request_url` (`advanceImplementStageAfterPR`); the **failure** body (`{outcome:"failed", category, reason}`, `failPullRequestStage`) fails the stage (category `C` retryable / `B` re-scope) +
advances + appends a `pull_request_failed` audit entry, so the run never reaches review with a null PR. The runner reports that failure from the implement post-processing block (`main.go::reportPullRequestFailure`, gated on the same standalone-implement condition) when `openPRAndShipArtifact` errors — unifying #728's `ErrCommitWouldNotCompile` (category B) into this report
path. An **empty** diff is the no-changes carve-out: the agent made no edits, opens no PR, and never POSTs `/pull-request`, so `pushAndOpenPRGated` returns false and the stage advances via the trace handler as before (gating it would hang the stage in `running`). Non-implement stages and a false flag (older bundles, `--no-pr`) are byte-identical to the prior trace-driven
advance. **Decomposed-child push gate (#771)**: a decomposed child opens no PR (the parent run opens one consolidated PR, ADR-032) but still commits + pushes onto the shared branch *after* the trace ships, so the same zombie hole applied — a child push failure left the child stage at terminal `succeeded` with no code on the shared branch, which the childcompletion sweeper would
then consolidate into a PR silently missing that child's work. The runner stamps a separate `push_to_shared_branch` manifest flag (mutually exclusive with `push_and_open_pr`) for decomposed children; `trace.go::childPushGated` mirrors `pushAndOpenPRGated` to leave the child stage in `running`, and the `/pull-request` upload drives the terminal transition via a new
`{outcome:"pushed", branch, head_sha, base_sha}` success variant (`succeedChildPushStage`: running → terminal + `child_pushed` audit entry, no PR artifact) or the existing `{outcome:"failed"}` variant on a push failure (category `C` retryable via the #698 parent-park). **Fix-up push gate (#794)**: a fix-up re-dispatch (#762) opens no new PR (it commits onto the EXISTING PR
branch *after* the trace ships) but the same zombie hole applied in its worst form — a fix-up stamped *neither* `push_and_open_pr` *nor* `push_to_shared_branch`, so it terminalized at trace upload *before* its commit/push, and a push or `#728` compile-gate failure was silently swallowed: the stage stayed `succeeded`, the implement re-review approved an *unlanded* diff, and the
#788 fix-up recovery never fired (the stage never went terminal-failed). The runner now stamps a third mutually-exclusive `push_fixup` manifest flag for fix-up passes; `trace.go::fixupPushGated` mirrors `childPushGated` to leave the fix-up stage in `running` (the advisory implement re-review still fires at trace time on the bundle diff — only the TERMINAL transition is
deferred), and the `/pull-request` upload drives the terminal transition via a new `{outcome:"fixup_pushed", branch, head_sha, base_sha}` success variant (`succeedFixupPushStage`: running → terminal + `fixup_pushed` audit entry, no PR artifact, idempotency-guarded on `(stage_id, head_sha)`) or the existing `{outcome:"failed"}` variant on a push/compile-gate failure — which now
genuinely reaches `failPullRequestStage → maybeRecoverFixupFailure`, restoring the run to its pre-fix-up review gate with the original mergeable PR intact. **Dangling-PR close on a gating implement-review reject (#877)**: a gating agent implement-review (`reviewers.agent>0`, `human==0`) runs SYNCHRONOUSLY during the raw-trace upload
(`advanceStageAfterTrace::runImplementReviews`) and fails the implement stage category-B with reason `implement_review_rejected: …` (hoisted to the const `implementReviewGatingRejectReason`) BEFORE the runner — which has no view of that verdict — opens its PR and POSTs `/pull-request`. By then the stage is terminally failed, so `handleShipPullRequest` records the PR artifact +
`pull_request_opened` audit + URL backfill (the honest open) but `advanceImplementStageAfterPR` no-ops (stage not running), leaving the rejected change with an open PR that will never merge. After the backfill and BEFORE the lineage guard, the handler detects that exact state — implement + `failed` + category-B + the `implementReviewGatingRejectPrefix` reason prefix (same
source of truth as the trace failure site; other category-B sources are excluded) — and calls `closePRAfterGatingReject`: it posts a short explanatory PR comment, closes the PR via the new `githubclient.ClosePullRequest` (`PATCH .../pulls/{number}` body `{"state":"closed"}`, mirroring `ForceUpdateRef`), and writes a `pull_request_closed_after_review_reject` audit entry (system
actor, payload `{run_id, stage_id, artifact_id, pr_number, pr_url, failure_reason}`). Fail-open throughout: GitHub unconfigured, a nil `installation_id`, an unparseable repo, or a close error WARNs and skips — the stage is already failed, so a failed close must never 500 the handler. Advisory reviews dispatch detached and never reject-fail the stage, so they never reach this
state and are byte-identical (PR-open timing for the advisory flow is unchanged; ADR-036/#876 enforces at the landing gate, not at PR-open).

**Authority resolution** (`backend/internal/planreview/review.go::ResolveAuthority`):

| `reviewers.agent` | `reviewers.human` | Mode |
|---|---|---|
| > 0 | == 0 | `gating` — agent rejection blocks stage advancement |
| > 0 | > 0 | `advisory` — agent verdict recorded but human gate is authoritative |
| == 0 | any | `gateless` — no agent review; today's default |

**Verdict types** (closed set per ADR-027): `approve`, `approve_with_concerns`, `reject`.

**Schema-constrained verdicts (structured outputs, #1324).** The two first-class review backends request a schema-guaranteed verdict body rather than free-text JSON, so `planreview.DecodeVerdict`'s fence-strip + invalid-escape sanitizer is now the documented FALLBACK, not the primary decode path. Both backends build the constraint from a single source of truth —
`planreview.VerdictSchema()` (`backend/internal/planreview/schema.go`), the JSON Schema derived from the `ReviewVerdict`/`Concern`/`ConcernResolution` shapes and the `AllVerdicts`/`AllConcernSeverities` constant lists in `review.go` that the verdict/severity validation path also consumes (so the schema enums cannot silently diverge from the validated set; a reflection drift
test pins it). The **Anthropic** SDK adapter sets `MessageNewParams.OutputConfig.Format = json_schema` with that schema; the **codex** adapter writes `VerdictSchemaJSON()` to a temp file and passes `codex exec --output-schema <file>`. **claudecode**'s `claude` CLI exposes no response-schema flag, so it stays on the `DecodeVerdict` fallback — as does any error or unconstrained
response from the constrained backends. `DecodeVerdict` remains fully exercised and unchanged.

**Audit category `plan_reviewed`**: one entry per review-agent invocation, payload shape `PlanReviewedPayload{ReviewerKind, ReviewerModel, Authority, Verdict, Concerns[], FreeForm}`. Surfaced by `fishhawk_get_plan` (MCP tool) in `GetPlanOutput.Reviews[]`.

**Audit category `plan_review_started`** (#600): a single best-effort entry emitted by `runPlanReviews` at dispatch — **only** on the branch where a reviewer will actually run (`reviewers.agent > 0` AND a `PlanReviewer` is wired), never for the gateless (`agent == 0`) or nil-reviewer (`plan_review_skipped`) branches. Payload shape `ReviewStartedPayload{ConfiguredAgents,
Authority}` (`backend/internal/planreview/review.go`). It is emitted synchronously before the per-reviewer loop, so its audit sequence always precedes every `plan_reviewed` entry under both gating (synchronous) and advisory (detached) authority. This is the MCP-readable proxy that lets a consumer distinguish *configured + running* (a `started` entry with no terminal entry) from
*none configured* (no entry at all) without re-fetching the workflow spec — the basis for the `review_status` = `pending` state surfaced by `fishhawk_get_run_status` / `fishhawk_get_plan` and waited on by `fishhawk_await_review` (see the MCP row in §10 "Where to look"). **Completeness is count-gated (#1127)**: the payload's `ConfiguredAgents` is the threshold the MCP
`review_status` resolves `complete` against — `review_status` stays `pending` until `landed_terminal >= configured_agents` (ANY terminal kind — `plan_reviewed` / `plan_review_failed` / `plan_review_skipped` — counts toward the round), the SAME completeness rule `checkPlanReviewSettled` uses for the approval gate, so `await_review` / `get_run_status` no longer report a partial
heterogeneous result (a poll that catches the window after the first of N reviewers landed but before the rest finish now reports `pending`, not `complete` with one verdict). An absent or non-positive `ConfiguredAgents` (old/malformed payload) degrades to the prior complete-on-first-verdict predicate.

**Production wiring** (`backend/internal/anthropic/`, wired in `backend/cmd/fishhawkd/serve.go`):

`serve.go` wires `cfg.PlanReviewers`, a `server.ReviewerSet` (resolved by the pure `resolvePlanReviewers` helper so the wiring seam is unit-testable; every configured adapter logs at startup). Since #955 ALL configured backends are concurrently available — the set replaces the old single precedence-selected `cfg.PlanReviewer`:

- **`ReviewerSet.Default()`** serves the bare `reviewers.agent: N` count form by the historical four-way precedence (unchanged behavior — the default adapter is invoked N times):
  1. **`FISHHAWKD_ANTHROPIC_API_KEY` set** → `anthropic.NewReviewer` (production SDK adapter, #572). Wins over everything below.
  2. **else `FISHHAWKD_ENABLE_LOCAL_CLAUDE_REVIEWER=true`** → `claudecode.NewReviewer` (local-mode subprocess adapter, #575). See **Local-mode wire-up** below.
  3. **else `FISHHAWKD_ENABLE_CODEX_REVIEWER=true`** → `codex.NewReviewer` (Codex subprocess adapter, #844). See **Codex reviewer wire-up** below.
  4. **else** → `Default()` returns nil; review invocations are skipped (the #574 degradation below).
- **`ReviewerSet.For(provider, model, reasoningEffort)`** resolves one entry of the heterogeneous `reviewers.agents` list (#955, see `docs/spec/workflow-v0.md` § Plan reviewers) to its adapter, constructed with the spec-requested model (empty → that provider's deployment default model) and, for codex only, the spec-requested `reasoning_effort` (#1493, workflow-v1.x). The codex
  branch resolves effort through the per-reviewer ladder **`FISHHAWKD_CODEX_REASONING_EFFORT` (deployment default) < `reviewers.agents[i].reasoning_effort`** (`server.ResolveReviewerReasoningEffort`); an empty spec value falls back to the env default exactly as before. The anthropic/claudecode branches take no reasoning-effort parameter and ignore it. Adapters are stateless
  config-struct wrappers (`NewReviewer` does no I/O), so per-resolve construction is cheap. An unconfigured provider errors, and as of #1495 that is treated as a **deployment capability gap** (the provider's `FISHHAWKD_ENABLE_*` / API-key gate is off), NOT a policy override of the spec: run creation no longer hard-fails on it. Both gating and advisory stages **degrade
  per-reviewer** with a capability-framed `plan_review_skipped` / `implement_review_skipped` entry (reason `reviewer_unavailable`, carrying `provider` + `optional`) while the loop continues — loud (ERROR) when the reviewer's `optional:false` (default), quiet (INFO) when `optional:true`. This is deliberately distinct from a genuine reviewer error (`*_review_failed`). Only the
  coarse "no reviewer backend wired at all" case still hard-fails run creation (see **Reviewer-capability degradation** below).

The review loops (`runPlanReviewLoop` / `runImplementReviewLoop`) iterate a resolved `[]reviewerInvocation` (one per declared reviewer) instead of repeating a single adapter; per-invocation audit entries, reviewer-cost recording, and the per-invocation self-review guard are unchanged. The effective agent count everywhere (authority table, skip/started payloads) is
`spec.ReviewersConfig.AgentCount()` — `len(agents)` when the list is present, else the bare `agent` integer.

**Local-mode wire-up** (`backend/internal/claudecode/`, #575). The local-mode adapter is the dogfood sibling of `anthropic.Reviewer`: instead of provisioning an API key, it spawns the operator's existing `claude` CLI as a subprocess (`claude --print --output-format json --model <model> -p <prompt>`), inheriting `os.Environ()` so the operator's subscription auth or
already-present `ANTHROPIC_API_KEY` is reused with zero new plumbing. It decodes the CLI's single JSON envelope, reads the `result` field, and validates the verdict against the same closed set (`approve` / `approve_with_concerns` / `reject`) as the SDK adapter.

**Subprocess hardening (`backend/internal/procgroup`, #1805).** Both local reviewer adapters (`claudecode`, `codex`) apply `procgroup.Harden(cmd, killGrace)` to the review `cmd` before `cmd.Output()`. This makes the EXISTING size-aware review budget deadline (#747, applied as a `context` deadline at the server call site) actually terminate a wedged reviewer instead of hanging:
`cmd.Output()` does not return until every writer of the child's stdout pipe closes it, so a reviewer CLI that forks a grandchild (a shell exec, an MCP server) inheriting that write-end can keep `Output()` blocked long after the direct child is killed — the latent defect behind #1805, where a hung codex reviewer ran 60+ minutes past its budget until the operator killed it by
hand. `Harden` sets `SysProcAttr.Setpgid` (child leads its own process group), overrides `cmd.Cancel` to SIGKILL the whole group (`kill(-pgid)`, reaping the grandchild so the pipe writer closes), and sets `cmd.WaitDelay` (~5s grace) so `Output()` force-closes the parent-side pipe fd even if a group member escaped the kill. Because a `WaitDelay`-forced return may be a
NON-`*exec.ExitError` (e.g. `ErrWaitDelay`), both adapters HOIST the `ctx.Err()==DeadlineExceeded` timeout classification ABOVE the `errors.As(err, &exitErr)` type gate, so the terminal `implement_review_failed{Timeout:true}` label the server records from `invocationCtx.Err()` stays correct on every return path. This is the backend-reviewer analog of the group-kill technique
the runner's codex EXECUTOR adapter already uses (`runner/internal/agent/codex`). **No new implement-review timeout knob was added** (Option A per #1805): the single shared `ReviewBudget` is retained; the fix is purely the subprocess-termination mechanism.

The retry budget (`FISHHAWKD_PLAN_REVIEW_MAX_RETRIES`, see the env-var table) is wired through a post-construction `(*claudecode.Reviewer).SetMaxRetries` call rather than `claudecode.Config.MaxRetries` directly: `NewClient` normalises a zero `MaxRetries` to 1, so passing the env value through the constructor could never express *disable*. `serve.go` therefore constructs the
reviewer, then calls `SetMaxRetries(*planReviewMaxRetries)` to assign the env-resolved value past that normalisation. The result: unset → 1 (one retry, unchanged from #620's code default), an explicit `0` → retry disabled (single attempt), `N` → N+1 attempts, and a negative value clamps to 0.

Large-plan expectation (#606): small plans review in ~60–90s, but large backend `standard_v1` plans (10+ files) need the wider budget — the old 60s default timed them out and the agent verdict never landed. Because advisory plan review is async/detached (#584), a longer budget blocks nothing: it only lengthens the worst-case `pending` window that re-polling
`fishhawk_get_run_status` (the poll-authoritative terminal source, #879) surfaces via `poll_interval_seconds`, and that `fishhawk_await_review` (default 360s since #878, the optional non-stranding convenience block per ADR-036 #874) already returns gracefully (#600). The override above is redundant with the new 300s code default but kept in the copy-pasteable block for
explicitness. This is the genuine timeout-on-large-plans mode only; the separate ~1s immediate-crash mode is owned by #620.

The review model (`claude-sonnet-4-6`) deliberately differs from the plan **author's** typical default (Opus): the plan-review prompt's self-review guard keys on the model identifier, so reviewing with a different model than authored keeps the self-review WARN meaningful rather than firing on every plan. The adapter returns the configured `FISHHAWKD_LOCAL_CLAUDE_MODEL` verbatim
as the verdict's model identifier (the CLI envelope does not reliably echo the model), keeping that guard deterministic.

Accepted local-mode trade-offs vs. the SDK adapter: (a) softer verdict-shape guarantee — the subprocess has tool access via the `claude` CLI and only the prompt forbids repo grepping, so a non-compliant agent degrades to a per-call `reject` rather than silent corruption; (b) self-review WARN noise if the review model matches the author model (mitigated by the recommended config
above); (c) no prompt caching — the full prompt is sent as one `-p` argument with no system/user split; (d) ~1–2s process-spawn overhead per review.

**Codex reviewer wire-up** (`backend/internal/codex/`, #844). The Codex adapter is the OpenAI sibling of `claudecode.Reviewer` and the reviewer analog of the runner's Codex **executor** (`runner/internal/agent/codex`, #840/#842): it spawns `codex exec --json --skip-git-repo-check <prompt>` for inference-only review and reads one JSON event per line from stdout. The verdict body
is the final `item.completed` line whose nested item is an `agent_message` (its `text`); per-turn token usage is summed from each `turn.completed` line's `usage` block and normalized at the adapter boundary (#1010): the CLI's raw `input_tokens` is cache-INCLUSIVE (it counts `cached_input_tokens`), so the adapter reports `Usage.InputTokens` = summed input − summed cached,
clamped at 0 — the cache-EXCLUSIVE fresh count every adapter now reports — with the cached sum in `Usage.CachedInputTokens` and the turn count in `Usage.Turns` (#995); `reasoning_output_tokens` is added to the output side. Two advisory WARN ceilings watch the normalized figures in `recordReviewerCost`: 100k on fresh input (a genuine context-assembly blowup) and 500k on total
input (fresh + cached — a runaway total context that heavy caching keeps off the fresh ceiling). The event shapes are pinned against the installed Codex CLI (codex-cli 0.137.0, the same version the executor pinned) and encoded in the fake-binary test fixtures so a future CLI drift fails a test rather than silently dropping the reviewer. **Scratch-dir cwd contract (#995):** each
invocation runs from a fresh EMPTY `os.MkdirTemp` directory set as `cmd.Dir` (removed after the call). The review prompt forbids tool use but cannot prevent it — `codex exec` non-interactive runs read-only shell commands without approval, so an unset `cmd.Dir` inherits fishhawkd's cwd (the repo checkout, in the dogfood loop) and an exploring reviewer reads it across many turns,
each turn re-sending the full growing conversation (the observed ~400k-input-token review, ~100x the claude reviewer). The empty cwd makes exploration fruitless; a `MkdirTemp` failure FAILS CLOSED as a per-invocation error (degrading the advisory review gracefully via the per-invocation failure path) rather than silently regaining the unbounded workspace.
`--skip-git-repo-check` is required because the scratch dir is not a git repo (and the backend has no checkout anyway); unlike the executor the reviewer needs **no** `--dangerously-bypass-approvals-and-sandbox` (confirmed against a real `codex exec --json` from a non-repo dir, which returns cleanly without blocking on approvals). The decoded `text` is run through the shared
`planreview.DecodeVerdict` (#739 fence-strip + #889 invalid-escape tolerance), validated against the same closed verdict set, and the summed usage is attached with `Known=false` when no `turn.completed` usage line appeared (no silent $0, #681/#682). `OPENAI_API_KEY` is forwarded explicitly from `FISHHAWKD_OPENAI_API_KEY` on top of the inherited `os.Environ()`; an empty key is
not an error (ChatGPT-login auth on the host is supported). The retry budget is wired through `SetMaxRetries` exactly as the claudecode adapter does. The review (and `--version` probe) `cmd` is `procgroup.Harden`ed so a wedged reviewer holding an inherited stdout pipe is terminated at the review budget deadline rather than hanging (#1805 — see the subprocess-hardening note
under "Local-mode wire-up"). This enables the **Claude main, Codex reviewer** topology.

**Prompt-caching strategy**: the `plan_review` prompt (`buildPlanReview` in `backend/internal/prompt/prompt.go`) is split at `prompt.PlanReviewSplitMarker` (`"\n### Plan artifact\n\n"`). Everything before the marker — the role-constraint preamble — is placed in the Anthropic system block with `cache_control: {type: ephemeral}`. Everything from the marker onward (plan artifact +
issue context + verdict schema) is placed in the user message. This keeps the cached prefix stable across calls for the same fishhawkd process lifetime while allowing the variable plan content to change per invocation.

**Reviewer-capability degradation (#574, #1495).** The workflow spec is authoritative for WHICH reviewers a stage runs; the `FISHHAWKD_ENABLE_*` / `FISHHAWKD_ANTHROPIC_API_KEY` flags are deployment **capability gates**, not policy switches. When a declared reviewer's provider is unavailable on the deployment, degradation is split by *how much* is missing:

- **Coarse — no reviewer backend wired at all** (`defaultPlanReviewer() == nil`): a run with a gating plan stage (`agent>0, human==0`) that could never be reviewed on this deployment is **still rejected before any run row is minted, on both entry points**. `handleCreateRun` (`backend/internal/server/runs.go`, `gatingReviewerProblem`) writes a `run_rejected_misconfigured`
  **global-chain** audit entry (`AppendGlobalChained` — no run row exists yet) and returns `400 plan_reviewer_unconfigured`. The webhook-dispatcher path (`Dispatcher.Handle`, gated on a plain `PlanReviewerConfigured` bool wired from `cfg.PlanReviewers != nil && …Default() != nil` to avoid the server→webhook import cycle) applies the **symmetric** guard (#577): same
  `run_rejected_misconfigured` entry, refuses to dispatch. This is a deployment-wide misconfiguration (zero review infrastructure) — the per-reviewer `optional` flag does NOT apply. Because the dispatcher returns 202 to GitHub and cannot surface a 400, the audit entry plus a WARN log are its rejection surface.
- **Per-reviewer capability gap** (`Default() != nil`, but a specific `reviewers.agents[]` provider unresolvable): #1495 **no longer hard-fails** run creation — the spec is valid, only the deployment capability is missing. `handleCreateRun` (`unavailableSpecReviewers` → `emitReviewerCapabilityUnavailable`) writes a `reviewer_capability_unavailable` global-chain audit per
  unavailable reviewer (honoring `optional`: ERROR log for `optional:false`, INFO for `optional:true`) and **proceeds** (201). The reviewer degrades again at the runtime review loop with a capability-framed `*_review_skipped` entry. The webhook dispatcher needs no change here: with a backend wired its coarse `!PlanReviewerConfigured` guard is skipped, the run is created, and
  the SAME runtime `*_review_skipped` degradation applies — so both run-create paths behave consistently for the per-reviewer case.
- **Advisory mode** (`agent>0, human>0`): allowed through entirely — the human gate remains authoritative; the runtime review loops record the degradation.

**Audit category `plan_review_skipped` / `implement_review_skipped`** (#574, #1495): a terminal review entry recording that a spec-requested agent review did NOT run, payload `ReviewSkippedPayload{Reason, ConfiguredAgents, Authority, Provider?, Optional?}` (`backend/internal/planreview/review.go`). Two reasons share it: `reviewer_not_configured` (coarse — no backend wired at
all, no `Provider`) and `reviewer_unavailable` (#1495 per-reviewer capability gap — carries `Provider` + `Optional`). It counts as a terminal entry for the N-of-N review-settled gate (`planreview.Settled`), so a degraded reviewer never strands `fishhawk_await_review`. Surfaced by `fishhawk_get_plan` as a synthesized review with `verdict: "skipped"` and the recorded reason.

**Audit category `run_rejected_misconfigured`** (#574, #577): global-chain entry written when a gating plan stage is rejected for the COARSE no-backend-at-all capability gap — by `handleCreateRun` on the run-create path and by `Dispatcher.Handle` on the webhook-dispatcher path. Payload names the offending stage, workflow_id, repo, and configured agent count; the dispatcher
additionally carries `delivery_id` and `event` for GitHub-trigger correlation. On the dispatcher path, when the trigger is an issue (labeled or `/fishhawk run`), the refusal is also surfaced to the customer as an issue comment via `IssueNotifier.NotifyRunRejected` (#599). This entry is now scoped to the coarse case only; the finer per-reviewer capability gap emits
`reviewer_capability_unavailable` and proceeds instead.

**Audit category `reviewer_capability_unavailable`** (#1495): global-chain entry written by `handleCreateRun` when a spec-declared reviewer's provider is unavailable on the deployment but a backend IS wired — the per-reviewer capability gap that DEGRADES rather than rejects. Payload `{reason: "reviewer_unavailable", provider, optional, stage, workflow_id, repo,
configured_agents}`. It is an internal (non-issue-comment) audit kind; the run proceeds to creation.

A startup WARN fires at `fishhawkd` boot (`backend/cmd/fishhawkd/serve.go`) whenever the resolved `PlanReviewer` is nil (neither the API key nor the local-mode flag is set), describing both degradation modes; it disappears once either adapter is wired.

**Review dispatch: advisory-async vs. gating-sync (#584).** Both `runPlanReviews` and `runImplementReviews` resolve the stage's review config on the request goroutine (cheap reads: `GetRun`, `resolveStageReviewers`, plan parse/load, prompt build), then split on authority:

- **Advisory** (`agent>0, human>0`): the per-reviewer loop is dispatched on a **detached** context (`context.WithoutCancel(ctx)`) in a goroutine tracked by `Server.bgReviews`, and the handler returns its response (`201`/`202`) immediately without waiting on the reviewer. This is the fix for the reviewer being SIGKILLed at the runner's upload-client timeout: the review
  previously ran on `r.Context()`, which the server cancels when the runner's client closes the connection (default was 60s), so `context.WithTimeout(r.Context(), FISHHAWKD_PLAN_REVIEW_TIMEOUT)` fired at `min(parent, budget) = 60s` and `exec.CommandContext` killed `claude`. Detaching lets the review run to its own size-aware review budget (#747; floor
  `FISHHAWKD_PLAN_REVIEW_TIMEOUT`, scaled by prompt size). Because the detached `reviewCtx` carries no deadline, that per-invocation budget — applied at the call site via `context.WithTimeout(reviewCtx, ReviewBudget.Budget(len(promptText)))` and honoured by the adapter — is the only deadline in the chain (`context.WithTimeout` takes the earlier of parent/child, so a larger
  budget can't extend a shorter parent that doesn't exist here). The advisory verdict is non-blocking (the human gate stays authoritative), so it landing after the stage reaches `awaiting_approval` is correct.
- **Gating** (`agent>0, human==0`): the review loop runs **synchronously** on the (still detached) context. Synchronicity is required because a reject must transition the stage to `failed` (category B) *before* the trace handler's terminal `awaiting_approval` advance — the block relies on the `awaiting_approval → failed` (plan path) / `running → failed` (implement path)
  state-machine edges. The detached context still removes the client-disconnect clamp; it just doesn't return early.

**Graceful-shutdown drain.** `Server.Shutdown` waits on `bgReviews` after `http.Shutdown` returns, bounded by the same shutdown context, so in-flight advisory reviews finish (or are abandoned at the deadline) rather than being dropped. `Server.waitBackgroundReviews()` is the test sync point for deterministic async assertions. The runner's upload-client default timeout was also
raised 60s→120s as defense-in-depth (`runner/internal/upload/upload.go`), but the decoupling — not the bump — is the real fix; a retried upload short-circuits at `GetByHash`.

**Plan-approval completion gate (ADR-036 / #875).** Because an advisory plan review is dispatched **detached** (above), the plan stage reaches `awaiting_approval` while the reviewer is still running — so a human could approve before reading the agent's verdict. `checkPlanReviewSettled` (`backend/internal/server/approvals.go`) closes that window: on a plan-stage **approve**,
when the stage declares `reviewers.agent > 0` AND a `plan_review_started` entry is present AND fewer than the configured agent count of TERMINAL verdicts have landed, it refuses with `409 agent_review_pending` (`details`: `stage_id`, `configured_agents`, `landed_terminal`). ANY terminal kind counts toward the unblock — `plan_reviewed`, `plan_review_failed` (the #747 budget-kill
emits this), or `plan_review_skipped` — so a timed-out or skipped reviewer never strands the gate. The gate is placed **before** `ApprovalRepo.Submit`, so a refused approval records no row and the operator's poll-then-retry (idempotent) flows normally once verdicts land. A hard backstop — `planReviewBackstop(agentCount) = ReviewBudget.Cap × max(1, agentCount)`, measured from
the earliest `plan_review_started` — ALLOWS the approval once elapsed even with no terminal verdict, writing a `plan_review_backstop_elapsed` audit entry (actor `system`; payload `{stage_id, configured_agents, landed_terminal, started_at, elapsed_seconds}`), so a reviewer that dies emitting nothing can never brick approvals. Every read fails OPEN (WARN-log, allow) mirroring
`checkPlanBudget`. `reviewers.agent == 0` (no agent reviewer) and a configured-but-not-dispatched stage (no `plan_review_started`) are byte-for-byte the pre-ADR-036 path. Surfaced through `fishhawk_approve_plan` as a typed poll-until-landed error carrying the landed/configured counts (no auto-retry in the tool).

**Implement-review / merge completion gate (ADR-036 / #876).** The same completion-gate discipline extends to the implement-review (ADR-027) / merge gate, but at the **merge-driven resolution** path rather than an HTTP approve. `checkImplementReviewSettled` (`backend/internal/server/pullrequest_review_events.go`) gates `resolveReviewStageOnMerge`: on a merge (`merged==true`) of
a run that has a review stage, when the IMPLEMENT stage declares `reviewers.agent > 0` AND an `implement_review_started` entry is present AND fewer than the configured count of TERMINAL `implement_reviewed`/`implement_review_failed`/`implement_review_skipped` verdicts have landed, it refuses to resolve the stage to `succeeded` and leaves the run **parked in
`awaiting_approval`**. The merge reconciler (`backend/internal/mergereconciler/`) re-scans parked review stages each tick and routes them back through `ResolveReviewFromPollState → resolveReviewStageOnMerge`, so the gate retries identically on both the `pull_request.closed` webhook (one-shot) and the poll (retrying), and **composes with the #862 lineage re-check** already on
the tick path (both must pass for the stage to resolve). ANY terminal kind unblocks (a budget-killed reviewer emits `implement_review_failed` — see `trace.go`), so a timed-out reviewer never strands the merge. A hard backstop reusing `planReviewBackstop(agentCount) = ReviewBudget.Cap × max(1, agentCount)` (measured from the earliest `implement_review_started`) ALLOWS the
resolution once elapsed even with no terminal verdict, recording an `implement_review_backstop_elapsed` audit entry (actor `system`; payload `{stage_id, configured_agents, landed_terminal, started_at, elapsed_seconds}`) exactly once (the resolve completes the run, so the reconciler stops re-polling). Every read fails OPEN (WARN-log, resolve); `reviewers.agent == 0` and a
not-dispatched stage are byte-for-byte the pre-ADR-036 merge resolution. **Behavioral change — deferred `pr_merged` audit:** the gate is placed BEFORE `writePRMergedAudit`, which DEFERS the `pr_merged` row from merge-observation time to resolution time so a held run does not write a duplicate `pr_merged` row on every re-poll tick. A GitHub-merged run that is held by this gate
therefore has NO `pr_merged` row until the re-poll clears the gate; the backstop guarantees the run always eventually resolves, so `pr_merged` is always eventually recorded. **Accepted limitations** (mirror ADR-036 / #862): the terminal count is run-wide rather than head-SHA-matched (exact for the common `agent=1` case; squash-merge SHA divergence makes per-SHA matching
unreliable), and this gate observes a merge GitHub already performed — it refuses to mark the run `succeeded` but does not block the GitHub-side merge; the cosmetic dangling-open-PR residue is tracked in #877. The `merged==false` (closed-without-merge → `cancelled`) branch is **never** gated — an abandoned PR is terminal regardless of review state.

### Implement-review-agent invocation (ADR-027 impl 2/2)

The same review-agent machinery extends to the **implement** stage: after the implement-stage diff lands in the trace bundle, if the implement stage's `reviewers.agent > 0`, the backend spawns that many review agents against the diff + approved plan + issue. This reuses the verdict shape, `ResolveAuthority` table, self-review WARN, and `PlanReviewer` interface verbatim — no
interface change. The single new prompt variant is `implement_review` (`buildImplementReview` in `backend/internal/prompt/prompt.go`).

**Where the hook runs.** Unlike plan-review (which runs in `handleShipPlan`), implement-review runs in the **trace handler** (`backend/internal/server/trace.go::advanceStageAfterTrace`), because the diff is what's reviewed and it arrives via the trace bundle (the same `bundle.ExtractDiff` the policy re-eval uses, so it works in local `--no-pr` mode too — the diff lives in the
trace regardless of whether a PR was opened). Ordering within the trace handler:

1. Manifest `agent_failed` check (fail-A short-circuit) → policy re-evaluation (`reEvaluatePolicy`).
2. `advanceStageAfterTrace`: `dispatched → running`.
3. **Implement-review hook** (guarded by `stage.Type == implement`): extract the diff via `bundle.ExtractDiff`; if extraction errors, skip review and proceed. Otherwise call `runImplementReviews`.
4. Terminal transition: `running → awaiting_approval` (or `succeeded` for gateless).

The hook is placed between step 2 and step 4 so a **gating** reject fails the stage to `failed` (category B) *before* the terminal transition — the terminal `awaiting_approval` advance is then blocked by the state machine, exactly mirroring `runPlanReviews`' contract. **Advisory** implement review is dispatched asynchronously (#584; see **Review dispatch: advisory-async vs.
gating-sync** above): `runImplementReviews` returns `false` immediately and the terminal transition proceeds while the detached reviewer runs, so the upload is never held open for an advisory review. When `stage.Type != implement` or `reviewers.agent == 0`, the hook is a no-op and the trace path is byte-identical to the pre-ADR-027 flow.

**`runImplementReviews`** (`backend/internal/server/trace.go`) mirrors `runPlanReviews`: it resolves the implement stage's `ReviewersConfig` via the shared `resolveStageReviewers(runRow, spec.StageTypeImplement)`, loads the approved plan (self-review guard + prompt context), builds the `implement_review` prompt with the changed-files diff summary, invokes `PlanReviewer.Review`
per configured agent, WARNs on self-review (the approved plan's `GeneratedBy.Model` is used as an approximation of the implement author's model — v0 does not record the implement agent's model separately), and appends one `implement_reviewed` audit entry per invocation. It returns `true` (gating block) only when authority is gating (`agent>0 && human==0`) and at least one
verdict is `reject`.

**Decomposition fan-out review coverage (#677)**: `runImplementReviews` carries **no** `decomposed_from` guard — it fires on every implement-stage trace upload and resolves reviewers from the child run's own `WorkflowSpec` via `resolveStageReviewers(StageTypeImplement)`. So each fan-out child slice **already** receives its own per-child implement-review (with the #703
orthogonal lens) whenever the workflow's implement stage configures reviewers — no new code; the acceptance criterion "a fan-out child slice receives at least one advisory review" is satisfied by the existing machinery, not deferred (verified live: child run `1cd5ae2e`, sub-plan 1 of #688, emitted `implement_review_started → implement_reviewed(approve_with_concerns)` on
2026-06-02). What a child gets **no** per-child equivalent of is the **plan** review — by design it has no plan stage: the parent plan is reviewed pre-fan-out, and (per #677) the parent's binding approval-conditions now propagate into each child implement prompt via `resolveApprovalConditions` (see "Per-stage prompt construction"). The parent's own review stage remains the
post-`children_settled` aggregate gate over the combined child branch.

**Gating consolidated review on the parent (#1060)**: per-child reviews are advisory-to-the-slice early signal — none of them sees the *consolidated* diff that actually merges, so a child-raised high never gated the parent merge and drive advanced `review:awaiting_approval → awaiting_merge` on vacuously-terminal review evidence (zero reviewers ever ran against the parent). Two
coupled pieces close this. (1) **Drive safety** (slice 1): `implementReviewRound` now resolves the configured agent count from the spec even when no `implement_review_started` round was dispatched, so `ObserveParkedReviewForDrive` can distinguish a genuinely reviewer-less run (`configured==0`, vacuously terminal — still advances, never wedges) from a parent with reviewers
configured but no round yet dispatched (`configured>0 && !started`, non-terminal — parks instead of stamping `checks_green_awaiting_merge`). (2) **Consolidated dispatch** (slice 2): when `orchestrator.Advance` dispatches a decomposed parent's review stage WITH the consolidated PR present, it fires `ConsolidatedReviewDispatcher` (interface field on the orchestrator, satisfied
server-side by `*server.Server` to keep the dependency direction `server→orchestrator` and avoid an import cycle). `Server.DispatchConsolidatedReview` sources the consolidated diff via `githubclient.ComparePatch(base=DefaultRef, head=consolidatedBranch(parentID))` and runs `runImplementReviews` against it with the stageID of the **parent's implement stage** (the
`awaiting_children→succeeded` stage). That stage id is the load-bearing seam: the `implement_review_started`/`implement_reviewed` round and any concerns attach there, so the round is non-vacuous for the drive gate AND `fishhawk_fixup_stage` on that stage resolves the consolidated concern (same `(run, stage)` concern store, no new fix-up code path). The dispatch is best-effort +
idempotent (runs detached on `bgReviews`; `runImplementReviews`' started-key guard dedups the sweeper/event-driven double-advance) and no-ops for ordinary runs (a `decomposed_from==nil` run with a PR but no children). A GitHub-truncated consolidated diff (the large-diff fan-out case — the 300-file cap or an omitted oversized patch) is surfaced loudly: a WARN log plus a durable
`consolidated_review_diff_truncated` audit entry (system actor, payload `{reason, changed_files}`, attached to the parent implement stage), and the review still runs on the partial diff rather than aborting. **Parent fix-up re-invokes the shared branch (#1063)**: a fix-up on a decomposed parent's implement stage re-opens it to `pending`, where `Advance` re-enters
`fanoutIfDecomposed`. That helper is now guarded on existing children — when the parent already has minted children (a `ListRuns(DecomposedFrom==parent, Limit:1)` probe returns ≥1), it skips the fan-out and returns `(false, nil)` so `Advance` falls through to `dispatchStage` and re-invokes the parent's implement stage against the existing shared branch rather than re-minting a
fresh fan-out. The fix-up branch derivation matches: `server.fixupBranchForRun` resolves a decomposed PARENT (`DecomposedFrom==nil` WITH minted children) to the consolidated/shared branch `fishhawk/run-<shortID(runID)>` (byte-matching `orchestrator.consolidatedBranch`) instead of the per-stage branch, so the fix-up commit lands on the consolidated PR head. Only the first
fan-out (zero children) mints; the concern-attach + resolve seam is unchanged.

**Review lens is re-aimed at the orthogonal gaps** (#703): the `implement_review` prompt no longer re-verifies plan adherence or generic-bug-hunts — mechanical correctness is already gated by the policy gate, the implement agent's test run, build/lint, and CI, so re-stating it adds no signal. Instead it applies the three lenses the deterministic gates are blind to: security /
authz (anchored to Fishhawk's code-execution threat model — lethal trifecta / uncontrolled egress, ADR-029 / #650), test vacuity (a test that asserts nothing load-bearing still passes CI), and untested error / edge / concurrency paths. The security lens is **self-gating**: on a low-risk diff that touches no sensitive surface it states so briefly and stops rather than
manufacturing a concern. The reviewer stays advisory (no authority change); the grounded-citations and style-is-lint criteria (#595) are retained.

**scope.files drift is flag-only** (ADR-027 Decision Q6): the prompt instructs the review agent to emit a `{category: "scope"}` concern when the diff touches files outside the plan's `scope.files`, but drift alone never forces a `reject`. Only an overall verdict of `reject` blocks under gating authority. A reviewer returning `approve_with_concerns` with only a scope concern
advances the stage.

**Operator-stageable scope_drift is surfaced to the reviewer** (#695, #581): the review's `git_diff` is **scope-bounded** — it excludes paths the implement stage dirtied but did not declare, which the operator may `git add` into the final commit. The runner already emits those paths as a `scope_drift` `policy_event` in the trace bundle (from `computeAndEmitDiff`); the trace
handler now reads them via `bundle.ExtractScopeDrift` and threads them onto `prompt.Trigger.ScopeDrift`, so `buildImplementReview` renders a "Scope drift (excluded from the diff above — operator may stage)" section naming each path. This stops the advisory reviewer false-rejecting a required test/doc as "absent from the committed artifact" when it actually landed via a drifted
path. A **standing anti-false-reject rule** backs this up even when the event is absent or stripped (best-effort extraction degrades to `nil`): the reviewer must treat an absence it cannot positively confirm as unverifiable (→ `approve_with_concerns`), never confirmed-missing. The reviewer sees only the drifted path **names**, not their content (the bundle's scoped diff
excludes drift content), so it can flag-vs-false-reject but cannot test-vacuity-check a drifted test file — an inherent limit of the backend-only approach.

**Machine-verified gate evidence is surfaced to the reviewer** (#963): the runner appends a single synthesized `gate_evidence` trace event (kind `gate_evidence`, composed by `composeGateEvidence` immediately before `bundle.PackBytes` so BOTH variants carry it) digesting the stage's deterministic gate results: per-`verify_run` command/exit/outcome with a bounded (last 30 lines,
~4KB) output tail, the `verify_summary` outcome/iterations/detail, `verify_infra_flake_retry` count, scope-enforcement facts (declared `scope.files` count, staged-file count, the `scope_drift` undeclared list), and constraint-violation `policy_events`. Every free-text field is pre-redacted runner-side via `redaction.RedactDefault` (the shared `redaction/` module, #1106) —
required by ordering (the implement review dispatches on the RAW variant, before the redacted variant exists); the raw `verify_run` events keep their full unredacted output for compliance, only the derived digest is prompt-bound. The trace handler reads it via `bundle.ExtractGateEvidence` (lockstep wire-contract structs; absent event → `ErrNoGateEvidence`, fail-open like
`ErrNoHeadSHA`) and threads it onto `prompt.Trigger.GateEvidence`, so `buildImplementReview` renders a "### Gate evidence (machine-verified — outranks text-level findings)" section with BINDING guidance: a failed verify gate or verified/staged divergence must be recorded as a high-severity concern named first and may shortcut the remaining lenses; a skipped gate means
compile/test state is UNVERIFIED (never assume CI-green); a passed gate certifies only exit-0, never test quality (the #703 lenses still apply). A failed verify gate is a committed-tree (high-severity) blocker ONLY when it is the TERMINAL (non-superseded) `verify_run` or the `verify_summary` outcome is `failed` (#1205): the verify-fix loop (#651/#800) re-runs the SAME gate per
iteration on an evolving tree, so `composeGateEvidence` marks every `verify_run` except the last with an additive `superseded` flag (omitempty across all four lockstep layers — runner composer → bundle wire → server mapping → prompt struct — so older bundles decode to not-superseded), the renderer flags such a run `— SUPERSEDED`, and an absorbed-then-passed iteration MUST NOT
be read as a committed-tree failure (a reviewer false-rejected HIGH on exactly such an absorbed failure, run fa5a6416). The last `verify_run` is NEVER marked, so a genuine terminal failure — including a budget-exhausted `[fail,fail]` loop — stays an independent blocker the marking can never mask. When evidence is present the non-goals preamble defers to the section instead of
asserting "mechanical correctness is already gated upstream" — the unconditional claim that licensed the run-07bce059 reviewer to produce a careful text-level verdict on a head the gates knew did not compile. Nil evidence (older bundles, no gate ran, extraction error) omits the section and keeps the prompt byte-identical (the #984 pattern). Prompt-context only — ADR-027
advisory/gating semantics, dispatch plumbing, and verdict handling are untouched. The scope facts carry per-path drift A/B categories (#991): the runner's `scope_drift` emitter partitions the drift list with `gitops.UntrackedPaths` (the same primitive as the #818/#825 created-out-of-scope gate, so evidence and enforcement agree by construction) into an additive
`undeclared_categorized` field — category `A` (tracked-file edit excluded from the commit; the pushed head may be missing a required change) vs `B` (created out of scope), each with a disposition (`excluded_from_commit` | `would_fail_loud`; decomposed children's `B` paths get `excluded_from_commit` since they're exempt from the created-out-of-scope gate). Categorization is
best-effort advisory evidence (on error the runner logs `scope_drift_categorize_failed` and emits today's uncategorized payload), the legacy `undeclared` list is byte-for-byte unchanged for its existing consumers, and decode is tolerant in both directions: older runners' bundles render the uncategorized path lines byte-identically, and an uncategorized entry renders bare even
when sibling paths are annotated. **Advisory fix-up self-report divergence (#1210).** On a fix-up pass the slim prompt instructs the agent to write a tiny run/stage-keyed JSON sidecar (`prompt.FixupSelfReportPath(run_id, stage_id)` = `/tmp/fishhawk-fixup-selfreport-<run_id>-<stage_id>.json`, mirrored verbatim by the runner's `fixupSelfReportPath`) declaring its claimed
`verify_status` (`passed` | `failed`). After the agent settles, `run()` (guarded `stageType == "implement" && cfg.fixup`) calls `loadFixupSelfReport` — validated **fail-closed** exactly like `loadScopeExemptions` (well-formed JSON, embedded `run_id`/`stage_id` matching the stage, status one of the two literals; else zero + a named log event; sidecar deleted on every return,
swept pre-invoke by `sweepStaleFixupSelfReport`) — and deterministically compares the claim against `terminalVerifyOutcome(res.Events)` (the `verify_summary.Outcome`, else the last `verify_run.Outcome`, the same committed-tree verdict `composeGateEvidence` digests; production literals `passed`/`failed` determinate, `""`/`skipped` indeterminate). A determinate disagreement
appends a `fixup_selfreport_divergence` trace event (claimed-vs-actual) that `composeGateEvidence` folds into `gate_evidence` (`FixupSelfReportDivergence`, omitempty across all four lockstep layers) and `writeGateEvidence` renders as an ADVISORY honesty flag for the reviewer to arbitrate. This closes the #1210 blind spot (a fix-up claimed its change verified one way while the
committed tree verified the other, caught only by heterogeneous agent review). The signal is **advisory throughout**: it never demotes `res.OK`, never fails or re-opens the pass, and never consumes or refunds budget (avoiding the #1150 budget-wedge family). Any indeterminate side (no claim, an invalid/stale sidecar, or a `""`/`skipped` actual) emits nothing — near-zero false
positives. The advisory→hard-gate promotion is a deferred follow-up.

**Supplemental base-rebase re-invoke review (ADR-042 / #1250).** The first implement review (above) dispatches at trace-upload time, anchored to the **sealed** trace bundle, strictly BEFORE any base-rebase re-invoke runs. On the re-invoke path the runner re-runs the implement agent on a freshly-rebased base AFTER the bundle sealed, and the final scope-completeness gate may
honor ADDITIONAL declared-scope-file exemptions the sealed `gate_evidence` event never carried. #1218 surfaced that delta to the **audit log only** (a supplemental `scope_files_exempted` row, `origin=base_rebase_reinvoke`); the review surface stayed blind to it. #1250 closes the gate_evidence half: when a success ship carries a non-empty `SupplementalScopeExemptions` delta
(`backend/internal/server/pullrequest.go::handleShipPullRequest`, in the #742 push-and-open-pr terminal-drive branch, BEFORE `advanceImplementStageAfterPR`), the backend dispatches a bounded, ADDITIVE supplemental implement-review pass (`trace.go::runSupplementalReinvokeReview`) anchored to the PUSHED re-landed tree — the exact point the re-landed tree becomes durable under
#742 forward gating. The supplemental prompt (`buildImplementReview` with `Trigger.SupplementalReinvoke=true`) renders NO diff — an exempted path is unchanged by definition — and judges ONLY whether each additional exemption is sound, reusing the same `writeGateEvidence` ScopeExemptions section the first review applies to exemptions. The verdict is recorded as an additive
`implement_reviewed` entry carrying `Origin=base_rebase_reinvoke` + the re-landed `HeadSHA` (both `omitempty`, byte-identical for the first review). Two invariants make it ADDITIVE rather than burying the first verdict: (1) it does NOT call `emitReviewStarted`, so the anchor's verdict-counting floor (the latest `implement_review_started` Sequence,
`issuecomment/anchor_template.go`) stays at the first review and the supplemental `implement_reviewed` counts ABOVE it; (2) the pre-dispatch idempotency dedup keys on `(stage_id, Origin, HeadSHA)` (`supplementalReinvokeReviewAlreadyRecorded`), so a retried PR-upload with the same re-landed head does not double-dispatch (best-effort/non-atomic — acceptable because the runner
drives a stage's PR-uploads serially). Authority mirrors `runImplementReviews`: advisory dispatches detached on `bgReviews`; gating runs synchronously and a reject fails the stage **category-B** (reusing `implementReviewGatingRejectReason`) then closes the dangling PR (reusing the #877 `closePRAfterGatingReject` helper) — an accepted consequence that a gating reject fires AFTER
the runner already opened the PR, since the honored delta is only known post-PR-open. The #1218 audit row and this review verdict are COMPLEMENTARY surfaces; both ride the same delta.

**Binding approval conditions are surfaced to the reviewer** (#1021): the implement-review prompt renders the run's approve-with-conditions text (#558 amendments) via the same `resolveApprovalConditions` resolver the implement-stage prompt uses (decomposed-child/retry-parent fallback included, threaded in `runImplementReviews`), as a "### Approval conditions (binding — AMEND
the plan, win on conflict)" section placed immediately before the plan artifact. The framing instructs the reviewer that a diff implementing a condition that contradicts the original plan text is NOT a plan deviation — the condition is the controlling instruction (runs 338d6b0f/256032f6 were rejected for correctly implementing operator amendments). Fix-up re-reviews inherit it
because the post-fixup re-dispatch routes through `runImplementReviews`; nil conditions omit the section and keep the prompt byte-identical (the #984 pattern).

**Audit category `implement_review_started`** (#600): the implement-stage twin of `plan_review_started`, emitted by `runImplementReviews` at dispatch under the same agent>0-AND-wired guard and ordering invariant (started precedes every `implement_reviewed` entry). `review_status` = `pending` state for the implement stage on `fishhawk_get_run_status` and `fishhawk_await_review`. **`review_action_hint` (#777)**: a DISPLAY-ONLY timing nudge surfaced on `fishhawk_get_run_status` (and on `fishhawk_run_stage` after an implement stage) computed entirely in the MCP layer (`backend/cmd/fishhawk-mcp/review_action_hint.go::reviewActionHintFor`) from audit data it already reads — no backend API, persistence, or schema change. `complete` with one or more unresolved `approve_with_concerns` concerns (summed across reviewers from the `implement_reviewed` entries) AND the bounded fix-up budget is not yet spent (counting prior `stage_fixup_triggered` entries against the mirrored `maxFixupPasses`, kept in sync with `server.defaultMaxFixupPasses`), it returns `{concerns, remaining_fixup_budget, message}`

`fishhawk_start_run` (no implement review exists at run start) nor on plan/review `fishhawk_run_stage` calls. In `getRunStatus` the hint is derived from the SAME `implement_review_status` already computed for the response, so the two cannot disagree within one read.

**Audit categories `plan_review_failed` / `implement_review_failed`** (#664): the terminal entry written when a *wired* reviewer invocation errors or times out (a reviewer killed at `FISHHAWKD_PLAN_REVIEW_TIMEOUT` surfaces as a `Review` error). Emitted per failed invocation by the shared `Server.emitReviewFailed` helper from both review-error branches (`runPlanReviewLoop` in
`plan.go`, `runImplementReviewLoop` in `trace.go`), payload `ReviewFailedPayload{Reason, ReviewerModel, Authority, Timeout}`. The `Timeout` bool (#747, `omitempty`) is the additive discriminator that tells a budget-deadline kill (`context.DeadlineExceeded` at the call site) apart from a transport/decode failure; the audit category is unchanged so existing #664 and MCP
`await_review` readers are unaffected. Kept distinct from `*_reviewed` (which carries only the closed approve/approve_with_concerns/reject verdict set) so a timeout is never decoded as a real verdict. Internal audit categories, not issue-comment surfaces. They back the MCP `review_status` = `failed` terminal state on `fishhawk_get_run_status` and `fishhawk_await_review`
(precedence `complete > skipped > failed > pending > none`), resolving what previously fell through to an ambiguous `pending`. **Observability-only**: emitting this entry does NOT change gating advance/degrade semantics — a gating reviewer that times out still degrades to advance per the #574 advisory-degradation decision (`hasRejection` is untouched).
