# Fishhawk — Architecture

> **Status:** Living document; tracks the current technical realization of the v0 spec.
> **Audience:** Agents and engineers working in this repo. Optimized for density, not narrative.
> **Authority:** `docs/MVP_SPEC.md` defines scope; this doc defines the *technical realization* of that scope. When they disagree, MVP_SPEC wins for "what" and this doc wins for "how".

## 1. System summary

Fishhawk is a **governance and workflow layer for agent-driven software changes**. A customer commits a YAML workflow spec to their repo, triggers it (issue label, CLI, UI), and the system runs the agent on their CI under typed constraints, captures a signed trace, and gates each stage on human approval. The product surface is the workflow execution and audit history; the agent itself is pluggable.

See `docs/MVP_SPEC.md` §1–§4 for product framing, primitives, and customer-side spec syntax.

## 4. Workflow run lifecycle

Per `docs/MVP_SPEC.md` §5.2. Concrete realization in this codebase:

1. **Trigger** — GitHub issue label/assignment (webhook → backend), CLI `fishhawk run start`, or UI button. Backend validates the workflow spec at the issue's `.fishhawk/workflows.yaml` SHA, creates a `runs` row, and emits `workflow_dispatch` to the customer's repo invoking `fishhawk/runner@runner/vX.Y.Z`.
2. **Plan stage** — Runner checks out the repo, calls backend `POST /v0/runs/{id}/signing-key` (with GitHub OIDC token) and receives an Ed25519 private key + run metadata. Invokes Claude Code with the plan prompt. Captures full trace as JSON Lines events. Validates the plan artifact against `standard_v1` schema (E1.5 / #20). Signs `sha256(bundle)`. Ships `(bundle, sig)` to `POST /v0/runs/{id}/trace`.
3. **Backend ingest** — Verifies signature against the stored public key for `run_id`. Stores the bundle in S3 keyed by content sha256. Persists the plan artifact in `artifacts` and renders it as a comment on the originating GitHub issue (mode: `rendered_comment`, kept in sync). Transitions stage state.
4. **Plan approval** — Approver reads the plan in the Web UI (canonical surface) or the issue comment (read-only echo). Clicks Approve. Backend records `(approver_subject, surface, ts)` in the audit log and transitions to implement.
5. **Implement stage** — Runner re-invokes with the implement prompt + approved plan. Captures trace. Post-hoc evaluates constraints (`forbidden_paths`, `max_files_changed`, `required_outcomes`) against the produced diff. If a constraint hits, the stage fails as category B. Otherwise pushes branch, opens PR via the GitHub App's installation token, ships final trace.
   - **Pre-push compile gate (#728).** On the standalone (non-decomposed) PR-opening path, after the scope-only commit is created and **before** the push, the runner (`runner/cmd/fishhawk-runner/main.go::verifyCommittedTreeCompiles`, wired via `gitops.CommitAndPushArgs.VerifyCommit`) compiles the *committed* tree in an isolated `git worktree --detach` at the new HEAD SHA — not the working tree, which still carries the excluded scope-drift on disk — and runs `go vet ./...` per `go.work` module. `go vet` typechecks test files (which `go build` skips), so it catches the interface-ripple class where a scope-bounded commit drops a build-required conformance stub left in a test file (`does not implement <iface> (missing method ...)`). A real compile failure fails the stage **category B** (`gitops.ErrCommitWouldNotCompile`, re-scope/re-plan) with the drift file list, and because it runs pre-push, origin is untouched — no non-compiling PR is opened. Every infrastructure case (no drift, non-Go repo, no Go toolchain, worktree/`go work edit` failure, or a `go vet` non-zero caused by dependency resolution rather than a typecheck error) is a non-blocking skip (`compile_gate_skipped`), degrading to the prior "open the PR" behavior. Decomposed children are intentionally ungated (a child may legitimately not compile before later children land); gating the consolidated decomposition tip is a follow-up.
   - **Pre-push test gate (#800).** After the `go vet` compile check passes, the same committed-HEAD worktree runs a second phase: `go test` (no `-race` — CI runs the race build; this is the fast touched-package check) on the *touched packages* — the packages containing the declared `scope.files`, derived per `go.work` module (`touchedPackageArgs` → `./<reldir>/...` patterns), **not** the drift dirs (drift is excluded from the committed tree). This catches the drift-excluded-test-failure class (#780/#776) where a scope-only commit compiles but a necessary test fake/helper was dropped as scope drift, so the committed tree's tests are red while the agent's full working tree is green. A genuine test failure — positively identified by a `--- FAIL`/`panic:` allowlist (`looksLikeTestFailure`, the inverse of the compile gate) — fails the stage **category B** (`gitops.ErrCommittedTestsFailed`, naming the failing package) and, running pre-push, leaves origin untouched. Test-infrastructure failures (`[build failed]`/`[setup failed]`, dep resolution, exec error) skip non-blocking (`test_gate_skipped`); the asymmetry deliberately favors skip, since a false block stalls the loop on gate infra while a false skip just misses one detection CI still catches. The test phase runs ONLY on the drift-present path (the `len(drift)==0` and empty-scope fast paths return first), bounding cost to the drift-excluded-test-failure class; an always-run broad verify is the #651 evaluator-optimizer's job, which builds remediation on this detection foundation.
   - **Pre-push created-out-of-scope gate (#818, generalized to the open-PR path by #825).** A net-new (untracked) file outside the approved `scope.files` is silently stripped from the scope-only commit by `StageScoped` (#581) while an in-scope edit that *references* it still lands, shipping a self-inconsistent, misleadingly-green partial result. The same `gitops.CommitAndPushArgs.VerifyCommit` hook, before the compile gate, calls `gitops.UntrackedPaths` (`git ls-files --others --exclude-standard`) on the post-commit working tree to find the CREATED (untracked) subset of the scope drift; if non-empty it fails the stage **category B** (`gitops.ErrCreatedOutOfScope`, naming the files) BEFORE the push, so origin is untouched. The gate runs on the normal **open-PR implement push** and the **fix-up pass** (`cfg.fixup`, `fishhawk_fixup_stage`), but **not** on decomposed children (`cfg.fixup || (!isFixup && !isDecomposed)`) — a child may legitimately create files a later child declares. The two paths emit distinct events and wrap distinct sentinels: the open-PR path emits `created_out_of_scope`, wraps `gitops.ErrCreatedOutOfScope`, and directs the operator to amend the scope (name the files in the approval condition, or use the structured `add_scope_files` param once it exists) or replan; the fix-up path emits `fixup_created_out_of_scope`, wraps the `gitops.ErrFixupCreatedOutOfScope` specialization (which itself wraps `ErrCreatedOutOfScope`, so the single `errors.Is(err, gitops.ErrCreatedOutOfScope)` classification check matches both), and the backend's #788 fix-up recovery then restores the run to its pre-fix-up review gate so the operator hand-applies the named files or starts a fresh run with corrected scope. Only CREATED files trip this — modified-but-out-of-scope drift stays flag-only (ADR-027) — and a *legitimately-declared* new file is IN `scope.files` and therefore not in the drift set, so the gate only ever trips on net-new files OUTSIDE the declared scope. Making the fix-up tool itself succeed for a legitimately-needed new file (a fix-up-scoped allow-list) is deferred.
   - **Committed-tree verify-fix loop (#651).** When `executor.verify.max_iterations > 0`, the implement push path (`stageType == "implement" && !cfg.noPR`) runs an optional bounded evaluator-optimizer fix loop (`runner/cmd/fishhawk-runner/main.go::runVerifyFixLoop`) AFTER constraint evaluation and AFTER the ADR-023 self-retry `for{}` loop settles — closing gap G2 (execution-grounded verification) by remediating on top of the #728/#800 detection foundation. The verify command (`executor.verify.command`, served as `verify_command` with `verify_timeout_seconds` / `verify_max_iterations` on the prompt response, operator `--verify-cmd`/`--verify-timeout`/`--verify-max-iterations` win when set) runs against the isolated committed SCOPE-ONLY tree — the **same** drift-excluded HEAD the #728/#800 compile+test gate checks, reached via a fresh `git worktree add --detach` at a throwaway-commit SHA (`runVerifyCommittedTree`, reusing `runVerifyGate`'s `Setpgid` + `cmd.Cancel` process-group SIGKILL so `go test` grandchildren can't hold the output pipe open and block `CombinedOutput`) — **NOT** the dirty working tree, so a drift-excluded test failure (a necessary fake dropped as scope drift, #780/#776) surfaces here exactly as it would in CI rather than false-greening. Each iteration stages scope-only (#581 `StageScoped`), makes a THROWAWAY local commit (`commitVerifyWIP`, bot identity, `--no-verify`, `gpgsign=false`) to materialize a committed-HEAD SHA, verifies, then `git reset --soft HEAD~1` (`gitResetSoftHEAD1`) to undo it — `--soft` moves HEAD without touching the index or working tree, so the converged edits + staged scope survive for either the real push or the next iteration. On PASS it breaks and the push proceeds (the real `CommitAndPush` in `openPRAndShipArtifact` makes the single push commit and runs its own #728/#800 gate); on FAIL with budget remaining it re-invokes the agent in-place with a fix prompt embedding the captured verify output (`verify_fix_reinvoke` log line) and the next iteration re-commits scope-only onto the SAME base — one converged commit, never a stack of fix commits on a bad base. The fix re-invocation is itself bounded against a transient agent-API/transport failure (#804): an `invoker.Invoke` non-nil error is RETRIED IN PLACE up to `maxFixInvokeInfraRetries` (=2) total attempts within the same outer iteration WITHOUT advancing `iter`, so a transient blip (e.g. a #798 Claude usage-limit) does not burn a fix-loop budget unit by re-verifying an unchanged tree — worst-case total fix-Invokes stays hard-bounded at `O(max_iterations × maxFixInvokeInfraRetries)`. Each failed attempt emits a `verify_fix_reinvoke_error` log line + trace event (never silently swallowed), and only a SUCCESSFUL invoke (`fixErr == nil`) folds its events/tokens/model into `res`; exhausting all attempts is an infra failure routed to the non-blocking skip (below), never category-A. on FAIL with the budget exhausted (`iter == max_iterations`) it demotes the stage to **category-A** and SKIPS the push. **Terminal, non-compounding (DECISION c2, 2026-06-04):** the loop lives OUTSIDE the self-retry `for{}` loop by construction, so a verify-fix exhaustion can never reach the `continue` that calls `client.RetryStage` — total verify-fix agent invocations are capped at `max_iterations + 1` (initial invoke + up to `max_iterations` fixes) with no multiplicative compounding against the ADR-023 self-retry budget. Each fix re-invocation's `Result.Events` are appended to `res.Events` and its token usage folded into `res` (every *other* `invoker.Invoke` site replaces `res` wholesale; here append is the correct handling), and the OTel stage span (`EmitStage`) is emitted AFTER the loop settles, so a multi-iteration run produces a complete audit trace and an honest cost (ADR-030). A `verify_run` trace event is emitted per attempt (carrying the committed `head_sha`, alongside `command`/`exit_code`/`output`/`outcome`) and a single `verify_summary` event (`{outcome: passed|failed|skipped, iterations, max_iterations}`, emitted EXACTLY ONCE on every exit path — `skipped` carries a `detail` and is the outcome on the infra-abort/non-blocking-skip path, which previously mislabelled itself `failed`, #804 Gap 2) once the loop settles — both are runner *trace* events (like the existing working-tree `verify_run`), NOT Notifier issue-comment surfaces or audit kinds. **Single-shot committed-tree gate (#802).** Default `max_iterations == 0` on the implement push path (`stageType == "implement" && !cfg.noPR`) no longer runs against the dirty working tree — it now runs the configured verify command ONCE against the committed SCOPE-ONLY worktree via `runVerifyGateCommitted` (the same #651 scaffolding: `StageScoped` + throwaway `commitVerifyWIP` + `runVerifyCommittedTree` + `gitResetSoftHEAD1`), demoting to **category-B** by wrapping `gitops.ErrCommittedTestsFailed` on a non-zero exit. This is the language-agnostic single-shot twin of the Go-specific #728/#800 committed-test gate, closing the residual #651 did not cover: at `max_iterations == 0` the old path false-greened the drift-excluded class (#780/#776) for any non-Go verify command. The classifier is exit-code-only (the sole universal signal for an arbitrary command); the configurable `executor.verify.skip_on` escape hatch is deferred. Pre-commit infra errors (`StageScoped`/`commitVerifyWIP`/rev-parse, HEAD never moved) are non-blocking skips, but a POST-commit `gitResetSoftHEAD1` failure is FATAL (a swallowed reset would leave HEAD on the throwaway commit and push it into the PR). Only `--no-pr` (dirty working tree, no committed tree/PR to gate) and non-implement stages keep the single-shot **working-tree** demote-on-failure gate (#441 `runVerifyGate`, **category-A**), which is gated off on every implement-push committed-tree path so it can't double-fire inside the self-retry loop. The three verify guards thus partition the `verifyCmd != ""` space with no overlap: working-tree gate = `plan || --no-pr`; committed single-shot gate = `implement && !noPR && max_iterations == 0`; committed fix loop = `implement && !noPR && max_iterations > 0`. An in-loop git/stage infra failure — or a fix re-invocation that fails on infra across all `maxFixInvokeInfraRetries` attempts (#804) — is a non-blocking skip (`verify_fix_skipped`, `verify_summary` outcome `skipped`) — never invent a failure from gate plumbing; the stage proceeds to the real push and its own #728/#800 gate. **Wall-clock bound (ADR-030):** worst-case wall clock is `(max_iterations + 1) × executor.timeout + (max_iterations + 1) × verify.timeout`; `max_iterations` is the control surface and ties to the periodic cost budget. Distinct from `tests_added_or_updated` (touch-check) and `fishhawk_verify_run` (audit-chain integrity).
   - **Empty captured diff → retryable category C (#691/#692).** A *genuine* constraint hit (forbidden/allowed paths, max_files, ci-not-green) stays category B. But an implement stage whose bundle carries a *present-but-empty* `git_diff` event (zero changed files, the #691 staging-bug signature) fails only the `tests_added_or_updated` required-outcome — a capture/staging miss, not a policy breach. The trace handler (`backend/internal/server/trace.go::noDiffCaptured`) reclassifies that exact case to a retryable category C with a `no_diff_captured:` reason, so the existing `failed → pending` retry path gives operators an escape hatch (an un-redrivable B would also doom a fan-out parent). The carve-out is strict: any other violation — notably an `invalid pattern` config error a malformed glob emits even on an empty diff — keeps the failure as category B. A bundle with *no* `git_diff` event at all remains the `no_diff_in_bundle` skip→pass path, untouched.
6. **Review stage** — Backend awaits human approval on the PR plus blocking checks (`ci_pass`, `fishhawk_audit_complete`).
7. **Merge** — On merge, backend writes the final audit entry and closes the run.

Failure categories (per §6) are captured in the `audit_entries` table with `category` ∈ {A: agent, B: constraint, C: infra, D: approval timeout}. Re-execution is allowed for all four; idempotency keys prevent double-fire.

### 4.1 Self-retry decision tree (ADR-023)

When `executor.agent_self_retry: true` is set on a stage in the workflow spec, the runner applies a four-gate decision after each invocation before exiting with failure:

1. **Spec opt-in** — `cfg.agentSelfRetry == true` (surfaced via the prompt-fetch response). Stages without the flag always exit on failure.
2. **Signing key available** — `issuedKey != nil`. Only the `--fetch-prompt` path issues the key before the invoke loop; without it the gate closes and no retry fires. Guarantees the runner can authenticate the POST.
3. **Remaining budget > 0** — `maxRetriesSnapshot - retryAttempt - selfRetryCount > 0`. `maxRetriesSnapshot` and `retryAttempt` are snapshotted from the run row at prompt-fetch time and returned in the prompt response. `selfRetryCount` tracks in-process retries on the current runner invocation.
4. **Failure category A or C** — category B (constraint/policy) and D (approval timeout) are not retried in-process; those require a spec change or human action.

**Call path on retry**: runner calls `POST /v0/stages/{id}/retry` (signed with the Ed25519 per-run key, same mechanism as trace upload). The backend transitions the stage `failed → pending` and the orchestrator fires `workflow_dispatch`, launching a second GitHub Actions runner. Meanwhile the current runner process emits a `stage_self_retry` log line, resets its result state, and re-invokes the agent in the same for-loop iteration.

**Known v0 limitation**: the `workflow_dispatch` fires a second GHA runner that also fetches the stage prompt (the dispatched state is not blocked by the prompt handler's state guard) and begins a concurrent agent invocation. The two agents race; whichever uploads its trace bundle last wins at the backend's idempotent-store layer. No v0 guard prevents this; the orphaned runner's trace upload races with the winner's. An integration test asserting only one trace-upload call per retried stage is deliberately deferred.

### 4.2 Plan-review-agent invocation (ADR-027)

When a stage's `reviewers.agent > 0` is configured in the workflow spec, the backend invokes that many review agents inline inside `handleShipPlan` (in `backend/internal/server/plan.go`), after the plan artifact is stored and validated but before the stage transitions to `awaiting_approval`. Each review agent receives the plan artifact and the originating issue body via the `plan_review` prompt (`buildPlanReview` in `backend/internal/prompt/prompt.go`) and is instructed to return only a structured verdict JSON — re-planning and proposing edits are explicitly forbidden.

**Invocation flow:**

1. Plan artifact POSTed to `POST /v0/runs/{id}/plan?stage_id=…`.
2. Backend verifies signature, validates against `standard_v1`, stores in `artifacts`.
3. If `stage.reviewers.agent > 0`: spawn N review-agent invocations synchronously, each receiving the plan artifact + issue body via `prompt.Build("plan_review", …)`.
4. Self-review guard: if the review agent's model matches `plan.GeneratedBy.Model`, log WARN (ADR-027 §5 — no hard block in v0).
5. For each review response, decode the `ReviewVerdict` JSON, append a `plan_reviewed` audit entry (category string `"plan_reviewed"`, payload `PlanReviewedPayload` from `backend/internal/planreview/review.go`).
6. Authority check: if `ResolveAuthority(reviewers) == AuthorityGating` (i.e. `agent>0 && human==0`) and any verdict is `reject`, the stage stays in its current state rather than advancing to `awaiting_approval`.
7. Otherwise `handleShipPlan` drives the plan stage's terminal advance (`advancePlanStageTerminal`): `running → awaiting_approval`, or `running → succeeded` + an orchestrator `Advance` for a gateless plan stage.

**Plan-stage advancement invariant** (#603): a plan stage's terminal transition is driven by a valid `standard_v1` plan artifact landing via the plan-upload handler (`handleShipPlan`), **not** by trace upload alone. The runner ships both trace variants before the plan, so `advanceStageAfterTrace` (`backend/internal/server/trace.go`) leaves a plan stage in `running` until `planArtifactExists` confirms a plan is stored — guarding against a gated plan stage reaching `awaiting_approval` with nothing to review when the later plan-ship fails `standard_v1` validation. On an invalid plan, `handleShipPlan` fails the stage category-B (`running → failed`) and calls `advanceAfterFailure` so the run walks to terminal `failed` rather than stranding. The transition is idempotent (same-state re-application is a repo-layer no-op), so a future plan-first upload ordering — where the trace handler finds the artifact and advances first — does not double-fault. When `ArtifactRepo` is unwired (minimal config), `planArtifactExists` returns true so the gate is a no-op and the prior trace-driven advance is preserved.

**Push-and-open-pr implement advancement invariant** (#742): mirroring the plan-stage gate, an implement stage that will open a PR has its terminal transition driven by the **`/pull-request` upload** (`handleShipPullRequest`), **not** by trace upload alone. The runner ships the trace *before* it commits/pushes/opens the PR, so advancing to `awaiting_approval` on trace upload would strand the run at the review gate with a null `pull_request_url` if the PR-open step then fails (the b6811dc9 zombie). The runner stamps `push_and_open_pr` in the bundle manifest (`bundle.ManifestData` / `bundle.Manifest`, both `omitempty`, lockstep across the two modules — see the trace-bundle manifest row in §10) for every standalone implement stage (not `--no-pr`, not a decomposed child). `advanceStageAfterTrace` (`trace.go::pushAndOpenPRGated`) then leaves the stage in `running` when the flag is set **and** the bundle carries a non-empty `git_diff` (a commit + PR-open will follow) — emitting the `runtime_observed` calibration row before the early return, since the bundle timing is only available at trace time. The `/pull-request` upload becomes the authoritative terminal driver: the **success** body advances `running → awaiting_approval`/`succeeded` (+ orchestrator `Advance` for a gateless stage) after stamping `run.pull_request_url` (`advanceImplementStageAfterPR`); the **failure** body (`{outcome:"failed", category, reason}`, `failPullRequestStage`) fails the stage (category `C` retryable / `B` re-scope) + advances + appends a `pull_request_failed` audit entry, so the run never reaches review with a null PR. The runner reports that failure from the implement post-processing block (`main.go::reportPullRequestFailure`, gated on the same standalone-implement condition) when `openPRAndShipArtifact` errors — unifying #728's `ErrCommitWouldNotCompile` (category B) into this report path. An **empty** diff is the no-changes carve-out: the agent made no edits, opens no PR, and never POSTs `/pull-request`, so `pushAndOpenPRGated` returns false and the stage advances via the trace handler as before (gating it would hang the stage in `running`). Non-implement stages and a false flag (older bundles, `--no-pr`) are byte-identical to the prior trace-driven advance. **Decomposed-child push gate (#771)**: a decomposed child opens no PR (the parent run opens one consolidated PR, ADR-032) but still commits + pushes onto the shared branch *after* the trace ships, so the same zombie hole applied — a child push failure left the child stage at terminal `succeeded` with no code on the shared branch, which the childcompletion sweeper would then consolidate into a PR silently missing that child's work. The runner stamps a separate `push_to_shared_branch` manifest flag (mutually exclusive with `push_and_open_pr`) for decomposed children; `trace.go::childPushGated` mirrors `pushAndOpenPRGated` to leave the child stage in `running`, and the `/pull-request` upload drives the terminal transition via a new `{outcome:"pushed", branch, head_sha, base_sha}` success variant (`succeedChildPushStage`: running → terminal + `child_pushed` audit entry, no PR artifact) or the existing `{outcome:"failed"}` variant on a push failure (category `C` retryable via the #698 parent-park). **Fix-up push gate (#794)**: a fix-up re-dispatch (#762) opens no new PR (it commits onto the EXISTING PR branch *after* the trace ships) but the same zombie hole applied in its worst form — a fix-up stamped *neither* `push_and_open_pr` *nor* `push_to_shared_branch`, so it terminalized at trace upload *before* its commit/push, and a push or `#728` compile-gate failure was silently swallowed: the stage stayed `succeeded`, the implement re-review approved an *unlanded* diff, and the #788 fix-up recovery never fired (the stage never went terminal-failed). The runner now stamps a third mutually-exclusive `push_fixup` manifest flag for fix-up passes; `trace.go::fixupPushGated` mirrors `childPushGated` to leave the fix-up stage in `running` (the advisory implement re-review still fires at trace time on the bundle diff — only the TERMINAL transition is deferred), and the `/pull-request` upload drives the terminal transition via a new `{outcome:"fixup_pushed", branch, head_sha, base_sha}` success variant (`succeedFixupPushStage`: running → terminal + `fixup_pushed` audit entry, no PR artifact, idempotency-guarded on `(stage_id, head_sha)`) or the existing `{outcome:"failed"}` variant on a push/compile-gate failure — which now genuinely reaches `failPullRequestStage → maybeRecoverFixupFailure`, restoring the run to its pre-fix-up review gate with the original mergeable PR intact. **Dangling-PR close on a gating implement-review reject (#877)**: a gating agent implement-review (`reviewers.agent>0`, `human==0`) runs SYNCHRONOUSLY during the raw-trace upload (`advanceStageAfterTrace::runImplementReviews`) and fails the implement stage category-B with reason `implement_review_rejected: …` (hoisted to the const `implementReviewGatingRejectReason`) BEFORE the runner — which has no view of that verdict — opens its PR and POSTs `/pull-request`. By then the stage is terminally failed, so `handleShipPullRequest` records the PR artifact + `pull_request_opened` audit + URL backfill (the honest open) but `advanceImplementStageAfterPR` no-ops (stage not running), leaving the rejected change with an open PR that will never merge. After the backfill and BEFORE the lineage guard, the handler detects that exact state — implement + `failed` + category-B + the `implementReviewGatingRejectPrefix` reason prefix (same source of truth as the trace failure site; other category-B sources are excluded) — and calls `closePRAfterGatingReject`: it posts a short explanatory PR comment, closes the PR via the new `githubclient.ClosePullRequest` (`PATCH .../pulls/{number}` body `{"state":"closed"}`, mirroring `ForceUpdateRef`), and writes a `pull_request_closed_after_review_reject` audit entry (system actor, payload `{run_id, stage_id, artifact_id, pr_number, pr_url, failure_reason}`). Fail-open throughout: GitHub unconfigured, a nil `installation_id`, an unparseable repo, or a close error WARNs and skips — the stage is already failed, so a failed close must never 500 the handler. Advisory reviews dispatch detached and never reject-fail the stage, so they never reach this state and are byte-identical (PR-open timing for the advisory flow is unchanged; ADR-036/#876 enforces at the landing gate, not at PR-open).

**Authority resolution** (`backend/internal/planreview/review.go::ResolveAuthority`):

| `reviewers.agent` | `reviewers.human` | Mode |
|---|---|---|
| > 0 | == 0 | `gating` — agent rejection blocks stage advancement |
| > 0 | > 0 | `advisory` — agent verdict recorded but human gate is authoritative |
| == 0 | any | `gateless` — no agent review; today's default |

**Verdict types** (closed set per ADR-027): `approve`, `approve_with_concerns`, `reject`.

**Audit category `plan_reviewed`**: one entry per review-agent invocation, payload shape `PlanReviewedPayload{ReviewerKind, ReviewerModel, Authority, Verdict, Concerns[], FreeForm}`. Surfaced by `fishhawk_get_plan` (MCP tool) in `GetPlanOutput.Reviews[]`.

**Audit category `plan_review_started`** (#600): a single best-effort entry emitted by `runPlanReviews` at dispatch — **only** on the branch where a reviewer will actually run (`reviewers.agent > 0` AND a `PlanReviewer` is wired), never for the gateless (`agent == 0`) or nil-reviewer (`plan_review_skipped`) branches. Payload shape `ReviewStartedPayload{ConfiguredAgents, Authority}` (`backend/internal/planreview/review.go`). It is emitted synchronously before the per-reviewer loop, so its audit sequence always precedes every `plan_reviewed` entry under both gating (synchronous) and advisory (detached) authority. This is the MCP-readable proxy that lets a consumer distinguish *configured + running* (a `started` entry with no terminal entry) from *none configured* (no entry at all) without re-fetching the workflow spec — the basis for the `review_status` = `pending` state surfaced by `fishhawk_get_run_status` / `fishhawk_get_plan` and waited on by `fishhawk_await_review` (see the MCP row in §10 "Where to look").

**Local-mode wire-up** (`backend/internal/claudecode/`, #575). The local-mode adapter is the dogfood sibling of `anthropic.Reviewer`: instead of provisioning an API key, it spawns the operator's existing `claude` CLI as a subprocess (`claude --print --output-format json --model <model> -p <prompt>`), inheriting `os.Environ()` so the operator's subscription auth or already-present `ANTHROPIC_API_KEY` is reused with zero new plumbing. It decodes the CLI's single JSON envelope, reads the `result` field, and validates the verdict against the same closed set (`approve` / `approve_with_concerns` / `reject`) as the SDK adapter.

The retry budget (`FISHHAWKD_PLAN_REVIEW_MAX_RETRIES`, see the env-var table) is wired through a post-construction `(*claudecode.Reviewer).SetMaxRetries` call rather than `claudecode.Config.MaxRetries` directly: `NewClient` normalises a zero `MaxRetries` to 1, so passing the env value through the constructor could never express *disable*. `serve.go` therefore constructs the reviewer, then calls `SetMaxRetries(*planReviewMaxRetries)` to assign the env-resolved value past that normalisation. The result: unset → 1 (one retry, unchanged from #620's code default), an explicit `0` → retry disabled (single attempt), `N` → N+1 attempts, and a negative value clamps to 0.

Recommended local config for dogfooding (copy-pasteable):

```sh
export FISHHAWKD_ENABLE_LOCAL_CLAUDE_REVIEWER=true
export FISHHAWKD_LOCAL_CLAUDE_MODEL=claude-sonnet-4-6
export FISHHAWKD_PLAN_REVIEW_TIMEOUT=300
```

Large-plan expectation (#606): small plans review in ~60–90s, but large backend `standard_v1` plans (10+ files) need the wider budget — the old 60s default timed them out and the agent verdict never landed. Because advisory plan review is async/detached (#584), a longer budget blocks nothing: it only lengthens the worst-case `pending` window that re-polling `fishhawk_get_run_status` (the poll-authoritative terminal source, #879) surfaces via `poll_interval_seconds`, and that `fishhawk_await_review` (default 360s since #878, the optional non-stranding convenience block per ADR-036 #874) already returns gracefully (#600). The override above is redundant with the new 300s code default but kept in the copy-pasteable block for explicitness. This is the genuine timeout-on-large-plans mode only; the separate ~1s immediate-crash mode is owned by #620.

The review model (`claude-sonnet-4-6`) deliberately differs from the plan **author's** typical default (Opus): the plan-review prompt's self-review guard keys on the model identifier, so reviewing with a different model than authored keeps the self-review WARN meaningful rather than firing on every plan. The adapter returns the configured `FISHHAWKD_LOCAL_CLAUDE_MODEL` verbatim as the verdict's model identifier (the CLI envelope does not reliably echo the model), keeping that guard deterministic.

Accepted local-mode trade-offs vs. the SDK adapter: (a) softer verdict-shape guarantee — the subprocess has tool access via the `claude` CLI and only the prompt forbids repo grepping, so a non-compliant agent degrades to a per-call `reject` rather than silent corruption; (b) self-review WARN noise if the review model matches the author model (mitigated by the recommended config above); (c) no prompt caching — the full prompt is sent as one `-p` argument with no system/user split; (d) ~1–2s process-spawn overhead per review.

**Codex reviewer wire-up** (`backend/internal/codex/`, #844). The Codex adapter is the OpenAI sibling of `claudecode.Reviewer` and the reviewer analog of the runner's Codex **executor** (`runner/internal/agent/codex`, #840/#842): it spawns `codex exec --json --skip-git-repo-check <prompt>` for inference-only review and reads one JSON event per line from stdout. The verdict body is the final `item.completed` line whose nested item is an `agent_message` (its `text`); per-turn token usage is summed from each `turn.completed` line's `usage` block (`input_tokens` already includes `cached_input_tokens`; `reasoning_output_tokens` is added to the output side). The event shapes are pinned against the installed Codex CLI (codex-cli 0.137.0, the same version the executor pinned) and encoded in the fake-binary test fixtures so a future CLI drift fails a test rather than silently dropping the reviewer. `--skip-git-repo-check` is required because the backend has no checkout; a real inference-only run executes no tools, so — unlike the executor — it needs **no** `--dangerously-bypass-approvals-and-sandbox` (confirmed against a real `codex exec --json` from a non-repo dir, which returns cleanly without blocking on approvals). The decoded `text` is run through the shared `planreview.DecodeVerdict` (#739 fence-strip + #889 invalid-escape tolerance), validated against the same closed verdict set, and the summed usage is attached with `Known=false` when no `turn.completed` usage line appeared (no silent $0, #681/#682). `OPENAI_API_KEY` is forwarded explicitly from `FISHHAWKD_OPENAI_API_KEY` on top of the inherited `os.Environ()`; an empty key is not an error (ChatGPT-login auth on the host is supported). The retry budget is wired through `SetMaxRetries` exactly as the claudecode adapter does. This enables the **Claude main, Codex reviewer** topology.

**Prompt-caching strategy**: the `plan_review` prompt (`buildPlanReview` in `backend/internal/prompt/prompt.go`) is split at `prompt.PlanReviewSplitMarker` (`"\n### Plan artifact\n\n"`). Everything before the marker — the role-constraint preamble — is placed in the Anthropic system block with `cache_control: {type: ephemeral}`. Everything from the marker onward (plan artifact + issue context + verdict schema) is placed in the user message. This keeps the cached prefix stable across calls for the same fishhawkd process lifetime while allowing the variable plan content to change per invocation.

**Nil-reviewer degradation (#574).** When a workflow declares `reviewers.agent > 0` but `server.Config.PlanReviewer` is nil (no `FISHHAWKD_ANTHROPIC_API_KEY` set — see Production wiring above), the behavior is two-tier, keyed on authority:

- **Gating mode** (`agent>0, human==0`): rejected before any run row is minted, on both entry points. `handleCreateRun` (`backend/internal/server/runs.go`) iterates the resolved spec's plan stages and, on an agent-gated stage with a nil reviewer, writes a `run_rejected_misconfigured` **global-chain** audit entry (`AppendGlobalChained` — no run row exists yet) and returns `400 plan_reviewer_unconfigured`. The webhook-dispatcher path (`Dispatcher.Handle`, gated on a plain `PlanReviewerConfigured` bool wired from `cfg.PlanReviewer != nil` to avoid the server→webhook import cycle) applies the symmetric guard as of #577: it writes the same `run_rejected_misconfigured` global-chain entry and refuses to dispatch. Because the dispatcher returns 202 to GitHub and cannot surface a 400, the audit entry plus a WARN log are the rejection surface (a customer-facing issue comment is a deferred follow-up). Either way a run that could never satisfy its own gate is never minted.
- **Advisory mode** (`agent>0, human>0`): allowed through — the human gate remains authoritative. At plan upload, `runPlanReviews` (`backend/internal/server/plan.go`) emits a `plan_review_skipped` audit entry and the stage continues to `awaiting_approval`.

**Audit category `plan_review_skipped`** (#574): emitted by `runPlanReviews` when the spec requested agent review (`reviewers.agent>0`) but no `PlanReviewer` is wired. Payload shape `ReviewSkippedPayload{Reason, ConfiguredAgents, Authority}` (`backend/internal/planreview/review.go`); `Reason` is `reviewer_not_configured`. Emitted for both gating and advisory authority so that any run which reaches plan upload with a nil reviewer is auditable rather than silent; in advisory mode this is the primary trail, and in gating mode it is a defensive backstop now that both create paths reject gating+nil before a run is minted. Surfaced by `fishhawk_get_plan` as a synthesized review with `verdict: "skipped"` and the recorded reason.

**Audit category `run_rejected_misconfigured`** (#574, #577): global-chain entry written when an agent-gated plan stage is rejected for a missing reviewer — by `handleCreateRun` on the run-create path and by `Dispatcher.Handle` on the webhook-dispatcher path. Payload names the offending stage, workflow_id, repo, and configured agent count; the dispatcher additionally carries `delivery_id` and `event` for GitHub-trigger correlation. On the dispatcher path, when the trigger is an issue (labeled or `/fishhawk run`), the refusal is also surfaced to the customer as an issue comment via `IssueNotifier.NotifyRunRejected` (#599) — the audit entry above stays the canonical operator-side record, the comment is best-effort customer-facing UX naming both fixes (configure the server reviewer, or change the stage `reviewers`).

A startup WARN fires at `fishhawkd` boot (`backend/cmd/fishhawkd/serve.go`) whenever the resolved `PlanReviewer` is nil (neither the API key nor the local-mode flag is set), describing both degradation modes; it disappears once either adapter is wired.

**Review dispatch: advisory-async vs. gating-sync (#584).** Both `runPlanReviews` and `runImplementReviews` resolve the stage's review config on the request goroutine (cheap reads: `GetRun`, `resolveStageReviewers`, plan parse/load, prompt build), then split on authority:

- **Advisory** (`agent>0, human>0`): the per-reviewer loop is dispatched on a **detached** context (`context.WithoutCancel(ctx)`) in a goroutine tracked by `Server.bgReviews`, and the handler returns its response (`201`/`202`) immediately without waiting on the reviewer. This is the fix for the reviewer being SIGKILLed at the runner's upload-client timeout: the review previously ran on `r.Context()`, which the server cancels when the runner's client closes the connection (default was 60s), so `context.WithTimeout(r.Context(), FISHHAWKD_PLAN_REVIEW_TIMEOUT)` fired at `min(parent, budget) = 60s` and `exec.CommandContext` killed `claude`. Detaching lets the review run to its own size-aware review budget (#747; floor `FISHHAWKD_PLAN_REVIEW_TIMEOUT`, scaled by prompt size). Because the detached `reviewCtx` carries no deadline, that per-invocation budget — applied at the call site via `context.WithTimeout(reviewCtx, ReviewBudget.Budget(len(promptText)))` and honoured by the adapter — is the only deadline in the chain (`context.WithTimeout` takes the earlier of parent/child, so a larger budget can't extend a shorter parent that doesn't exist here). The advisory verdict is non-blocking (the human gate stays authoritative), so it landing after the stage reaches `awaiting_approval` is correct.
- **Gating** (`agent>0, human==0`): the review loop runs **synchronously** on the (still detached) context. Synchronicity is required because a reject must transition the stage to `failed` (category B) *before* the trace handler's terminal `awaiting_approval` advance — the block relies on the `awaiting_approval → failed` (plan path) / `running → failed` (implement path) state-machine edges. The detached context still removes the client-disconnect clamp; it just doesn't return early.

**Graceful-shutdown drain.** `Server.Shutdown` waits on `bgReviews` after `http.Shutdown` returns, bounded by the same shutdown context, so in-flight advisory reviews finish (or are abandoned at the deadline) rather than being dropped. `Server.waitBackgroundReviews()` is the test sync point for deterministic async assertions. The runner's upload-client default timeout was also raised 60s→120s as defense-in-depth (`runner/internal/upload/upload.go`), but the decoupling — not the bump — is the real fix; a retried upload short-circuits at `GetByHash`.

**Plan-approval completion gate (ADR-036 / #875).** Because an advisory plan review is dispatched **detached** (above), the plan stage reaches `awaiting_approval` while the reviewer is still running — so a human could approve before reading the agent's verdict. `checkPlanReviewSettled` (`backend/internal/server/approvals.go`) closes that window: on a plan-stage **approve**, when the stage declares `reviewers.agent > 0` AND a `plan_review_started` entry is present AND fewer than the configured agent count of TERMINAL verdicts have landed, it refuses with `409 agent_review_pending` (`details`: `stage_id`, `configured_agents`, `landed_terminal`). ANY terminal kind counts toward the unblock — `plan_reviewed`, `plan_review_failed` (the #747 budget-kill emits this), or `plan_review_skipped` — so a timed-out or skipped reviewer never strands the gate. The gate is placed **before** `ApprovalRepo.Submit`, so a refused approval records no row and the operator's poll-then-retry (idempotent) flows normally once verdicts land. A hard backstop — `planReviewBackstop(agentCount) = ReviewBudget.Cap × max(1, agentCount)`, measured from the earliest `plan_review_started` — ALLOWS the approval once elapsed even with no terminal verdict, writing a `plan_review_backstop_elapsed` audit entry (actor `system`; payload `{stage_id, configured_agents, landed_terminal, started_at, elapsed_seconds}`), so a reviewer that dies emitting nothing can never brick approvals. Every read fails OPEN (WARN-log, allow) mirroring `checkPlanBudget`. `reviewers.agent == 0` (no agent reviewer) and a configured-but-not-dispatched stage (no `plan_review_started`) are byte-for-byte the pre-ADR-036 path. Surfaced through `fishhawk_approve_plan` as a typed poll-until-landed error carrying the landed/configured counts (no auto-retry in the tool).

**Implement-review / merge completion gate (ADR-036 / #876).** The same completion-gate discipline extends to the implement-review (ADR-027) / merge gate, but at the **merge-driven resolution** path rather than an HTTP approve. `checkImplementReviewSettled` (`backend/internal/server/pullrequest_review_events.go`) gates `resolveReviewStageOnMerge`: on a merge (`merged==true`) of a run that has a review stage, when the IMPLEMENT stage declares `reviewers.agent > 0` AND an `implement_review_started` entry is present AND fewer than the configured count of TERMINAL `implement_reviewed`/`implement_review_failed`/`implement_review_skipped` verdicts have landed, it refuses to resolve the stage to `succeeded` and leaves the run **parked in `awaiting_approval`**. The merge reconciler (`backend/internal/mergereconciler/`) re-scans parked review stages each tick and routes them back through `ResolveReviewFromPollState → resolveReviewStageOnMerge`, so the gate retries identically on both the `pull_request.closed` webhook (one-shot) and the poll (retrying), and **composes with the #862 lineage re-check** already on the tick path (both must pass for the stage to resolve). ANY terminal kind unblocks (a budget-killed reviewer emits `implement_review_failed` — see `trace.go`), so a timed-out reviewer never strands the merge. A hard backstop reusing `planReviewBackstop(agentCount) = ReviewBudget.Cap × max(1, agentCount)` (measured from the earliest `implement_review_started`) ALLOWS the resolution once elapsed even with no terminal verdict, recording an `implement_review_backstop_elapsed` audit entry (actor `system`; payload `{stage_id, configured_agents, landed_terminal, started_at, elapsed_seconds}`) exactly once (the resolve completes the run, so the reconciler stops re-polling). Every read fails OPEN (WARN-log, resolve); `reviewers.agent == 0` and a not-dispatched stage are byte-for-byte the pre-ADR-036 merge resolution. **Behavioral change — deferred `pr_merged` audit:** the gate is placed BEFORE `writePRMergedAudit`, which DEFERS the `pr_merged` row from merge-observation time to resolution time so a held run does not write a duplicate `pr_merged` row on every re-poll tick. A GitHub-merged run that is held by this gate therefore has NO `pr_merged` row until the re-poll clears the gate; the backstop guarantees the run always eventually resolves, so `pr_merged` is always eventually recorded. **Accepted limitations** (mirror ADR-036 / #862): the terminal count is run-wide rather than head-SHA-matched (exact for the common `agent=1` case; squash-merge SHA divergence makes per-SHA matching unreliable), and this gate observes a merge GitHub already performed — it refuses to mark the run `succeeded` but does not block the GitHub-side merge; the cosmetic dangling-open-PR residue is tracked in #877. The `merged==false` (closed-without-merge → `cancelled`) branch is **never** gated — an abandoned PR is terminal regardless of review state.

### 4.2.1 Implement-review-agent invocation (ADR-027 impl 2/2)

The same review-agent machinery extends to the **implement** stage: after the implement-stage diff lands in the trace bundle, if the implement stage's `reviewers.agent > 0`, the backend spawns that many review agents against the diff + approved plan + issue. This reuses the verdict shape, `ResolveAuthority` table, self-review WARN, and `PlanReviewer` interface verbatim — no interface change. The single new prompt variant is `implement_review` (`buildImplementReview` in `backend/internal/prompt/prompt.go`).

**Where the hook runs.** Unlike plan-review (which runs in `handleShipPlan`), implement-review runs in the **trace handler** (`backend/internal/server/trace.go::advanceStageAfterTrace`), because the diff is what's reviewed and it arrives via the trace bundle (the same `bundle.ExtractDiff` the policy re-eval uses, so it works in local `--no-pr` mode too — the diff lives in the trace regardless of whether a PR was opened). Ordering within the trace handler:

1. Manifest `agent_failed` check (fail-A short-circuit) → policy re-evaluation (`reEvaluatePolicy`).
2. `advanceStageAfterTrace`: `dispatched → running`.
3. **Implement-review hook** (guarded by `stage.Type == implement`): extract the diff via `bundle.ExtractDiff`; if extraction errors, skip review and proceed. Otherwise call `runImplementReviews`.
4. Terminal transition: `running → awaiting_approval` (or `succeeded` for gateless).

The hook is placed between step 2 and step 4 so a **gating** reject fails the stage to `failed` (category B) *before* the terminal transition — the terminal `awaiting_approval` advance is then blocked by the state machine, exactly mirroring `runPlanReviews`' contract. **Advisory** implement review is dispatched asynchronously (#584; see **Review dispatch: advisory-async vs. gating-sync** above): `runImplementReviews` returns `false` immediately and the terminal transition proceeds while the detached reviewer runs, so the upload is never held open for an advisory review. When `stage.Type != implement` or `reviewers.agent == 0`, the hook is a no-op and the trace path is byte-identical to the pre-ADR-027 flow.

**`runImplementReviews`** (`backend/internal/server/trace.go`) mirrors `runPlanReviews`: it resolves the implement stage's `ReviewersConfig` via the shared `resolveStageReviewers(runRow, spec.StageTypeImplement)`, loads the approved plan (self-review guard + prompt context), builds the `implement_review` prompt with the changed-files diff summary, invokes `PlanReviewer.Review` per configured agent, WARNs on self-review (the approved plan's `GeneratedBy.Model` is used as an approximation of the implement author's model — v0 does not record the implement agent's model separately), and appends one `implement_reviewed` audit entry per invocation. It returns `true` (gating block) only when authority is gating (`agent>0 && human==0`) and at least one verdict is `reject`.

**Decomposition fan-out review coverage (#677)**: `runImplementReviews` carries **no** `decomposed_from` guard — it fires on every implement-stage trace upload and resolves reviewers from the child run's own `WorkflowSpec` via `resolveStageReviewers(StageTypeImplement)`. So each fan-out child slice **already** receives its own per-child implement-review (with the #703 orthogonal lens) whenever the workflow's implement stage configures reviewers — no new code; the acceptance criterion "a fan-out child slice receives at least one advisory review" is satisfied by the existing machinery, not deferred (verified live: child run `1cd5ae2e`, sub-plan 1 of #688, emitted `implement_review_started → implement_reviewed(approve_with_concerns)` on 2026-06-02). What a child gets **no** per-child equivalent of is the **plan** review — by design it has no plan stage: the parent plan is reviewed pre-fan-out, and (per #677) the parent's binding approval-conditions now propagate into each child implement prompt via `resolveApprovalConditions` (see "Per-stage prompt construction"). The parent's own review stage remains the post-`children_settled` aggregate gate over the combined child branch.

**Review lens is re-aimed at the orthogonal gaps** (#703): the `implement_review` prompt no longer re-verifies plan adherence or generic-bug-hunts — mechanical correctness is already gated by the policy gate, the implement agent's test run, build/lint, and CI, so re-stating it adds no signal. Instead it applies the three lenses the deterministic gates are blind to: security / authz (anchored to Fishhawk's code-execution threat model — lethal trifecta / uncontrolled egress, ADR-029 / #650), test vacuity (a test that asserts nothing load-bearing still passes CI), and untested error / edge / concurrency paths. The security lens is **self-gating**: on a low-risk diff that touches no sensitive surface it states so briefly and stops rather than manufacturing a concern. The reviewer stays advisory (no authority change); the grounded-citations and style-is-lint criteria (#595) are retained.

**scope.files drift is flag-only** (ADR-027 Decision Q6): the prompt instructs the review agent to emit a `{category: "scope"}` concern when the diff touches files outside the plan's `scope.files`, but drift alone never forces a `reject`. Only an overall verdict of `reject` blocks under gating authority. A reviewer returning `approve_with_concerns` with only a scope concern advances the stage.

**Operator-stageable scope_drift is surfaced to the reviewer** (#695, #581): the review's `git_diff` is **scope-bounded** — it excludes paths the implement stage dirtied but did not declare, which the operator may `git add` into the final commit. The runner already emits those paths as a `scope_drift` `policy_event` in the trace bundle (from `computeAndEmitDiff`); the trace handler now reads them via `bundle.ExtractScopeDrift` and threads them onto `prompt.Trigger.ScopeDrift`, so `buildImplementReview` renders a "Scope drift (excluded from the diff above — operator may stage)" section naming each path. This stops the advisory reviewer false-rejecting a required test/doc as "absent from the committed artifact" when it actually landed via a drifted path. A **standing anti-false-reject rule** backs this up even when the event is absent or stripped (best-effort extraction degrades to `nil`): the reviewer must treat an absence it cannot positively confirm as unverifiable (→ `approve_with_concerns`), never confirmed-missing. The reviewer sees only the drifted path **names**, not their content (the bundle's scoped diff excludes drift content), so it can flag-vs-false-reject but cannot test-vacuity-check a drifted test file — an inherent limit of the backend-only approach.

**Audit category `implement_reviewed`**: one entry per invocation, payload shape `ImplementReviewedPayload{ReviewerKind, ReviewerModel, Authority, Verdict, Concerns[], FreeForm}` (identical to `PlanReviewedPayload`). This is an internal audit category, not an issue-comment surface (no `docs/issue-comment-surfaces.md` entry). Surfaced by `fishhawk_get_run_status` (MCP) in `GetRunStatusOutput.ImplementReviews[]`.

**Audit category `implement_review_started`** (#600): the implement-stage twin of `plan_review_started`, emitted by `runImplementReviews` at dispatch under the same agent>0-AND-wired guard and ordering invariant (started precedes every `implement_reviewed` entry). Same `ReviewStartedPayload` shape. Internal audit category, not an issue-comment surface. It backs the `review_status` = `pending` state for the implement stage on `fishhawk_get_run_status` and `fishhawk_await_review`.

**Nil-reviewer / skip behavior** mirrors `runPlanReviews` exactly: an `implement_review_skipped` entry (payload `ReviewSkippedPayload`, reason `reviewer_not_configured`) is emitted only when `reviewers.agent > 0` but no `PlanReviewer` is wired.

**`review_action_hint` (#777)**: a DISPLAY-ONLY timing nudge surfaced on `fishhawk_get_run_status` (and on `fishhawk_run_stage` after an implement stage) computed entirely in the MCP layer (`backend/cmd/fishhawk-mcp/review_action_hint.go::reviewActionHintFor`) from audit data it already reads — no backend API, persistence, or schema change. When the implement `review_status` is `complete` with one or more unresolved `approve_with_concerns` concerns (summed across reviewers from the `implement_reviewed` entries) AND the bounded fix-up budget is not yet spent (counting prior `stage_fixup_triggered` entries against the mirrored `maxFixupPasses`, kept in sync with `server.defaultMaxFixupPasses`), it returns `{concerns, remaining_fixup_budget, message}` pointing at `fishhawk_fixup_stage` (route the concerns back to the agent) vs approving to merge. It is `omitempty` and never gates — mirroring the periodic-budget block (#693/#759). Suppressed once the concerns are resolved or the fix-up budget is exhausted (while `maxFixupPasses == 1`, budget-exhaustion suppression also covers the resolved-after-fix-up case). Not surfaced on `fishhawk_start_run` (no implement review exists at run start) nor on plan/review `fishhawk_run_stage` calls. In `getRunStatus` the hint is derived from the SAME `implement_review_status` already computed for the response, so the two cannot disagree within one read.

**Audit categories `plan_review_failed` / `implement_review_failed`** (#664): the terminal entry written when a *wired* reviewer invocation errors or times out (a reviewer killed at `FISHHAWKD_PLAN_REVIEW_TIMEOUT` surfaces as a `Review` error). Emitted per failed invocation by the shared `Server.emitReviewFailed` helper from both review-error branches (`runPlanReviewLoop` in `plan.go`, `runImplementReviewLoop` in `trace.go`), payload `ReviewFailedPayload{Reason, ReviewerModel, Authority, Timeout}`. The `Timeout` bool (#747, `omitempty`) is the additive discriminator that tells a budget-deadline kill (`context.DeadlineExceeded` at the call site) apart from a transport/decode failure; the audit category is unchanged so existing #664 and MCP `await_review` readers are unaffected. Kept distinct from `*_reviewed` (which carries only the closed approve/approve_with_concerns/reject verdict set) so a timeout is never decoded as a real verdict. Internal audit categories, not issue-comment surfaces. They back the MCP `review_status` = `failed` terminal state on `fishhawk_get_run_status` and `fishhawk_await_review` (precedence `complete > skipped > failed > pending > none`), resolving what previously fell through to an ambiguous `pending`. **Observability-only**: emitting this entry does NOT change gating advance/degrade semantics — a gating reviewer that times out still degrades to advance per the #574 advisory-degradation decision (`hasRejection` is untouched).

**Prompt-caching strategy**: the `implement_review` prompt is split at `prompt.ImplementReviewSplitMarker` (`"\n### Diff under review\n\n"`), same pattern as `PlanReviewSplitMarker` — the role-constraint preamble is the cacheable prefix, the variable diff/plan/issue content follows.

## 5. Storage model

### 5.1 Postgres (`fishhawkd` schema)

Designed under E2.1 (#22). Tables (immutable schema once frozen at Day 21):

- `runs` — one row per workflow execution. `(run_id, repo, trigger_source, workflow_sha, state, created_at, …)`
- `stages` — one row per stage execution. `(stage_id, run_id, kind, executor, state, started_at, ended_at)`
- `artifacts` — typed outputs. `(artifact_id, stage_id, kind, schema_version, data_jsonb, content_hash)`
- `signing_keys` — per-run ephemeral key chain. `(run_id, public_key_b64, issued_at, expires_at)`. Immutable.
- `audit_entries` — append-only event log. `(entry_id, run_id, ts, category, kind, actor_subject, payload_jsonb, prev_hash)`. Application-layer enforces no UPDATE/DELETE.
- `approvals` — `(approval_id, stage_id, approver_subject, surface, decision, ts)`.
- `sessions` — opaque session IDs for the Web UI cookie auth.
- `api_tokens` — scoped CLI/API tokens with revocation state.

Connection pool: `pgxpool.Pool` per service instance.

### 5.2 S3 (trace bundles)

- Bucket-per-environment: `fishhawk-traces-{env}`.
- Key layout: `{run_id}/redacted/{sha256}.jsonl.gz` and `{run_id}/raw/{sha256}.jsonl.gz`.
- Bucket policy denies `s3:DeleteObject` to all principals except a dedicated lifecycle-management role.
- Object Lock (Compliance Mode) on `raw/` is gated by retention SLA decisions (post-v0).
- Local dev: MinIO container at port 9000, same key layout.

### 5.3 Trace bundle format

`*.jsonl.gz`, one JSON event per line, UTF-8. Schema versioned via the manifest event's `bundle_schema` field (current: `"v1"`).

| Event kind | Position | Purpose |
|---|---|---|
| `manifest` | First line | Schema version, run/stage IDs, agent/model identity |
| `prompt` / `tool_call` / `tool_result` / `model_response` | Middle | The actual trace |
| `policy_event` / `gate_event` | Middle | Constraint hits, approval transitions |
| `error` | Middle | Errors during agent execution (category A) |
| `agent_retry` | Middle | Boundary marker the runner's Claude Code driver emits between attempts when it re-spawns the agent after a transient thinking-block API 400 (payload: `attempt`, `reason`, `tokens_so_far`). See the bounded in-driver retry row in §10. |
| `out_of_tree_write` | Middle | Warning the Claude Code driver emits when an agent file-writing tool call (`Write`/`Edit`/`MultiEdit`/`NotebookEdit`) targets a path outside the working tree + `/tmp` allowlist (payload: `path`, `tool`, `run_id`, `stage`). Purely additive surfacing — never flips `Result.OK`, never fails the stage, fail-open on parse error. Catches the #601 Write/Edit-tool class only; Bash-mediated `>` writes are NOT detected (full confinement is the OS-sandbox ADR's domain). See the out-of-tree write detection row in §10. |
| `trailer` | Last line | `event_count`, content hash of preceding lines |

**Not a bundle event — `stage_progress` liveness heartbeat (#580)**: during agent invocation the Claude Code driver writes a single-line `{"event":"stage_progress","elapsed_seconds":N,"turns":N,"tokens_so_far":N,"last_event_kind":"…"}` to the runner's **logSink (stderr) only**, every ~15s (`claudecode.Invoker.HeartbeatInterval`, default 15s). It carries coarse structural counters — never agent payload text — so a long stage is visibly progressing. It is **never** appended to `Result.Events`, so the signed trace bundle is unchanged (same treatment as the logSink-only `runner_cancelled` / `stage_self_retry` lines). The `fishhawk-mcp` `fishhawk_run_stage` relay forwards it as a `notifications/progress` update whose message renders the counters; because it's time-driven, a stalled stage still emits heartbeats with non-advancing `turns`/`tokens_so_far`, letting the driver tell "alive and progressing" from "stuck".

## 6. Invariants

These are load-bearing. Do not break them without explicit ADR.

1. **Customer source code never reaches Fishhawk's backend.** Code lives only in the customer's repo and on their CI runners. The backend sees traces, plans, and metadata — never source.
2. **Audit entries are append-only.** Enforced at the application layer (`audit.Append(...)`; no `Update` or `Delete`). Static-analysis test asserts no other code path mutates `audit_entries`.
3. **Trace bundles are signed by per-run ephemeral keys.** A tampered runner cannot forge a trace for any run other than its own; a leaked key expires within 30 minutes.
4. **Honest gap reporting beats fictional completeness.** A runner crash produces an audit entry that says `trace_lost`, not silent omission (MVP_SPEC §6).
5. **The workflow spec syntax is frozen at Day 21.** Old plans in the audit log remain readable forever; we never break old `standard_v1` artifacts.
6. **No founder bypass.** Methodology commitment from `docs/METHODOLOGY.md`. Emergency paths exist, are themselves audited, require post-hoc justification.

## 7. Module boundaries

The Go monorepo is a workspace, not a single module. Each top-level directory is independently taggable:

- `/backend` — `github.com/kuhlman-labs/fishhawk/backend`. Internal packages only (no exported API to other modules).
- `/runner` — `github.com/kuhlman-labs/fishhawk/runner`. The published GitHub Action artifact (composite action manifest at `runner/action.yml`). Customers pin a tag.
- `/cli` (planned) — `github.com/kuhlman-labs/fishhawk/cli`. Single binary.

Cross-module type sharing is intentionally avoided in v0. If two modules need the same struct (e.g., the `standard_v1` plan schema), the canonical source is a JSON Schema in `/docs/spec/` and each side parses independently. This keeps the runner's dependency graph small and the supply-chain surface minimal.

The frontend (`/frontend`, planned) is its own npm workspace; it talks to the backend over the REST API only.

## 9. CI and release shape

- **CI workflow** at `.github/workflows/ci.yml`. Path-aware via `dorny/paths-filter`. The Go job iterates `go.work` `use` directives — adding a new module Just Works.
- **Lint config** at `/.golangci.yml` (v2 format). Shared across all Go modules in the workspace.
- **Coverage targets**, tiered to the autonomy levels in `docs/METHODOLOGY.md`:
  - **Low-autonomy code** (audit log integrity, signing/crypto, policy evaluator, run state machine, workflow spec parser): ≥ 85% statement coverage.
  - **Medium-autonomy code** (HTTP handlers, runner adapters, REST endpoints, UI logic): ≥ 75%.
  - **High-autonomy code** (docs, dep bumps, lint/format): no target.
  - **Generated code** (sqlc outputs, etc.): excluded from numerator and denominator.
  - **Aggregate floor (excluding generated): ≥ 80%.** Enforced by `scripts/check-coverage.py` in the Go CI job; PRs that drop below fail `CI Pass`. Run locally with `(cd backend && go test -race -coverprofile=coverage.out -covermode=atomic ./...) && python3 scripts/check-coverage.py --threshold 80 --exclude internal/run/db backend/coverage.out`.
- **Release**: each module is tagged independently. The runner is the customer-facing one — `kuhlman-labs/fishhawk/runner@v0.1` etc. — built with signed releases + SBOM (E5.7 / #54, E13.6 / #63).

Env: `FISHHAWK_API_TOKEN` required, `FISHHAWK_BACKEND_URL` defaults to `http://localhost:8080`. Uses the `github.com/modelcontextprotocol/go-sdk/mcp` SDK over stdio. **Auth split**: runner-side `fhm_*` tokens carry `mcp:read` scope only — write tools called from the runner-side hit the bearer middleware, resolve a read-only identity, and the handler-side role check returns 403 (the SDK surfaces it as a tool error, no code change needed). Operator-side `fhk_*` apitokens carry `write:runs` / `write:approvals` / `write:stages` scopes. **Schema-reflection trap (#371)**: the SDK auto-generates output schemas via Go reflection over the tool's return type. `uuid.UUID` (which is `[16]byte`) renders as `type: array`, and `json.RawMessage` (which is `[]byte`) does the same, but on the wire UUIDs are strings and per-category audit payloads are JSON objects — schema validation rejects every response. `client.go` types UUIDs as `string` and `Artifact.Content` / `AuditEntry.Payload` as `any`; callers `uuid.Parse` at the API-client boundary and the plan decoder re-marshals + unmarshals into the typed `PlanContent`. Release pipeline at `.github/workflows/mcp-release.yml` (E19.7 / #347) — triggered by `mcp/v*` tags; cross-builds darwin-arm64 / darwin-amd64 / linux-amd64 / linux-arm64; cosign-signed `SHA256SUMS`; operator install path at `docs/mcp/install.md`. **Cross-component E2E test** (#371): `backend/internal/integration/mcp/e2e_test.go` spins up Postgres + the real backend HTTP server + builds + spawns the actual `fishhawk-mcp` binary as a subprocess, fetches an `fhm_` token via the runner's HTTP shape, and exercises a tool call end-to-end. The revocation + malformed-token assertions assert at the `mcptoken.Repository.Authenticate` layer rather than the tool call because v0 reads don't enforce identity (the bearer middleware falls through to anonymous; per-handler enforcement is out of scope for v0). **`fishhawk_start_run` field parity (#426)**: the start_run tool accepts the same local-runner convenience inputs the CLI's `fishhawk run start` does — `working_dir` walks for `.fishhawk/workflows.yaml` and ships the bytes inline (#411), `issue` shells to `gh issue view` and caches the payload (#415), `runner_kind=local` tags the run for the local-runner backend (ADR-022 / #388). Spec discovery / gh fetch live in `backend/cmd/fishhawk-mcp/{spec_discover,issue_fetch}.go` — local copies of the CLI's helpers rather than a shared package (the cli → backend import direction precludes the inverse). Auto-flip rule: when `trigger_source` is defaulted (empty) and an issue resolves via `issue` or `trigger_ref=issue:N`, the MCP server flips to `github_issue`; an explicit `trigger_source` is preserved. **`fishhawk_run_stage` runner-subprocess tool ([ADR-024](https://github.com/kuhlman-labs/fishhawk/issues/433) / #434)**: an MCP tool that spawns `fishhawk-runner` on the operator's host so an agent can drive a stage end-to-end through MCP without a terminal handoff. Mirrors the CLI's `fishhawk runner start` argv composition; stdout is parsed line-by-line as JSONL and either streamed as `notifications/progress` (when the client supplied a progress token) or accumulated for the final tool result. The audit log carries the durable record. **Compact-by-default result (#647)**: `summarizeRunStageEvents` walks the accumulated events once and (a) distils five scalar summary fields onto `RunStageOutput` — `outcome`/`tokens_used` from the terminal `{"event":"runner_completed","outcome":…,"tokens_used":N}` event (the only runner-level terminal event relayed on the JSONL **stderr** stream the relay reads; the bundle-only `kind=="invocation_end"` is deliberately *not* keyed on, as it never reaches this stream), and `turns`/`elapsed_seconds`/`last_event_kind` from the last `stage_progress` heartbeat — and (b) drops the routine `stage_progress` heartbeats from the returned `events` list while retaining every non-heartbeat event (`runner_completed`, `git_diff`, `runner_cancelled`, …) in arrival order. This roughly halves the driver's per-stage context cost; the signed trace bundle and audit log are untouched. `verbose:true` on the input restores the full heartbeat-inclusive list. The MCP `review_action_hint` (`backend/cmd/fishhawk-mcp/review_action_hint.go`) no longer suppresses on a spent budget when concerns remain: it surfaces the exhaustion plus `OverrideAvailable` and counts only the LATEST review round's concerns (scoped by the most-recent `stage_fixup_triggered` audit sequence). `POST /v0/stages/{stage_id}/fixup` (`backend/internal/server/fixup.go`) resolves the addressable concern set from the stage's `implement_reviewed` audit entries, validates the request's `concerns` indices against it, **bounds the pass by counting prior `stage_fixup_triggered` audit entries** (default 1 — no dedicated column), then re-opens and hands off to `Orchestrator.Advance` exactly like retry (pending → dispatched → workflow_dispatch). It writes a `stage_fixup_triggered` audit entry carrying the selected concern objects, which `server/prompt.go::resolveFixupConcerns` reads back to deliver the concerns to the agent as binding instructions (the #558 condition-delivery framing, rendered under `### Fix-up concerns` by `prompt.go`) and to fold any concern-named file into the effective `scope.files`. The runner commits the fix-up onto the SAME PR branch via the rebase-from-remote shared-branch path and UPDATES the existing PR (distinct from `/retry`'s fresh diff); it stamps `push_fixup` in the trace manifest so the backend forward-gates the fix-up stage's terminal transition (#794, see §4.2.1) and reports push success via a `/pull-request` `{outcome:"fixup_pushed"}` report (or `{outcome:"failed"}` on a push/compile-gate failure). **Auth**: `write:stages` OR `write:fixups`; a run-bound MCP token may fix up only its own run's stages (`403 cross_run_fixup`). MCP verb: `fishhawk_fixup_stage`. **#788 failure recovery**: a fix-up re-dispatch that FAILS must NOT destroy the intact original work (the PR is open and mergeable) — a fix-up is a best-effort optional pass on top of an already-succeeded implement + open PR. `run.RestoreFixupStage` (in `fixup.go`) restores the run to its pre-fix-up review gate via a dedicated `stageFixupRecoveryTransitions` table in `transition.go` — implement `failed → succeeded` (push_and_open_pr) / `failed → awaiting_approval` (commit-yourself) + review `pending → awaiting_approval` (re-park restore). That table is kept SEPARATE from `stageRetryTransitions`/`stageFixupTransitions` because admitting `failed → succeeded` must never leak into the ordinary path, where it would fake success; it is reachable only through `ValidStageFixupRecoveryTransition` (consulted by `TransitionStage`) and guarded by `RestoreFixupStage`. The server detector `server/fixup.go::maybeRecoverFixupFailure` keys off the durable `stage_fixup_triggered` audit entry (its `prior_state` + `reparked_review_stage_id`): since a fix-up re-opens from a HEALTHY gate, {entry present + implement stage failed} uniquely identifies a re-dispatch failure. It restores the gate, writes a `stage_fixup_recovered` audit entry (system actor; restored states + source failure category/reason), and returns true so the two implement-failure chokepoints — `trace.go::advanceAfterFailure` (cat-A agent-fail, cat-B implement-review reject, failStageCategoryB/C) and `pullrequest.go::failPullRequestStage` (the #742 PR-open / #794 fix-up push failure) — SKIP the run-failing `Orchestrator.Advance`, leaving the run `running` at its gate. **#794 closed the gap that made this recovery unreachable for fix-ups**: before #794 a fix-up stamped no forward-gate flag, terminalized at trace upload, and so never reached `failPullRequestStage` in a `failed` state — `maybeRecoverFixupFailure` could not fire. The `push_fixup` gate (above) defers the terminal transition so a fix-up push/compile-gate failure genuinely lands the stage `failed` and triggers recovery. With `defaultMaxFixupPasses == 1` a failed fix-up still consumes the budget, so the post-recovery operator path is "merge the original PR," not "re-fire the fix-up." Cross-component coverage: `backend/internal/integration/mcp/fixup_test.go` drives concern → MCP-triggered fix-up → re-open → binding-instruction prompt render → bound-exhausted refusal end-to-end, plus the #794 forward-gate seam end-to-end: a `push_fixup` trace upload leaves the fix-up stage `running` (terminal transition deferred), then the `/pull-request` `{outcome:"fixup_pushed"}` success report drives it terminal + writes a `fixup_pushed` audit entry, AND the `{outcome:"failed"}` re-dispatch-failure report restores the run to its review gate with a `stage_fixup_recovered` audit entry. |

| CSRF enforcement | `backend/internal/server/csrf.go` ships the double-submit pattern per ADR-005. The OAuth callback (`server.handleGitHubCallback`) mints a 32-byte hex token and sets it in the `__Host-csrf` cookie alongside `fishhawk_session`; logout clears both. The `csrf` middleware sits after `bearerAuth` in the chain (`recovery → requestID → logging → bearerAuth → csrf → mux`) and enforces `X-CSRF-Token` ≡ `__Host-csrf` on POST/PUT/PATCH/DELETE for session-cookie identities only — bearer-token clients (CLI, server-to-server) and GET-style methods bypass; safe-listed paths (`/v0/auth/github/*`, `/webhooks/github`) bypass too. Mismatch returns `403 csrf_required`. Frontend's `frontend/src/api/client.ts` reads the cookie via `getCookie()` (`frontend/src/lib/cookie.ts`) and auto-attaches the header on every state-changing call. Vitest runs jsdom under `https://localhost/` so `__Host-` cookies are accepted (jsdom rejects them under HTTP). |
