# Fishhawk — Architecture

> **Status:** Living document; tracks the current technical realization of the v0 spec.
> **Audience:** Agents and engineers working in this repo. Optimized for density, not narrative.
> **Authority:** `docs/MVP_SPEC.md` defines scope; this doc defines the *technical realization* of that scope. When they disagree, MVP_SPEC wins for "what" and this doc wins for "how".

## 1. System summary

Fishhawk is **a governed, auditable workflow layer for agent-driven software changes**. A customer commits a YAML workflow spec to their repo, triggers it (issue label, CLI, UI), and the system runs the agent on their CI under typed constraints, captures a signed trace, and gates each stage on human approval. The product surface is the workflow execution and audit history; the agent itself is pluggable. With ADR-038 (`workflow-v1`) the envelope extends past merge to a governed `deploy` stage. The invariant holds: Fishhawk holds no deploy logic and no prod credentials — the external pipeline runs under its own identity — and the product surface stays the gate + the signed deploy record, not the deploy mechanics.

See `docs/MVP_SPEC.md` §1–§4 for product framing, primitives, and customer-side spec syntax.

## 4. Workflow run lifecycle

Per `docs/MVP_SPEC.md` §5.2. Concrete realization in this codebase:

1. **Trigger** — GitHub issue label/assignment (webhook → backend), CLI `fishhawk run start`, or UI button. Backend validates the workflow spec at the issue's `.fishhawk/workflows.yaml` SHA, creates a `runs` row, and emits `workflow_dispatch` to the customer's repo invoking `fishhawk/runner@runner/vX.Y.Z`.
2. **Plan stage** — Runner checks out the repo, calls backend `POST /v0/runs/{id}/signing-key` (with GitHub OIDC token) and receives an Ed25519 private key + run metadata. Invokes Claude Code with the plan prompt. Captures full trace as JSON Lines events. Validates the plan artifact against `standard_v1` schema (E1.5 / #20). Signs `sha256(bundle)`. Ships `(bundle, sig)` to `POST /v0/runs/{id}/trace`.
   - **Structured-output-constrained plan artifact (E22.X / #1325).** The plan-stage invocation is structurally constrained by the claude CLI's `--json-schema` structured-output flag: the runner passes a runtime-derived schema (`runner/internal/plan.StructuredOutputSchema` — the embedded canonical `standard_v1` schema with all `$ref`s inlined against `$defs` and the
     CLI-subset-rejected keywords `format`/`$schema`/`$id`/`x-coerce-*` stripped; the never-drift discipline of #1324 — there is no second committed schema to fall out of sync) into `agent.Invocation.JSONSchema`, and adopts the terminal `result` event's top-level `structured_output` object (`agent.Result.StructuredOutput`) as the plan artifact. Adoption precedence is
     **clarification(file) > structured_output > agent-written file**: a `clarification_request` written to the plan-out file (the existing `detectClarificationRequest` file+`kind` channel — the CLI subset cannot express the plan-OR-clarification union as one schema, so parking stays on this channel) always wins and ignores any `structured_output`; otherwise the captured
     `structured_output` overwrites the plan-out file; the existing coerce+validate gate then runs against whatever sits there. Feature-gated entirely on `JSONSchema` being non-empty and honored only by the claudecode backend; a derivation error, a CLI that rejects the derived schema, or any non-claudecode backend leaves `structured_output` absent and the run **degrades
     gracefully** to the historical `TryCoerce`+validate+retry fallback — never a hard regression.
3. **Backend ingest** — Verifies signature against the stored public key for `run_id`. Stores the bundle in S3 keyed by content sha256. Persists the plan artifact in `artifacts` and renders it as a comment on the originating GitHub issue (mode: `rendered_comment`, kept in sync). Transitions stage state.
4. **Plan approval** — Approver reads the plan in the Web UI (canonical surface) or the issue comment (read-only echo). Clicks Approve. Backend records `(approver_subject, surface, ts)` in the audit log and transitions to implement.
5. **Implement stage** — Runner re-invokes with the implement prompt + approved plan. Captures trace. Post-hoc evaluates constraints (`forbidden_paths`, `max_files_changed`, `required_outcomes`) against the produced diff. If a constraint hits, the stage fails as category B. Otherwise pushes branch, opens PR via the GitHub App's installation token, ships final trace.
   - **Per-run working-tree isolation (E22.X / #1137, local loop only).** Immediately after the prompt fetch resolves `decomposed_from_run_id` — the first point the runner learns its lineage — `runner/cmd/fishhawk-runner/worktree.go` provisions a **lineage-keyed git worktree** and relocates `cfg.workingDir` into it, so concurrent runs driven on one local host no longer share a
     single working tree (the hosted GitHub Actions path is already per-job-isolated by `actions/checkout`). The worktree is a detached checkout of HEAD at `<git-common-dir>/fishhawk-worktrees/run-<root>`, where `root` is `lineageRoot(runID, decomposedFromRunID)` = the **parent** run id for a decomposed child (so all siblings share one lineage worktree) or the run's **own** id
     for a solo run (its own worktree). **Seed-ancestry guard (#1866, ADR-035 declared-base).** Because a FRESH worktree is seeded from the operator checkout's HEAD, a leftover unmerged commit on that HEAD (the MCP-cwd-default footgun) would contaminate the agent's tree, the reviewer diff, and the NoChanges probe — so before the `worktree add`, `verifySeedAncestry` refuses
     provisioning (`runner_failed` reason `working_dir_diverged_from_base`, an actionable message naming the offending HEAD SHA, the base tip SHA, and the recovery) when the seed checkout's HEAD is **not** an ancestor of (or equal to) the declared base's remote-tracking ref `refs/remotes/origin/<base>`. To close the check-then-add TOCTOU on the mutable operator checkout,
     provision resolves HEAD to a concrete SHA **once** (`resolveHead`) and both checks *and* seeds (`git worktree add --detach <path> <pinned-sha>`) that same immutable commit — so a HEAD that advances to a local-only commit in the window between the guard and the `worktree add` can no longer slip a diverged seed past the guard. The check is offline/auth-free by design (no
     fetch — `git push` already updates the tracking ref, so genuinely-pushed commits cannot false-refuse). It degrades to a logged skip — never a block — when evidence is unavailable (remote unconfigured, tracking ref unresolvable, or the ancestry probe itself errors → `lineage_worktree_base_guard_skipped` with reason `remote_unconfigured` / `base_ref_unresolvable` /
     `ancestry_probe_failed`), mirroring the #1302/#1363 not-wired-vs-transient degrade contract. The guard applies ONLY to fresh provisions: the **reuse** path (mid-lineage stages, decomposed siblings) returns before it, so a since-moved operator HEAD never refuses a stage of an already-seeded lineage (its worktree HEAD is legitimately the run branch, and its seed was
     validated at first provision). Under ADR-041 (E24.1 / #1141) each child no longer layers its commit onto a single shared `fishhawk/run-<parent>` branch; instead each cuts and pushes its **own sole-writer slice branch** `fishhawk/run-<parent>/slice-<n>` (n = the child's `slice_index`, the sub_plan position). The shared lineage worktree sequentially checks out each distinct
     slice branch in turn — E24.1 retains **serial** child dispatch (the lineage lockfile serializes same-lineage stages, so git never sees one branch checked out in two worktrees at once; parallel dispatch is E24.3+). Fan-in of the slice branches onto the consolidated branch is ADR-041 / E24.2, out of scope here. The common gitdir is resolved via `git rev-parse
     --git-common-dir` (not `--git-dir`) so an operator checkout that is itself a linked worktree still lands under the one shared root. Because every downstream git op re-derives `repoDir := cfg.workingDir` (the FreshFetchBase / checkoutChildBase / commit / verify-gate sequences), relocating that single field isolates the whole stage with no per-block change. An O_EXCL
     lockfile beside the worktree (`run-<root>.lock`) serializes same-lineage stages and **fails loud** (category-A) if a live process already holds it — decomposition drives a lineage's stages sequentially, so a concurrent same-lineage stage is a corruption hazard, not a legitimate state; a lock left by a crashed prior stage (pid no longer alive) is reclaimed. The worktree
     lives under `.git`, invisible to the operator's `git status`. This is the **`working_dir`-means-repo-root-to-derive-from** semantic shift (previously `working_dir` *was* the run's tree) and the realization of **ADR-035's branch-ownership invariant extended to tree-ownership**. Teardown is reclamation-by-lazy-sweep at the next provision: `sweepTerminalWorktrees` (run at
     provision start, before the new worktree is added) enumerates `git worktree list`, and for each `run-<root>` under `fishhawk-worktrees/` reads the full lineage-root run id from a `run-<root>.runid` sidecar (written at provision beside the worktree, since the short dir name can't be resolved back to a run id) and asks the backend whether that lineage is terminal via the
     additive `lineage_complete` field on `GET /v0/runs/{id}` (`upload.Client.RunLineageComplete`). `lineage_complete` is true when the lineage-root run AND every decomposed child are terminal (computed in `backend/internal/server/reads.go` from the `decomposed_from` child graph). **fishhawkd is the authority on lineage terminality only** (it can't reach the operator's
     filesystem); the runner performs the physical `git worktree remove --force` host-side. So a worktree for a terminal lineage lingers until the next provision on that host — acceptable: it lives under `.git`, is invisible to `git status`, and is cheap. The sweep is entirely best-effort (a missing sidecar, backend error, or git error logs a degradation event and is skipped;
     it never fails the stage and never removes a worktree it can't prove reclaimable). Relocating the run's commit off the operator checkout's HEAD broke the MCP server's `diff_summary` (which shelled `git show --numstat HEAD` in `working_dir`); the numstat totals now ride on the `git_diff` wire event (`insertions`/`deletions`) and `backend/cmd/fishhawk-mcp/run_stage.go` reads
     them from there, keeping the MCP server worktree- and decomposition-unaware. **Parallel children (E24.4 / #1144).** A default-off runner flag `--parallel-isolate` (`runner/cmd/fishhawk-runner/flags.go`) inverts the child keying: when set, `lineageRoot(runID, decomposedFromRunID, parallelIsolate)` returns the **child's own** id, so each concurrent sibling provisions an
     **isolated** `run-<child>` worktree (its own lock + `.runid` sidecar) instead of racing the one shared `run-<parent>` tree — the correct shape now that each child owns a distinct sole-writer slice branch (E24.1). The serial/solo paths are byte-for-byte unchanged when the flag is off. The MCP tool `fishhawk_run_children` (`backend/cmd/fishhawk-mcp/run_children.go`) is the
     driver that sets it: it reads the parent's `plan_decomposed` audit entry (`child_run_ids` + `effective_max_parallel`, 0 = unlimited), spawns each **pending** child's implement stage concurrently — `--parallel-isolate` appended — under an `errgroup` bounded by the orchestrator-resolved effective cap (clamp-DOWN-only against an optional `max_parallel` override), and **awaits
     ALL with no sibling-cancel** (a child failure is data in the consolidated per-child result, never a tool error; re-invocation is idempotent — only `pending` children are dispatched). The spawn/parse/SIGKILL-group core is the single `spawnRunnerStage` helper shared with `fishhawk_run_stage`. Per-child SIGKILL crash-survival is inherited from the same stale-lock-reclaim +
     lazy-sweep path (a child has no children, so its own terminality == `lineage_complete`).
   - **Pre-push compile gate (#728).** On the standalone (non-decomposed) PR-opening path, after the scope-only commit is created and **before** the push, the runner (`runner/cmd/fishhawk-runner/main.go::verifyCommittedTreeCompiles`, wired via `gitops.CommitAndPushArgs.VerifyCommit`) compiles the *committed* tree in an isolated `git worktree --detach` at the new HEAD SHA — not
     the working tree, which still carries the excluded scope-drift on disk — and runs `go vet ./...` per `go.work` module. `go vet` typechecks test files (which `go build` skips), so it catches the interface-ripple class where a scope-bounded commit drops a build-required conformance stub left in a test file (`does not implement <iface> (missing method ...)`). The block
     condition (#959) is **any clean compiler/typecheck diagnostic line** — `(vet: )?file.go:LINE:COL: message` (`goDiagnosticBlocks`), with the `compileDiagnosticMarkers` substring allowlist kept only as a backstop — unless every such line is a recognized dependency-resolution form (`no required module provides`, `missing go.sum entry`, `cannot find module/package`, `cannot
     query module`; dep failures print in the same `file.go:LINE:COL:` shape under a cold cache/offline, so the exclusion is load-bearing). The marker-allowlist-only era chased the open-ended typecheck-message tail one production bug at a time (#774, then #959's `unknown field ... in struct literal` skipping as `vet_nonzero_non_compile` and pushing an uncompilable head on run
     07bce059). A real compile failure fails the stage **category B** (`gitops.ErrCommitWouldNotCompile`, re-scope/re-plan) with the drift file list, and because it runs pre-push, origin is untouched — no non-compiling PR is opened. Every infrastructure case (no drift, non-Go repo, no Go toolchain, worktree/`go work edit` failure, a nonzero exit with no parseable diagnostic, or
     dep-resolution-only diagnostics) is a non-blocking skip (`compile_gate_skipped`), degrading to the prior "open the PR" behavior — but a skip now requires evidence the toolchain/dependency infrastructure failed, not merely an unrecognized diagnostic message; a pure vet analyzer finding consequently blocks too (CI's golangci-lint bundles govet, and a block routes to
     `verify_fix_reinvoke` rather than a dead stage). Decomposed children are intentionally ungated (a child may legitimately not compile before later children land); gating the consolidated decomposition tip is a follow-up.
   - **Pre-push test gate (#800).** After the `go vet` compile check passes, the same committed-HEAD worktree runs a second phase: `go test` (no `-race` — CI runs the race build; this is the fast touched-package check) on the *touched packages* — the packages containing the declared `scope.files`, derived per `go.work` module (`touchedPackageArgs` → `./<reldir>/...` patterns),
     **not** the drift dirs (drift is excluded from the committed tree). This catches the drift-excluded-test-failure class (#780/#776) where a scope-only commit compiles but a necessary test fake/helper was dropped as scope drift, so the committed tree's tests are red while the agent's full working tree is green. A genuine test failure — positively identified by a `---
     FAIL`/`panic:` allowlist (`looksLikeTestFailure`) — fails the stage **category B** (`gitops.ErrCommittedTestsFailed`, naming the failing package) and, running pre-push, leaves origin untouched. A `FAIL <pkg> [build failed]` **accompanied by a clean compiler diagnostic line** (`goDiagnosticBlocks`, #959) is a definitive compile failure of the tree being pushed, not infra —
     it blocks as `gitops.ErrCommitWouldNotCompile` (previously it skipped as `test_nonzero_non_failure`, the second half of the run-07bce059 miss). Only genuine test-infrastructure failures (`[setup failed]`, bare `[build failed]` with no diagnostic, dep resolution, exec error) skip non-blocking (`test_gate_skipped`); the skip-on-uncertainty asymmetry narrows to: uncertain
     means **no parseable diagnostic**, not an unrecognized diagnostic message — and a block routes to `verify_fix_reinvoke` (an in-place agent fix) rather than killing the stage, so the false-block cost that originally justified the wide skip no longer holds. The test phase runs ONLY on the drift-present path (the `len(drift)==0` and empty-scope fast paths return first),
     bounding cost to the drift-excluded-test-failure class; an always-run broad verify is the #651 evaluator-optimizer's job, which builds remediation on this detection foundation.
   - **Pre-push created-out-of-scope gate (#818, generalized to the open-PR path by #825).** A net-new (untracked) file outside the approved `scope.files` is silently stripped from the scope-only commit by `StageScoped` (#581) while an in-scope edit that *references* it still lands, shipping a self-inconsistent, misleadingly-green partial result. The same
     `gitops.CommitAndPushArgs.VerifyCommit` hook, before the compile gate, calls `gitops.UntrackedPaths` (`git ls-files --others --exclude-standard`) on the post-commit working tree to find the CREATED (untracked) subset of the scope drift; if non-empty it fails the stage **category B** (`gitops.ErrCreatedOutOfScope`, naming the files) BEFORE the push, so origin is untouched.
     The gate runs on the normal **open-PR implement push** and the **fix-up pass** (`cfg.fixup`, `fishhawk_fixup_stage`), but **not** on decomposed children (`cfg.fixup || (!isFixup && !isDecomposed)`) — a child may legitimately create files a later child declares. The two paths emit distinct events and wrap distinct sentinels: the open-PR path emits `created_out_of_scope`,
     wraps `gitops.ErrCreatedOutOfScope`, and directs the operator to amend the scope (name the files in the approval condition, or use the structured `add_scope_files` param once it exists) or replan; the fix-up path emits `fixup_created_out_of_scope`, wraps the `gitops.ErrFixupCreatedOutOfScope` specialization (which itself wraps `ErrCreatedOutOfScope`, so the single
     `errors.Is(err, gitops.ErrCreatedOutOfScope)` classification check matches both), and the backend's #788 fix-up recovery then restores the run to its pre-fix-up review gate so the operator hand-applies the named files or starts a fresh run with corrected scope. Only CREATED files trip this — modified-but-out-of-scope drift stays flag-only (ADR-027) — and a
     *legitimately-declared* new file is IN `scope.files` and therefore not in the drift set, so the gate only ever trips on net-new files OUTSIDE the declared scope. Making the fix-up tool itself succeed for a legitimately-needed new file (a fix-up-scoped allow-list) is deferred.
   - **Pre-push scope-completeness (shortfall) gate (#1151).** The inverse of the created-out-of-scope (#818/#825) and #980 commit-in-scope assertions: those prove every *committed* path is *declared*; this proves every *declared concrete* path is *committed*. The same `gitops.CommitAndPushArgs.VerifyCommit` hook, after the created-out-of-scope gate and before the compile gate,
     calls `gitops.MissingScopeFiles` — `git diff-tree -r -z --no-commit-id --name-only <head>` (the same NUL-separated, quotePath-immune enumeration `assertCommitInScope` uses) — to find the set of declared `scope.files` paths the commit did NOT touch, and returns the committed set alongside so the runner renders a precise `declared N scope file(s), committed M; missing:
     <paths>` message (each missing path annotated with its declared operation) and populates the `scope_files_missing` event's `committed` field from real data. A non-empty shortfall fails the stage **category B** (`gitops.ErrScopeFilesMissing`) BEFORE the push, so origin is untouched — catching the subset-PR class (#1148, run a4cfd41b: 8 declared scope files, only 6
     committed, dropping the two edits that actually fixed the bug; PR #1149 opened anyway and burned a full review + a no-op fix-up). The gate runs on the **standalone open-PR implement push ONLY** (`!isFixup && !isDecomposed`): a fix-up legitimately touches fewer files than the full scope, and a decomposed child carries a narrowed slice scope on a shared branch.
     Trailing-slash directory-prefix scope entries are skipped (`newScopeMatcher`'s split semantics — a folded directory cannot require any specific touched path), and a declared `delete`-operation path the commit actually deletes shows up as a changed path so it counts as touched. **Run/stage-scoped in-band self-exempt (#1153).** A deliberately-unchanged declared scope file no
     longer forces a replan: the implement agent may justify it in-band by writing a JSON sidecar to the **run/stage-keyed** path `prompt.ScopeJustificationPath(run_id, stage_id)` = `/tmp/fishhawk-scope-justifications-<run_id>-<stage_id>.json` (mirrored verbatim by the runner's `scopeJustificationPath`, the same independent-module coordination as `PullRequestDescriptionPath`),
     listing `{path, reason}` entries inside `{run_id, stage_id, exemptions:[…]}`. After the agent settles, `run()` calls `loadScopeExemptions` (standalone open-PR path only), validates **fail-closed** (well-formed JSON, embedded `run_id`/`stage_id` matching the stage, each path concretely declared, each reason non-empty), emits a single `scope_files_exempted` trace event
     (folded into `gate_evidence` for the implement reviewer AND surfaced to the audit), and threads the validated set into `openPRAndShipArtifact`, where the gate subtracts the exemptions from `missing` (`partitionExemptedMissing`) before the `len>0` check — an all-exempted shortfall passes, a partial one still fails listing only the unexempted remainder. Three independent
     freshness defenses stop a stale sidecar from a different run exempting this run's missing file: the path is **run/stage-keyed**, the embedded `run_id`/`stage_id` are **validated** against the stage, and the keyed path is **deleted pre-invoke** (`sweepStaleScopeJustification`) and on every `loadScopeExemptions` return (consumed-cleanup). Every
     malformed/stale/empty-reason/undeclared case yields ZERO exemptions, so the strict gate is preserved — a defect can only ever make the gate MORE strict, never bypass it. The self-exempt is inert for fix-ups and decomposed children (the gate is already excluded there, and the prompt section is gated on `ScopeConstraint == nil`). **Presence != correctness (#1169).** This is
     a PRESENCE check — it proves a declared path was *touched*, not that the required change was *made*, so a comment-only / no-op edit to a declared file passes (run 5aaf89fa: an explanatory comment was added to the adr numbering block instead of wiring the value, and the gate passed). The robust gate for conventions / config / default-value changes is a done-means
     **behavioral** test that asserts the shipped behavior and runs in the committed-tree verify (`gitops.ErrCommittedTestsFailed`) — now required by the plan-prompt "Done-means test rule" (`backend/internal/prompt`), which closes the gap from the planning end — and its sibling plan-prompt "Per-failure-mode test rule" (#1199) extends this from one behavioral test per change to
     one assertion per enumerated failure / fail-closed mode, paired with an always-on implement-prompt per-failure-mode test checklist; the agent review remains the backstop. The diff-size / content-shape heuristic (the issue's option 2) was considered and rejected as low-confidence and false-positive-prone per #1169; `gitops.MissingScopeFiles` stays a presence check by
     design.
   - **Stale-base 3-dot pre-merge gate (ADR-043 rev 2 / #1294).** The implement-stage policy diff (`runner/cmd/fishhawk-runner/main.go::computeAndEmitDiff`) — the single source the `git_diff` event's `StagedFiles` feeds, consumed by `gate_evidence` and the #1151 scope-completeness gate — now measures the staged index against the run's **fork point** (`git merge-base <baseRef>
     HEAD`, `gitdiff.Runner.MergeBase`) instead of the base-branch **tip**. The old 2-dot comparison (`git diff --cached <tip>`) read a file the base branch added **orthogonally after the run branched** as a phantom deletion (status `D`), because that file is in the tip's tree but absent from the run's index — inflating the staged-file count and reproducing #1290's
     16-staged-vs-7-declared implement-review failure. The 3-dot comparison diffs against the merge-base, which by construction omits any orthogonal post-fork base advance, so the gate sees only the run's own increment (matching how the eventual GitHub merge already composes). The resolution is **provider-agnostic** — a local `git merge-base`, no forge compare API or
     merge-queue feature — and **fail-open**: an unresolvable merge-base (unrelated histories, shallow clone, a base ref not yet fetched locally) emits a `merge_base_unresolved` degradation line and falls back to the original tip `baseRef`, i.e. today's exact behavior, never blocking the diff. The displayed `git_diff` `base_ref` label stays the human-meaningful branch
     (`main`/`origin/<shared-branch>`); only the commit-ish the staged index is *measured against* changes to the merge-base SHA, keeping the name-status, patch, and numstat internally consistent. The base the commit is cut from is itself kept as fresh as the reapply-before-commit invariant allows (`gitops.CommitAndPush`'s `FreshFetchBase` fetch sits as late as the `fetch →

Fishhawk does not take over the merge or rely on any forge merge-queue feature; a resulting main-break is detectable downstream (CI on `main`) and is out of scope for
     ADR-043 rev 2.

     on FAIL with the budget exhausted (`iter == max_iterations`) it demotes the stage to **category-A** and SKIPS the push. **Terminal, non-compounding (DECISION c2, 2026-06-04):** the loop lives OUTSIDE the self-retry `for{}` loop by construction, so a verify-fix exhaustion can never reach the `continue` that calls `client.RetryStage` — total verify-fix agent invocations are
     capped at `max_iterations + 1` (initial invoke + up to `max_iterations` fixes) with no multiplicative compounding against the ADR-023 self-retry budget. Each fix re-invocation's `Result.Events` are appended to `res.Events` and its token usage folded into `res` (every *other* `invoker.Invoke` site replaces `res` wholesale; here append is the correct handling), and the OTel

A `verify_run` trace event is emitted per attempt (carrying the committed `head_sha`, alongside `command`/`exit_code`/`output`/`outcome`) and a single `verify_summary` event (`{outcome: passed|failed|skipped, iterations,

     `commitVerifyWIP` + `runVerifyCommittedTree` + `gitResetSoftHEAD1`), demoting to **category-B** by wrapping `gitops.ErrCommittedTestsFailed` on a non-zero exit — except a failure matching the testcontainers start-timeout signature, which gets the same one-shot #972 absorb (re-run `runVerifyCommittedTree` once against the same throwaway headSHA before the reset, both
     `verify_run` events + the `verify_infra_flake_retry` event in the trace; only a second failure classifies). This is the language-agnostic single-shot twin of the Go-specific #728/#800 committed-test gate, closing the residual #651 did not cover: at `max_iterations == 0` the old path false-greened the drift-excluded class (#780/#776) for any non-Go verify command. The
     classifier is exit-code-only (the sole universal signal for an arbitrary command); the configurable `executor.verify.skip_on` escape hatch is deferred. Pre-commit infra errors (`StageScoped`/`commitVerifyWIP`/rev-parse, HEAD never moved) are non-blocking skips, but a POST-commit `gitResetSoftHEAD1` failure is FATAL (a swallowed reset would leave HEAD on the throwaway
     commit and push it into the PR). Only `--no-pr` (dirty working tree, no committed tree/PR to gate) and non-implement stages keep the single-shot **working-tree** demote-on-failure gate (#441 `runVerifyGate`, **category-A**), which is gated off on every implement-push committed-tree path so it can't double-fire inside the self-retry loop. The three verify guards thus
     partition the `verifyCmd != ""` space with no overlap: working-tree gate = `plan || --no-pr`; committed single-shot gate = `implement && !noPR && max_iterations == 0`; committed fix loop = `implement && !noPR && max_iterations > 0`. An in-loop git/stage infra failure — or a fix re-invocation that fails on infra across all `maxFixInvokeInfraRetries` attempts (#804) — is a
     non-blocking skip (`verify_fix_skipped`, `verify_summary` outcome `skipped`) — never invent a failure from gate plumbing; the stage proceeds to the real push and its own #728/#800 gate. **Wall-clock bound (ADR-030):** worst-case wall clock is `(max_iterations + 1) × executor.timeout + (max_iterations + 1) × verify.timeout`; `max_iterations` is the control surface and ties
     to the periodic cost budget. Distinct from `tests_added_or_updated` (touch-check) and `fishhawk_verify_run` (audit-chain integrity).

     `stage_state=succeeded` could vouch for a pushed head no gate ever saw (run 07bce059). Both gates now capture the verified commit's **tree object hash** (`git rev-parse <sha>^{tree}` — content-addressed: content + modes + paths, independent of commit metadata) before the reset and return it; the pre-push `VerifyCommit` hook in `openPRAndShipArtifact` compares the real
     commit's tree against it, ordered LAST after the more specific created-out-of-scope and compile/test gates. Equal trees pass for free (`verified_tree_match` log event); a mismatch (legitimate cause: `origin/<base>` moved between gate and push) logs `verified_tree_mismatch` and triggers exactly ONE strict re-verify of the verify command against the real committed HEAD in
     the isolated worktree — only an explicit `passed` outcome lets the push proceed (`pushed_tree_reverified`; an infra-skip is NOT a pass, complementing #959), anything else returns `gitops.ErrPushedTreeNotVerified` BEFORE the push (origin untouched, **category-B**, symmetric with `ErrCommitWouldNotCompile`/`ErrCommittedTestsFailed`). Fail-closed corollary: a PASSING gate
     whose verified tree cannot be resolved is an inconsistent state and returns the same sentinel rather than silently disabling enforcement; the empty-tree/no-enforcement convention is reserved for the no-gate-ran paths (plan stage, `--no-pr`, `verifyCmd` unset, gate skipped). Audit chain: `verify_run` trace events carry `tree_sha` alongside `head_sha`,
     `CommitAndPushResult.TreeSHA` carries the pushed commit's tree, and the `pull_request_opened` / `implement_fixup_pushed` / `implement_child_pushed` log events are stamped with `verified_tree_sha` + `tree_sha`, so the trail proves `verified_tree_sha == tree_sha` unconditionally (#969): on the reverify-pass path the hook rebinds the effective verified tree to the RE-VERIFIED
     pushed tree before the stamp sites read it, `pushed_tree_reverified` carries BOTH trees for forensics (`verified_tree_sha` = original gate tree, `tree_sha` = re-verified pushed tree — with `verified_tree_mismatch`, the only places the original gate tree's provenance survives), and the decisive re-verify emits its own `verify_run` structured log event
     (command/head_sha/tree_sha/exit_code/outcome, output omitted; emitted on failed re-verifies too — the trace bundle is already sealed and uploaded at push time per #742 forward-gating, so logSink is this record's channel).
   - **Index-authority staging + commit-side scope assertion (#980).** `StageScoped` (`runner/internal/gitops/commit.go`) opens with a mixed whole-tree `git reset -q` (index → HEAD, working tree untouched) before enumerating dirty paths, so its postcondition is **index == HEAD + exactly the declared dirty set** by construction. Without it, a file the agent PRE-STAGED with its
     own `git add` was never unstaged — `git commit` commits the index — so an out-of-scope file landed in the commit while `ScopeDrift` reported it as excluded (run 4c2c6374's 21MB accidental `fishhawk-runner` binary: the exclusion report disagreed with the commit's content). The same pre-staging blinded the #818 created-out-of-scope gate: `UntrackedPaths` uses `git ls-files
     --others`, which excludes index-resident files, so a staged net-new file was invisible and the run died later on the #960 tree-mismatch path instead of `ErrCreatedOutOfScope` naming the file — the reset makes the file untracked again and restores the gate's reach. Defense-in-depth backstop: on the ScopeFiles path (not the `add -A` fallback), `CommitAndPush` runs a
     **post-commit out-of-scope assertion** (`assertCommitInScope`) immediately after the commit is created — `git diff-tree -r -z --no-commit-id --name-only <head>` against the (always-parented) new commit, every changed path checked against the SAME declared-set matcher (`scopeMatcher`: exact paths + #824 trailing-slash dir prefixes) StageScoped staged with, one source of
     truth — and a violation returns `gitops.ErrCommitOutOfScope` naming each violating path BEFORE the push and before the VerifyCommit gates (origin untouched, **category-B**, symmetric with `ErrCommitWouldNotCompile`/`ErrPushedTreeNotVerified`). This is the commit-side complement of #960's tree equivalence: #960 proves the pushed tree is the gate-verified tree, #980 proves
     the committed content is the declared scope and nothing else, so the drift report can never again disagree with the commit. Advisory layer: at the `scope_drift` emit site the runner classifies each drift path with `isBinaryArtifactDrift` (executable with size > 5MiB, or the Go module-binary shape `cmd/<name>/<name>`) and emits a log-only `scope_drift_binary_artifact` event
     naming path + size — false negatives acceptable; the assertion and the #818 gate are the enforcement layers.
   - **Gate-phase credential posture (ADR-029 #650 item 4).** Every gate above executes *committed agent-authored code* — `go vet`/`go test` (and the `go work edit` enumeration) inside `verifyCommittedTreeCompiles`, and the spec `executor.verify.command` (`sh -c …`) in `runVerifyGate`/`runVerifyCommittedTree`. Those subprocesses run under a **stripped, default-deny allow-listed
     env** (`runner/cmd/fishhawk-runner/gateenv.go::sanitizedGateEnv`): only PATH/HOME/system essentials and the Go toolchain vars (`GO*`/`CGO_*`/CC/CXX) survive, so the runner's GitHub App installation token (`FISHHAWK_GITHUB_TOKEN`/`GITHUB_TOKEN`/`GH_TOKEN`), the agent API keys (`ANTHROPIC_API_KEY`/`OPENAI_API_KEY`), and the MCP backend token (`FISHHAWK_API_TOKEN`) are NOT
     visible to agent code — defusing the lethal-trifecta shape (agent code + network + runner creds). The allow-list is default-deny so a secret env var added to the runner later is dropped automatically; an explicit denylist is belt-and-suspenders on top. The git-plumbing exec sites (worktree add/remove, rev-parse, reset, add) keep the inherited env — they run on
     runner-controlled refs and need git credential context. A gate exec site added later must opt into `sanitizedGateEnv` to preserve this invariant. (`GOPROXY` and other GO* values are preserved for dependency resolution, but embedded URL userinfo — `scheme://user:pass@host` credentials an operator may put in a GOPROXY/GOSUMDB URL — is redacted per comma/`|`-separated entry
     by `gateenv.go::redactGoEnvUserinfo` before it reaches gate code, keeping the proxy host/path usable while closing the #940 leak; non-URL GO* forms pass through byte-identical.)
   - **Empty captured diff → retryable category C (#691/#692).** A *genuine* constraint hit (forbidden/allowed paths, max_files, ci-not-green) stays category B. But an implement stage whose bundle carries a *present-but-empty* `git_diff` event (zero changed files, the #691 staging-bug signature) fails only the `tests_added_or_updated` required-outcome — a capture/staging miss, not a policy breach. The trace handler (`backend/internal/server/trace.go::noDiffCaptured`) reclassifies that exact case to a retryable category C with a `no_diff_captured:` reason, so the existing `failed → pending` retry path gives operators an escape hatch (an un-redrivable B would also doom a fan-out parent). The carve-out is strict: any other violation — notably an `invalid pattern` config error a malformed glob emits even on an empty diff — keeps the failure as category B. A bundle with *no* `git_diff` event at all remains the `no_diff_in_bundle` skip→pass path, untouched.
6. **Review stage** — Backend awaits human approval on the PR plus blocking checks (`ci_pass`, `fishhawk_audit_complete`).
7. **Acceptance stage** *(workflow-v1.1, ADR-049 / ADR-050; optional stage)* — Runs after the review settles when the workflow declares an `acceptance` stage. The orchestrator dispatches it against a running preview instance (an `acceptance_dispatched` emit) — and before the agent spawns, the runner verifies that instance **is** the run's merge candidate: it probes the first
   declared `egress.target_hosts` entry's `/healthz` and fails the stage pre-spawn (category C) when the served `git_sha` build identifier does not match the prompt's `acceptance_expected_head_sha`, optionally provisioning the preview first via the `FISHHAWK_ACCEPTANCE_PREVIEW_CMD` hook (E31.18 / #1569; outcomes + fail reasons in the §10 "Acceptance preview provisioning" row,
   and the full build-system-agnostic hook contract for non-fishhawk build systems in `docs/acceptance-preview.md`). The runner's **contained** acceptance agent (three lethal-trifecta legs assembled and each bounded — see the §6 Rule-of-Two acceptance row for the egress/credential/authority posture, ADR-050) validates the *running* instance against the approved plan's
   `verification.acceptance_criteria` and ships a signed verdict to `POST /v0/runs/{run_id}/acceptance`, persisted as a durable `acceptance` artifact. A **failed** verdict leaves the stage `succeeded` (it is advisory, zero-write) and routes through deterministic server-side triage: class 1 (error / all-failed explicit-source criteria) → an implement fix-up; class 2 (no failed
   criterion, an ambiguous skip/flake) → an acceptance re-run (the succeeded stage is re-opened to pending); class 3 (a failed inferred-source criterion) and class 4 (unitemized) → paged to the human, with a `plan_review_miss` record on the class-3 path; class 5 (all-skip where every skip carries `expectation_basis` — a posture-A externally-unvalidatable can't-exhibit skip,
   #1671) → the terminal `externally_unvalidatable_paged` disposition, which takes NO transition (the stage stays succeeded so the merge gate can clear) instead of a futile class-2 retry loop. Auto-routing is bounded (a rerun/fix-up budget; exhaustion pages). The acceptance outcome and triage decision render on the run's living-anchor comment. Merge is gated on the
   `acceptance_passed` condition per ADR-049 decision #6 — an operator-loop `next_actions` gate, **not** a hard spec gate; an acceptance stage that no run dispatches is functionally inert.
8. **Merge** — On merge, backend writes the final audit entry and closes the run.
9. **Deploy** *(workflow-v1, ADR-038; optional stage)* — Runs after merge. Inverts the pre-merge model because the effect *is* the side effect: the approval gate is **pre-execution** (approve the intent — this sha → this environment, like a `plan` gate), and constraints are **pre-flight** (`allowed_environments`, `change_freeze`, `required_upstream`), *not* post-hoc against a produced diff. Fishhawk triggers and monitors an external pipeline (delegating mode; no prod credentials) and records a signed `deployment` artifact `{environment, ref/sha, external_run_url, outcome, rollback_handle}`.

Failure categories (per §6) are captured in the `audit_entries` table with `category` ∈ {A: agent, B: constraint, C: infra, D: approval timeout}. Re-execution is allowed for all four; idempotency keys prevent double-fire. ADR-038 adds `partial` / `rolled_back` terminal outcomes for the deploy stage. Unlike A–D, a partially-succeeded deploy is **not** safely re-executable — rollback is an explicit sub-action, not a blind retry. The pre-execution gate + pre-flight constraints are a distinct evaluation point and do not reuse the post-hoc diff-evaluation path.

### 4.1 Self-retry decision tree (ADR-023)

When `executor.agent_self_retry: true` is set on a stage in the workflow spec, the runner applies a four-gate decision after each invocation before exiting with failure:

1. **Spec opt-in** — `cfg.agentSelfRetry == true` (surfaced via the prompt-fetch response). Stages without the flag always exit on failure.
2. **Signing key available** — `issuedKey != nil`. Only the `--fetch-prompt` path issues the key before the invoke loop; without it the gate closes and no retry fires. Guarantees the runner can authenticate the POST.
3. **Remaining budget > 0** — `maxRetriesSnapshot - retryAttempt - selfRetryCount > 0`. `maxRetriesSnapshot` and `retryAttempt` are snapshotted from the run row at prompt-fetch time and returned in the prompt response. `selfRetryCount` tracks in-process retries on the current runner invocation.
4. **Failure category A or C** — category B (constraint/policy) and D (approval timeout) are not retried in-process; those require a spec change or human action.

**Call path on retry**: runner calls `POST /v0/stages/{id}/retry` (signed with the Ed25519 per-run key, same mechanism as trace upload). The backend transitions the stage `failed → pending` and the orchestrator fires `workflow_dispatch`, launching a second GitHub Actions runner. Meanwhile the current runner process emits a `stage_self_retry` log line, resets its result state, and re-invokes the agent in the same for-loop iteration.

**Known v0 limitation**: the `workflow_dispatch` fires a second GHA runner that also fetches the stage prompt (the dispatched state is not blocked by the prompt handler's state guard) and begins a concurrent agent invocation. The two agents race; whichever uploads its trace bundle last wins at the backend's idempotent-store layer. No v0 guard prevents this; the orphaned runner's trace upload races with the winner's. An integration test asserting only one trace-upload call per retried stage is deliberately deferred.

### 4.2 Plan-review-agent invocation (ADR-027)

When a stage's `reviewers.agent > 0` (or a heterogeneous `reviewers.agents` list) is configured, the backend invokes
that many review agents inline in `handleShipPlan` (`backend/internal/server/plan.go`) — after the plan artifact is
stored and validated, before the stage transitions to `awaiting_approval`. Each agent receives the plan artifact +
issue body via the `plan_review` prompt (`buildPlanReview`, `backend/internal/prompt/prompt.go`) and must return only
a structured verdict — re-planning and proposing edits are forbidden. The full server-side contract — adapter wiring
and precedence, env vars, review budgets, subprocess hardening, capability degradation, caching, and per-audit-category
payload shapes — lives in `backend/internal/planreview/README.md`.

**Invocation flow:**

1. Plan artifact POSTed to `POST /v0/runs/{id}/plan?stage_id=…`.
2. Backend verifies signature, validates against `standard_v1`, stores in `artifacts`.
3. If agent reviewers are configured: resolve each to an adapter (`ReviewerSet`), invoke with the `plan_review` prompt.
4. Self-review guard: reviewer model == `plan.GeneratedBy.Model` → WARN (ADR-027 §5 — no hard block in v0).
5. Decode each `ReviewVerdict`, append one `plan_reviewed` audit entry per invocation.
6. Gating authority (`agent>0 && human==0`): any `reject` fails the stage instead of advancing.
7. Otherwise `handleShipPlan` drives the terminal advance: `running → awaiting_approval`, or `running → succeeded`
   (+ orchestrator `Advance`) for a gateless plan stage.

**Authority resolution** (`backend/internal/planreview/review.go::ResolveAuthority`):

| `reviewers.agent` | `reviewers.human` | Mode |
|---|---|---|
| > 0 | == 0 | `gating` — agent rejection blocks stage advancement |
| > 0 | > 0 | `advisory` — agent verdict recorded but human gate is authoritative |
| == 0 | any | `gateless` — no agent review; today's default |

**Verdict types** (closed set per ADR-027): `approve`, `approve_with_concerns`, `reject`. The anthropic and codex
backends request schema-guaranteed verdicts (`planreview.VerdictSchema()`, #1324); `planreview.DecodeVerdict` is the
fallback decode path (and claudecode's primary).

**Advancement invariants** — the state machine, not the trace upload, decides when a stage lands:

- **Plan stage** (#603): the terminal transition is driven by a valid `standard_v1` plan artifact landing via
  `handleShipPlan`, never by trace upload alone (`advanceStageAfterTrace` holds a plan stage in `running` until
  `planArtifactExists`). An invalid plan fails the stage category-B and walks the run to terminal `failed`.
- **Implement stage that will push** (#742 family): the terminal transition is driven by the **`/pull-request`
  upload**, not trace upload — the runner ships the trace before it commits/pushes, so advancing on trace would
  strand a run whose push/PR-open then fails. Three mutually-exclusive bundle-manifest flags select the gate:
  `push_and_open_pr` (standalone implement), `push_to_shared_branch` (decomposed child, #771), `push_fixup`
  (fix-up pass, #794); the corresponding success variants (`opened`/`pushed`/`fixup_pushed`) or the failure variant
  drive the stage terminal. A gating implement-review reject that lands before the runner's PR-open triggers
  `closePRAfterGatingReject` (#877) so a rejected change never leaves a mergeable dangling PR.
- **Dispatch model** (#584): advisory reviews run detached (`context.WithoutCancel`, tracked by `Server.bgReviews`,
  drained on shutdown) so the upload response never waits on them; gating reviews run synchronously because a reject
  must fail the stage before the terminal advance.
- **Completion gates** (ADR-036): a plan-stage approve is refused `409 agent_review_pending` until all configured
  verdicts land (`checkPlanReviewSettled`, #875); merge-driven review-stage resolution is likewise held until the
  implement-review round settles (`checkImplementReviewSettled`, #876). Both count ANY terminal kind
  (`*_reviewed` / `*_review_failed` / `*_review_skipped`), are count-gated on the started entry's
  `ConfiguredAgents` (#1127), fail open on read errors, and carry a `ReviewBudget.Cap × agents` backstop so a
  reviewer that dies emitting nothing can never brick the gate.

**Audit categories** (payload shapes + emission rules in `backend/internal/planreview/README.md`):
`plan_review_started` (#600), `plan_reviewed`, `plan_review_failed` (#664), `plan_review_skipped` (#574/#1495),
`run_rejected_misconfigured` (#574/#577), `reviewer_capability_unavailable` (#1495), plus the plan-approval backstop
entry `plan_review_backstop_elapsed` (ADR-036).

### 4.2.1 Implement-review-agent invocation (ADR-027 impl 2/2)

The same machinery reviews the **implement** stage's diff. It runs in the trace handler
(`backend/internal/server/trace.go::advanceStageAfterTrace`) — the diff arrives via the trace bundle
(`bundle.ExtractDiff`), so it works in local `--no-pr` mode too. Ordering within the trace handler:

1. Manifest `agent_failed` check (fail-A short-circuit) → policy re-evaluation.
2. `advanceStageAfterTrace`: `dispatched → running`.
3. **Implement-review hook** (implement stages only): extract the diff; on extraction error skip review and proceed;
   else `runImplementReviews`.
4. Terminal transition: `running → awaiting_approval` (or `succeeded` for gateless).

The hook sits between steps 2 and 4 so a **gating** reject fails the stage category-B before the terminal
transition; **advisory** review dispatches detached and never blocks the upload. `runImplementReviews` mirrors
`runPlanReviews` (shared `resolveStageReviewers`, self-review WARN, one `implement_reviewed` entry per invocation)
and returns a gating block only on gating authority + at least one `reject`.

**Review lens** (#703): the prompt skips plan-adherence re-verification and generic bug-hunting (already gated
deterministically) and applies the three lenses the gates are blind to — security/authz (ADR-029 threat model),
test vacuity, and untested error/edge/concurrency paths. `scope.files` drift is flag-only (a scope concern never
forces reject, ADR-027 Q6). Fan-out children each get their own per-child implement review (#677); the decomposed
parent additionally gets a **consolidated review** against the merged-diff (`DispatchConsolidatedReview`, #1060),
anchored to the parent's implement stage so fix-up resolves its concerns through the normal path.

**Reviewer context surfaces** — each rendered only when present, prompt byte-identical when absent (the #984
pattern); assembly details in `backend/internal/planreview/README.md`: operator-stageable `scope_drift` paths
(#695/#581), machine-verified `gate_evidence` (#963, with `superseded` verify-run marking #1205, drift A/B
categories #991, fix-up self-report divergence #1210), binding approval conditions (#1021), and the supplemental

### 4.3 Campaign lifecycle (ADR-047)

A **campaign** drives a whole epic's child issues as a dependency-ordered batch of runs — the multi-issue counterpart to the single-run lifecycle above. It assembles once from the epic, then the backend auto-drives the constituent runs to completion under the same `operator_agent` delegation a solo run uses, pausing and paging the human only when a run hits a `must_page_human` judgment point. The lifecycle is five steps; the implementing packages are in §10's campaign rows.

1. **Assembly** — `POST /v0/campaigns` (or `fishhawk_start_campaign`) is **runless**: with no run row to carry it, the handler resolves the repo's GitHub App installation directly (the same path `POST /v0/runs` uses), queries the epic's children + their issue-level `depends_on` edges via the work-management provider's `EpicChildren` capability (E25.1), and hands the result to `campaign.Assemble` (E25.3). Assembly maps each child to an ascending index, reuses `plan.Waves` (the Kahn topological sort that already orders decomposition fan-out) to wave-order the DAG, and persists the `campaign` + `campaign_item` rows (E25.2). It **fails closed**: a dependency targeting a non-child (`campaign_dangling_dependency`) or a cycle (`validation_failed`) refuses assembly rather than minting a partial campaign. `pause_policy` (`pause_campaign`, the default, vs `pause_item`) is fixed at create time.
2. **Drive (ADVANCE-then-START)** — the campaign driver ticker (`backend/internal/campaigndriver`, off by default, `--enable-campaign-driver` / `FISHHAWKD_ENABLE_CAMPAIGN_DRIVER=true`, `--campaign-driver-interval` default 60s) polls every `running` campaign and runs two passes per tick, re-reading the items between them so a predecessor that settles this tick AND its
   now-eligible dependent both move in the same tick. **ADVANCE**: each running item whose linked run reached a terminal state is transitioned to the mapped terminal item state and the campaign state is re-derived (`campaign.DeriveState`). **START**: items the pure engine (`campaign.NextEligible`) reports eligible — every dependency succeeded — are started up to a per-campaign
   `MaxParallel` budget (default 4) through the `RunStarter` seam, which mints a `feature_change` run (`--campaign-driver-workflow-id`) by resolving + parsing the repo's workflow spec at start time (a campaign carries no workflow context of its own). The driver is fail-closed: it refuses to start when any of `CampaignRepo`/`RunRepo`/`AuditRepo`/the GitHub client is unwired, and
   a flag-off start is a silent no-op.
3. **Auto-drive each run gate (E25.6)** — during ADVANCE, every running item whose linked run is NON-terminal is handed to the optional `GateActor` seam BEFORE terminal observation. `AutoDriveRunGate` re-evaluates the run's `operator_agent` delegation in-process (read-only) and acts on a gate ONLY when the delegated knob's condition is met AND the real gate state independently matches — **double-gated, fail-closed to observe-only** on any mismatch, evaluation error, unrecognized condition, or unconfigured merge seam. The mapped actions are `may_approve`→approve, `may_route_fixup`→fixup, `may_retry`→retry, `may_merge`→merge, taken under the campaign operator identity (subject `operatorrole.CampaignActorSubject` = `operator-agent/campaign`, the `CampaignActorScopes` write scopes, stamped `audit.ActorAgent`). When the merge client is unconfigured the actor is nil and the driver runs purely observe-only, parking every gate for the human/operator-agent.
4. **Pause/page on `must_page_human` (E25.7)** — a gate condition the contract flags `must_page_human` (`reviewer_reject`, `requirement_arbitration`) is REFUSED, not acted on: the actor emits the `campaign_gate_paged` hand-off (run chain) and takes no action. The driver then pauses the affected item (recording the `PauseReason` `{page_event, run_id}`) and — unless `pause_policy` is `pause_item` (continue the other items) — transitions the campaign `running`→`paused`, records a `campaign_paused` marker on the global chain, and fires the human page through the optional `Notifier` seam (nil → pause recorded but no page). A `paused` campaign is **sticky**: re-derivation is skipped so a sibling settling can never auto-unpause it, and the START pass is skipped while non-running.
5. **Resume** — resuming is an **explicit operator action, never derived**: `POST /v0/campaigns/{id}/resume` (or `fishhawk_resume_campaign`) flips the campaign `paused`→`running` (under `pause_campaign`) and every paused item `paused`→`running` so the next tick re-engages; `campaign_not_paused` (409) when nothing is paused. The operator-agent handles the distilled page first (per the campaign-scoped `conventions.campaign` procedure in the operator role spec, §8 / `docs/spec/operator-role.md`), then resumes — never resuming to mask an unresolved `must_page_human` cause.

**Operator-driven local drive (E26.2 / #1481)** — the auto-driver (steps 2–4) is the GitHub-Actions path; an operator-agent can instead drive a campaign through the local loop with neither `--enable-campaign-driver` nor GHA. `POST /v0/campaigns/{id}/runs` (`handleStartCampaignItemRun`, or `fishhawk_start_campaign_item_run`) is the operator counterpart to the driver's START
pass: for an `issue_ref` it refuses unless `campaign.NextEligible` reports the item eligible (naming the blocking dependency on refusal), mints the run through the same `Server.StartRunForCampaignIssue` seam — extended to carry `runner_kind` so `local` reaches `CreateRunForTrigger` — links it (`SetCampaignItemRun`), moves the item `pending`/`blocked`→`running`, and advances a
`pending` campaign to `running`. The matching ADVANCE counterpart is **reconcile-on-read**: `GET /v0/campaigns/{id}/status` (`reconcileCampaignItemsOnRead`) settles any running item whose linked run reached terminal and re-derives the campaign, mirroring the driver's ADVANCE pass through the same state-guarded `TransitionCampaignItem`/`TransitionCampaign` and emitting the same
`campaign_issue_settled`/`campaign_advanced` global-chain markers — best-effort (never fails the read) and idempotent (a re-poll over an already-settled item performs no further transition). A second, **run-less issue-closed settle** arm (`settleIssueClosedItems`, #1558) handles a campaign item completed OUTSIDE the run lifecycle — a human-led (`autonomy:low`) issue merged and
closed by a maintainer PR, which the run-linked pass can never see because it has no linked run: for a run-less, dependency-satisfied item it reads the item's GitHub issue and settles it `succeeded` ONLY when the issue is CLOSED as completed (`state_reason=completed`), emitting `campaign_issue_settled` tagged `settled_via=issue_closed` with NO `run_id`, so descendants become
eligible (and an all-human-led campaign rolls `pending`→`succeeded`) with zero operator action. That arm is best-effort / fail-closed: a nil GitHub client, a non-`owner/name` repo, an installation-resolution error, a `GetIssue` error, a non-`issue:N` ref, an open issue, or a `not_planned` closure each leaves the item unsettled and never fails the read (the installation is
resolved at most once per reconcile). So the rollup + `next_action` advance in DAG order as the operator drives each run to merge. There is deliberately **no campaign concurrency cap** on this manual path (the operator decides when to start each eligible item) and **no `idempotency_key`** (the server does not dedup the start; the eligibility gate refuses a re-start against a
running item).

### 4.4 Release arc — delegating publish (ADR-051)

The **release arc** turns a range of merged runs into a published GitHub Release with generated, evidence-linked notes. It is a **delegating** lifecycle: Fishhawk assembles the evidence, renders the notes, and records each version decision on an audit chain, but the **git tag push stays a human git action** — Fishhawk never pushes a tag or mints a Release, only writes to one an operator has already created. The endpoints are operator verbs driven through the CLI (`fishhawk release …`) and the `fishhawk_release_notes` MCP surface; the implementing packages are in §10's E33 rows and the operator walk is `docs/deploy/release-loop.md`.

The arc is five steps:

1. **prepare / preview** — `GET /v0/releases/notes/preview` and `POST /v0/releases/notes` (E33.2) assemble the merged-run evidence in a ref range (`releaseevidence.Assembler`) and render it to markdown (`releasenotes.Render`) with the advisory semver-bump hint (E33.4 classifier). Preview is read-only; prepare persists the rendered notes as a `release_notes` artifact keyed to a caller-supplied stage. Evidence is **never fabricated** (ADR-051 honesty constraint): a PR with no resolvable Fishhawk run renders as reduced-evidence, and the notes body's plan/reviewer-verdict/acceptance lines resolve to the merged runs' real audit rows.
2. **cut** — `POST /v0/releases/cut` (E33.5) records the operator's ratified version decision as a `release_cut` audit entry (`{repo, version, artifact_id, bump_level, content_hash}`, system actor) on the release run's chain, after kind-checking the target `release_notes` artifact. It records the DECISION only — **no git tag push, no GitHub write** — so it needs no GitHub client.
3. **(human-led tag push)** — the operator pushes the git tag and creates the GitHub Release. This is the deliberate human seam of the delegating posture; Fishhawk does not perform it.
4. **pipeline** — the operator's own release pipeline (module-tag-triggered) builds and publishes the tagged artifacts. Fishhawk does not own or trigger it.
5. **publish** — `POST /v0/releases/publish` (E33.3) sets the existing Release's body to the persisted notes markdown and attaches them as the fixed-name `release-notes.md` asset via the App installation token, then records a `release_published` audit entry (`{tag, release_url, artifact_id, content_hash}`, system actor) on the release run's chain. Idempotency keys on CONTENT HASH for both the body and the asset so they can never diverge; a re-invoke whose desired notes hash matches both surfaces is a full no-op.

The `release_cut` and `release_published` entries link into the release run's per-run audit hash-chain (`audit.AppendChained`), so the version decision and the publish are both **verifiable on the chain** by `prev_hash`→`hash` continuity — the same integrity the compliance export verifies. The deterministic in-tree proof of the whole seam (assemble → render → prepare → cut → publish → chain continuity, offline via a fake resolver + fake publisher) is the E33.6 capstone test (§10); the one real published Release is an operator-executed live walk. The cost cross-check and evidence-notes verification procedure are in `docs/deploy/release-loop.md`.

### 5.1 Postgres (`fishhawkd` schema)

Designed under E2.1 (#22). Tables (immutable schema once frozen at Day 21):

- `runs` — one row per workflow execution. `(run_id, repo, trigger_source, workflow_sha, state, created_at, …)`
- `stages` — one row per stage execution. `(stage_id, run_id, kind, executor, state, started_at, ended_at)`
- `artifacts` — typed outputs. `(artifact_id, stage_id, kind, schema_version, data_jsonb, content_hash)`
- `signing_keys` — per-run ephemeral key chain. `(run_id, public_key_b64, issued_at, expires_at)`. Immutable.
- `audit_entries` — append-only event log. `(entry_id, run_id, ts, category, kind, actor_subject, payload_jsonb, prev_hash)`. Application-layer enforces no UPDATE/DELETE.
- `approvals` — `(approval_id, stage_id, approver_subject, surface, decision, ts)`.
- `sessions` — opaque session IDs for the Web UI cookie auth.
- `api_tokens` — scoped CLI/API tokens with revocation state.

Connection pool: `pgxpool.Pool` per service instance.

#### 5.1.1 Tenancy schema (ADR-057) — forge-neutral by construction (ADR-058)

Migration `0052` (#1854, ADR-057 / ADR-058 keystone #1823/#1851) creates the first two tenancy tables **carrying a forge `provider` discriminator at birth**, so a second forge (GitLab, ADR-058) becomes an *additive* provider rather than a constraint migration that has to retrofit a GitHub-shaped schema. Migration `0055` (#1825, E44.1) extends the foundation additively: it adds the `account_members` membership table, threads a nullable `account_id` FK through every root entity, and relocates the per-forge endpoint columns from `accounts` to `installations` (Amendment A1). No server code reads or writes these tables yet — the `backend/internal/account` sqlc surface (`accountdb`) is unwired; readers/isolation land in later E44 children.

- `accounts` — one row per tenant forge account. `(id, provider, account_key, display_name, granularity, home_region, …)`. `provider TEXT NOT NULL DEFAULT 'github'` with `accounts_provider_check CHECK (provider IN ('github','gitlab'))`. `granularity` admits `('enterprise','organization','group')` — enterprise/organization are the ADR-057 GitHub grains, group is GitLab's. Natural key is forge-neutral: `UNIQUE (provider, account_key)` (GitHub enterprise slug / org login today, GitLab group path later — the same key string in two forges never collides). `UNIQUE (id, provider)` anchors the `installations` + `account_members` composite FKs. (The `forge_base_url` / `oauth_base_url` endpoint columns moved to `installations` in `0055`, Amendment A1.)
- `installations` — one row per credential scope. `(id, account_id, provider, installation_ref, forge_base_url, oauth_base_url, …)`. `provider` carries the same CHECK; `installation_ref TEXT NOT NULL` is the forge-neutral credential-scope key (the stringified GitHub App installation id today, a GitLab group OAuth-application authorization ref later — **TEXT, not the `installationID` int64** the pre-ADR-058 code assumed, the ADR-058 Phase 1 seam, #1756). `FOREIGN KEY (account_id, provider) REFERENCES accounts (id, provider) ON DELETE CASCADE` structurally pins an installation's provider to its account's, so the two can never diverge. Carries the relocated per-forge endpoint columns (Amendment A1).
- `account_members` (`0055`) — forge-neutral membership grants, the login-gate source (materialized from GitHub Enterprise / GitLab group membership later). `(id, account_id, provider, member_ref, role, …)`; `UNIQUE (account_id, provider, member_ref)`; the same composite FK as `installations` with `ON DELETE CASCADE` (a grant has no meaning without its account).
- `account_id` threading (`0055`) — a nullable `account_id UUID` column + per-table `<t>_account_id_fkey FOREIGN KEY … REFERENCES accounts (id) ON DELETE SET NULL` + index on the eight root entities (`runs`, `campaigns`, the four `refinement_*`, `api_tokens`, `audit_entries`). Nullable throughout — isolation is not enforced here; a later child tightens it to `NOT NULL` once every row is populated.

**Binding contract for the E44 children** (inherited from this schema):

1. Every tenancy-scoped table added by E44.1+ carries or joins through the `provider` dimension.
2. E44.6 RLS predicates and E44.4 per-account audit-chain keys scope by `account_id`, whose identity already embeds `provider` via `UNIQUE (provider, account_key)`.
3. Endpoint-config columns are forge-neutral (`forge_base_url` / `oauth_base_url`, NULL = provider default endpoints — api.github.com / github.com today), **never** provider-named (no `github_base_url`). They live **per-installation** (`0055`, Amendment A1) — a forge-agnostic workspace spanning a github.com install and a gitlab.com group cannot share one per-account base URL. Endpoint *resolution* is deferred to E44.2 (#1826).
4. `installation_ref` is the forge-neutral credential-scope key replacing the `installationID` int64 assumption.

### 5.3 Trace bundle format

`*.jsonl.gz`, one JSON event per line, UTF-8. Schema versioned via the manifest event's `bundle_schema` field (current: `"v1"`).

| Event kind | Position | Purpose |
|---|---|---|
| `manifest` | First line | Schema version, run/stage IDs, agent/model identity |
| `prompt` / `tool_call` / `tool_result` / `model_response` | Middle | The actual trace |
| `policy_event` / `gate_event` | Middle | Constraint hits, approval transitions |
| `error` | Middle | Errors during agent execution (category A) |
| `agent_retry` | Middle | Boundary marker the runner's Claude Code driver emits between attempts when it re-spawns the agent after a transient thinking-block API 400 (payload: `attempt`, `reason`, `tokens_so_far`). See the bounded in-driver retry row in §10. |
| `out_of_tree_write` | Middle | Warning the Claude Code driver emits when an agent file-writing tool call (`Write`/`Edit`/`MultiEdit`/`NotebookEdit`) targets a path outside the working tree + `/tmp` allowlist (payload: `path`, `tool`, `run_id`, `stage`). Purely additive surfacing — never flips `Result.OK`, never fails the stage, fail-open on parse error. Catches the #601 Write/Edit-tool class only; Bash-mediated `>` writes are NOT detected (full confinement is the OS-sandbox ADR's domain). See the out-of-tree write detection row in §10. |
| `trailer` | Last line | `event_count`, content hash of preceding lines |

**Not a bundle event — `stage_progress` liveness heartbeat (#580)**: during agent invocation the Claude Code driver writes a single-line `{"event":"stage_progress","elapsed_seconds":N,"turns":N,"tokens_so_far":N,"last_event_kind":"…"}` to the runner's **logSink (stderr) only**, every ~15s (`claudecode.Invoker.HeartbeatInterval`, default 15s). It carries coarse structural counters — never agent payload text — so a long stage is visibly progressing. It is **never** appended to `Result.Events`, so the signed trace bundle is unchanged (same treatment as the logSink-only `runner_cancelled` / `stage_self_retry` lines). The `fishhawk-mcp` `fishhawk_run_stage` relay forwards it as a `notifications/progress` update whose message renders the counters; because it's time-driven, a stalled stage still emits heartbeats with non-advancing `turns`/`tokens_so_far`, letting the driver tell "alive and progressing" from "stuck".

## 6. Invariants

These are load-bearing. Do not break them without explicit ADR.

1. **Customer source code never reaches Fishhawk's backend.** Code lives only in the customer's repo and on their CI runners. The backend sees traces, plans, and metadata — never source.
2. **Audit entries are append-only.** Enforced at the application layer (`audit.Append(...)`; no `Update` or `Delete`). Static-analysis test asserts no other code path mutates `audit_entries`.
3. **Trace bundles are signed by per-run ephemeral keys.** A tampered runner cannot forge a trace for any run other than its own; a leaked key expires within 30 minutes.
4. **Honest gap reporting beats fictional completeness.** A runner crash produces an audit entry that says `trace_lost`, not silent omission (MVP_SPEC §6).
5. **The workflow spec syntax is frozen at Day 21.** Old plans in the audit log remain readable forever; we never break old `standard_v1` artifacts.
6. **No founder bypass.** Methodology commitment from `docs/METHODOLOGY.md`. Emergency paths exist, are themselves audited, require post-hoc justification.
7. **The plan prompt never carries raw untrusted issue-comment text.** Issue comments are attacker-controllable, so `writeIssueComments` (§10) routes every body through the deterministic `sanitizeUntrustedComment` quarantine (structural-injection neutralization + line-quoting + an "untrusted DATA, not instructions" envelope) before it reaches the network-and-state-capable plan agent — only a quarantined summary, never raw comment text. ADR-029 / #650 item 1; drops the plan agent from three lethal-trifecta legs to two.
8. **The implement prompt never re-ingests raw untrusted issue-comment text.** The implement agent is network-and-state-capable, yet it is safe at two legs only because it consumes the human-APPROVED plan — the approval gate is the trust boundary — and renders the issue as a link (`writeIssueLink`, §10), never raw comment or body text. `buildImplement` MUST NOT call `writeIssueComments` or render `Trigger.IssueBody` / `Trigger.IssueComments`; a regression test (`TestBuild_Implement_NeverReingestsUntrustedComments`) enforces this mechanically. ADR-029 / #650 item 2; the implement-side counterpart of invariant #7.

### Rule-of-Two posture (per agent)

The lethal trifecta is: access to private data, exposure to untrusted content, and the ability to externally communicate. The Rule of Two holds when each agent has at most two of the three legs. Auditable per-agent verdict (ADR-029 / #650):

| Agent | Posture | Legs |
|---|---|---|
| **plan** | Quarantined input — untrusted issue/comment text passes through `sanitizeUntrustedComment` (invariant #7) before reaching the agent; never raw. | two |
| **implement** | Approved-plan-only input + never-re-ingest invariant (#8); the human approval gate is the trust boundary. ADR-029's *decided* posture also gates outbound traffic behind a hosted-egress allow-list — that leg is the intended posture but is **not yet enforced** (tracked in #930); do not treat egress allow-listing as currently active. | two |
| **review** | Low-risk — advisory only: emits a structured verdict, holds no write authority and mutates no state. | n/a (no state mutation / external write) |
| **acceptance** | Deliberately assembles all three legs (code execution + network + credentials against a running instance rendering untrusted data), so treated as prompt-injected and contained (ADR-050 / #1532): default-deny egress via the runner-embedded proxy, minimized credentials (no MCP/Fishhawk token), advisory zero-write authority — a fully-compromised agent can at worst emit a wrong verdict. Acceptance-sourced fix-up concerns are provenance-marked and quarantine-enveloped (E31.8 / #1613). Details: `runner/internal/egressproxy/README.md`. | contained (three legs assembled, each bounded) |

## 7. Module boundaries

The Go monorepo is a workspace, not a single module. Each top-level directory is independently taggable:

- `/backend` — `github.com/kuhlman-labs/fishhawk/backend`. Internal packages only (no exported API to other modules).
- `/runner` — `github.com/kuhlman-labs/fishhawk/runner`. The published GitHub Action artifact (composite action manifest at `runner/action.yml`). Customers pin a tag.
- `/cli` (planned) — `github.com/kuhlman-labs/fishhawk/cli`. Single binary.

Cross-module type sharing is intentionally avoided in v0. If two modules need the same struct (e.g., the `standard_v1` plan schema), the canonical source is a JSON Schema in `/docs/spec/` and each side parses independently. This keeps the runner's dependency graph small and the supply-chain surface minimal.

The frontend (`/frontend`, planned) is its own npm workspace; it talks to the backend over the REST API only.

### 8.1 Approval identity

Quorum-gate approvals (the `approvals` block, `workflow-v1`) resolve who an approver is, what they may do, and how their vote was authenticated through the forge-neutral identity seam. The §10 "Where to look" identity capability row (E39.1 / #1706) is the code map; this section is the runtime lifecycle.

**IdentityProvider seam.** `backend/internal/identity/identity.go` defines `IdentityProvider` with four methods: `VerifyUser` (drives the GitHub OAuth device flow to a provider-qualified subject `github:<login>`), `VerifyAccessToken` (re-verifies a CLI-obtained access token server-side to a subject), `PermissionLevel` (returns the subject's forge-neutral `Permission` tier — `none`/`read`/`triage`/`write`/`maintain`/`admin` — on a repo), and `ResolveMembership` (reports org or team membership for `org` / `org/team` refs). Two implementations: `GitHubIdentityProvider` (`github.go`, a hand-rolled REST + device-flow client) and `NoOpIdentityProvider` (`noop.go`, deny-by-default — `VerifyUser`/`VerifyAccessToken` fail closed with `ErrNotConfigured`). `serve.go::resolveIdentityProvider` constructs the GitHub impl only when OAuth client config is present; otherwise `server.New` defaults to the NoOp.

**`auth_method` (how the token was authenticated).** Recorded on the `api_tokens` row and flowed onto `Identity.AuthMethod` by the bearer-auth middleware. Two values, fixed by the migration `0050_api_tokens_auth_method` CHECK constraint: `static` (a CLI-issued operator token, `fishhawkd token issue`) and `oauth` (a device-flow login minted via `POST /v0/tokens/login`). The column is nullable with `DEFAULT 'static'`, so every pre-migration row and the static-issue path read back `static`.

**`channel` (how the vote arrived).** Computed by `approvalChannel` in `backend/internal/server/quorum.go` from the acting identity: `interactive` (a cookie-session identity — empty `TokenID`), `api` (a bearer-token identity), and `delegated` (an ADR-040 delegated / agent-kind submission). A `delegated` submission is recorded but **never counts toward the human quorum**.

**Predicate evaluation and `predicate_snapshot` recording.** When an `approvals` gate sets `min_permission` and/or `member_of`, `resolvePredicates` (`quorum.go`) resolves the predicate against the forge at **each** approval event — `PermissionLevel` for `min_permission`, `ResolveMembership` for `member_of` — with no caching between requests. `writeApprovalAudit`
(`backend/internal/server/approvals.go`) records a `predicateSnapshot` (JSON key `predicate_snapshot`) into the `approval_submitted` audit payload **IFF the gate declares an `approvals` block** — legacy count-only gates omit it. The snapshot carries `count_required`, `count_eligible`, `identity` (`{provider, subject}`), `submitter_class`, `auth_method`, `channel`,
`min_permission`, `member_of`, `quorum_reached`, plus the E39.5 / #1710 forge-resolution fields `resolved_permission`, `member_resolved`, and `predicate_result`. Rejection paths durably record a snapshot too even though no approval row is inserted: a `403 approver_predicate_unmet` (submitter below `min_permission` or not a member) and a `503 forge_unavailable` (forge error /
rate-limit / timeout, or an ad-hoc run with no repository to resolve against — fail-closed) each write their own `predicate_snapshot` entry.

## 9. CI and release shape

- **CI workflow** at `.github/workflows/ci.yml`. Path-aware via `dorny/paths-filter`. The Go job iterates `go.work` `use` directives — adding a new module Just Works.
- **Lint config** at `/.golangci.yml` (v2 format). Shared across all Go modules in the workspace.
- **Coverage targets**, tiered to the autonomy levels in `docs/METHODOLOGY.md`:
  - **Low-autonomy code** (audit log integrity, signing/crypto, policy evaluator, run state machine, workflow spec parser): ≥ 85% statement coverage.
  - **Medium-autonomy code** (HTTP handlers, runner adapters, REST endpoints, UI logic): ≥ 75%.
  - **High-autonomy code** (docs, dep bumps, lint/format): no target.
  - **Generated code** (sqlc outputs, etc.): excluded from numerator and denominator.
  - **Aggregate floor (excluding generated): ≥ 80%.** Enforced by `scripts/check-coverage.py` in the Go CI job; PRs that drop below fail `CI Pass`. Run locally with `(cd backend && go test -race -coverprofile=coverage.out -covermode=atomic ./...) && python3 scripts/check-coverage.py --threshold 80 --exclude internal/run/db backend/coverage.out`.
- **Full-suite parallelism bound (#972)**: `scripts/test` (both the default `test` and `coverage` subcommands) passes `go test -p "${FISHHAWK_TEST_P:-4}"` per module. ~16 backend packages each start a testcontainers Postgres; at the default `-p` (~GOMAXPROCS) a developer-Mac Docker daemon intermittently times out starting one (`context deadline exceeded` after 9 retries against docker.sock), failing the suite on load contention rather than a real bug. `-p` caps concurrent test binaries (tests within one binary are serial unless they call `t.Parallel()`), and modules run sequentially in the wrapper, so the bound is effectively global. `FISHHAWK_TEST_P` is the operator override; `scripts/test single` stays unbounded. CI is unaffected — `ci.yml` inlines its own per-module loop.
- **Release**: each module is tagged independently. The runner is the customer-facing one — `kuhlman-labs/fishhawk/runner@v0.1` etc. — built with signed releases + SBOM (E5.7 / #54, E13.6 / #63).
