# Fishhawk Backend API — v0

This is the human-readable companion to [`v0.openapi.yaml`](./v0.openapi.yaml).
The OpenAPI document is the source of truth; this file exists to give agents
and humans a faster way to grok the surface than reading 900 lines of YAML.
If the two disagree, the OpenAPI document wins.

## Why `/v0`

`/v0` is the prefix for everything the Web UI, CLI, and runner call. It is
**unstable by design** — fields may be renamed, endpoints may move, request
bodies may grow required fields — until the v1 cut at the end of phase 4
(MVP_SPEC §13). Once `/v1` ships, `/v0` and `/v1` coexist for one minor
version cycle, then `/v0` is retired.

The webhook receiver (`/webhooks/github`) is **unversioned** because GitHub
controls the request shape; the backend adapts as GitHub evolves.

## Auth at a glance — ADR-005 (#69)

Three security schemes for three caller classes; no other auth is supported
in v0.

| Scheme | Used by | Carrier |
|---|---|---|
| `sessionCookie` | Web UI (browser) | `Set-Cookie: fishhawk_session=…; HttpOnly; Secure; SameSite=Lax`; CSRF token via `X-CSRF-Token` matching `__Host-csrf` |
| `bearerToken` | CLI, programmatic | `Authorization: Bearer <opaque>` — server-issued, scope-bound, immediately revocable |
| `githubOIDC` | Runner action only | `Authorization: Bearer <github-oidc-jwt>` — verified against GitHub's JWKS per request, no server state |

Default for all UI/CLI endpoints is `sessionCookie OR bearerToken`. Runner-
facing endpoints override this individually; webhook ingestion uses neither
(GitHub HMAC over the body).

## Conventions

- **JSON everywhere** except `POST /v0/runs/{run_id}/trace`, which uploads
  `application/gzip` bytes and signs them via `X-Fishhawk-Signature`.
- **Cursor pagination** for every list endpoint. Response: `{ items, next_cursor }`.
  `next_cursor` is `null` at the end. Cursors are opaque; clients pass them back
  unmodified.
- **Error envelope** is `{ error: { code, message, details? } }`.
  - `code` is snake_case, stable, machine-readable. Clients switch on it.
  - `message` is human, may change between versions.
  - `details` is structured per-code; the keys ARE part of the contract for
    that code.
- **Timestamps** are RFC 3339, always UTC, always with `Z`.
- **IDs** are UUIDv7 strings; `format: uuid` in the schema.
- **Idempotency** — `POST /v0/runs` and `POST /v0/campaigns` honor an
  `Idempotency-Key` header: a replay with the same `(repo, key)` returns the
  existing resource (200) instead of minting a duplicate. The cancel endpoint
  and approval submission are idempotent on equal inputs.

# Audit export contract (Export v1)

The producer half of the audit-grade verifier contract (ADR-008 / ADR-054).
`GET /v0/audit/export` emits the EXACT wire shape the external verifier
consumes so an out-of-band party can chain-verify without trusting the
backend.

- **Whole-run page bounding.** `limit` bounds the number of RUNS per page;
  a run's chain is never split, so each page independently passes the
  verifier's `VerifyExport` from every run's nil-`prev_hash` genesis entry.
- **Header-borne continuation.** The verifier's `ParseExport` uses
  `DisallowUnknownFields`, so the body is exactly `{schema, exported_at,
  runs}` — nothing else. Partiality and the opaque keyset cursor ride the
  `X-Fishhawk-Export-Complete` (`true`/`false`) and
  `X-Fishhawk-Export-Next-Cursor` response headers. Never silent
  truncation: a partial page always advertises `false` + a cursor.
- **Reserved global-chain key.** Run-less entries (token issue/revoke,
  OAuth) export under the reserved nil-UUID key
  `00000000-0000-0000-0000-000000000000`, their `run_id` serialized as
  `null`. Emitted on the FIRST page only (empty cursor), even when empty,
  so consumers can tell "no global entries" from "not included" — never
  silently dropped. Opt out with `include_global=false`.
- **Filter mutual exclusivity.** An explicit `run_id` set (repeatable;
  a missing run is a fail-closed 404) is mutually exclusive with the
  `repo`/`from`/`to` filter shape (`repo` via the runs join, dates on run
  `created_at`). Combining them is a 400.
- **Auth: `read:audit-export` scope required** (E9.5/#1608, ADR-054) on
  ALL export surfaces — this endpoint, the CSV projection, and the
  agent-changes report (both renders). Anonymous callers get 401
  `authentication_required`; a bearer token without the scope gets 403
  `insufficient_scope` with `details.required_scope` named;
  cookie-session operators bypass scope checks per the standard
  contract. The scope is in the operator default set (`fishhawkd token
  issue` grants it; `fishhawkd token migrate --apply` promotes existing
  subset tokens), and is deliberately distinct from per-run reads so
  export-capable tokens are enumerable and revocable independently.
- **Redaction posture (ADR-054).** Exports carry entry payloads
  (redacted at write time where applicable) and content-hash POINTERS
  only — trace bundle bytes are NEVER inlined, and the raw-variant
  bundle stays behind the existing compliance gate (no read endpoint
  serves it). Pinned by test over a run with raw traces present.

## CSV export (`GET /v0/audit/export.csv`)

A flat CSV rendering of the same export for spreadsheet compliance
workflows (#1605). It is a **projection** over the JSON export's shared
run-selection and per-run assembly layer (`resolveExportPage` +
`assembleRunData`) — never a parallel query path — so its rows are a
field-for-field projection of the JSON `Export v1` body for the same
run-level filter set. A parity test locks this.

- **Columns.** One audit entry per row:
  `ts,run_id,repo,category,actor_kind,actor_subject,sequence,entry_hash,payload_summary`.
  `ts` is RFC3339 UTC; nil `actor_kind`/`actor_subject` and the global
  partition's `run_id`/`repo` are empty cells; `sequence` is a base-10
  integer; `payload_summary` is the compacted payload JSON bounded at 256
  runes (rune-boundary safe — never splits a multi-byte character) with a
  `...(truncated)` suffix when cut. Full payloads stay JSON-export-only.
- **Entry-level filters (CSV-only).** `approver` keeps only
  `approval_submitted` rows whose `actor_subject` equals the value;
  `category` keeps only rows of that exact category. Both compose (ANDed)
  with each other and with the inherited run-level filters. Entry-level
  filtering is CSV-only by design: dropping entries from the JSON body
  would break the verifier's hash-chain walk, so these filters are not
  offered on `GET /v0/audit/export`.
- **Download headers.** `Content-Type: text/csv; charset=utf-8` and
  `Content-Disposition: attachment; filename="fishhawk-audit-export-<UTC
  timestamp>.csv"`. Rows use RFC 4180 quoting (fields with commas,
  quotes, or newlines are double-quote wrapped, embedded quotes doubled);
  line endings are LF (accepted by Excel, Numbers, and Google Sheets).
- **Bounding & continuation.** Identical to the JSON export: `limit`
  bounds RUNS per page (not rows), and the same
  `X-Fishhawk-Export-Complete` / `X-Fishhawk-Export-Next-Cursor` headers
  drive continuation. Because bounding is whole-run, an entry-filtered
  page can legitimately have zero data rows while `Complete` is `false`.
- **No partial CSV on failure.** The whole page is buffered before any
  bytes are written, so a per-run assembly error is a clean JSON `500
  internal_error` with no half-written CSV.
- **Formula-injection non-goal.** Cells are raw audit values — no `=`/`+`/
  `-`/`@` escaping is applied, because prefixing an escape would mutate
  audit values and break the CSV-equals-JSON-projection guarantee. Import
  untrusted payload summaries via a spreadsheet's data-import flow rather
  than double-click-opening the file. Hardening, if wanted, belongs with
  the redaction sibling (#1608).

## CLI consumer & end-to-end external verification (`fishhawk export`, E9.4 / #1607)

The `fishhawk export` verb is the reference consumer of both export
endpoints. It follows the `X-Fishhawk-Export-Complete` /
`X-Fishhawk-Export-Next-Cursor` header continuation **automatically**,
assembling one complete bounded file so the operator never hand-stitches
pages:

- **JSON assembly.** Each page is decoded keeping every `runs` value and
  the `schema`/`exported_at` values as raw JSON, so run subtrees are
  byte-preserved (the entry hashes and Ed25519 signatures still verify).
  The per-page `runs` maps are unioned — disjoint because the global-chain
  partition rides the first page only — and re-marshalled as exactly
  `{schema, exported_at, runs}`, so the assembled file byte-parses through
  the verifier's `DisallowUnknownFields` `ParseExport`. A run key appearing
  on two pages, or a page reporting incomplete with no continuation cursor,
  is a hard error (never a silent merge or an infinite loop).
- **CSV assembly (`--csv`).** Pages are concatenated keeping only the first
  page's header row; a later page whose header differs is a hard error.
- **Atomic `--out`.** The assembled file is written temp-file-then-rename,
  so a mid-pagination failure never leaves a partial file at the
  destination; without `--out` the export streams to stdout.
- **Server-authoritative filters.** The CLI passes `--run` (repeatable) or
  the `--repo`/`--from`/`--to` shape through unchanged and renders the
  server's `400 validation_failed` on the mutual-exclusion violation
  verbatim rather than pre-checking.

End-to-end external-verification procedure (the audit-grade pitch):

1. Issue a `read:audit-export`-scoped token for the auditor (or export
   yourself with an operator token).
2. `fishhawk export --from <RFC3339> --to <RFC3339> --repo owner/name --out
   export.json` (or `--run <UUID>...` for an explicit set; `--csv` for the
   spreadsheet rendering). The assembled `export.json` is exactly the
   verifier's Export v1 wire shape.
3. Hand `export.json` — carrying each run's public signing key and full
   chained audit trail — to the external party.
4. `fishhawk-verify --export export.json` recomputes every entry hash and
   chain link with no backend trust required, exiting `0` (all chains
   verified), `1` (one or more issues, e.g. `kind=hash_mismatch` on a
   tampered entry), or `2` (usage error). A pgtest-backed round-trip test
   (`backend/internal/server/audit_export_roundtrip_test.go`) exercises
   this whole path against real Postgres data with `--limit 1` forcing
   multi-page assembly, plus a tamper case proving the verifier fails.

A worked instance of this whole procedure — Fishhawk's own development audit
log exported over the E31 window and published as a repo artifact with
provenance and verification instructions — lives at
[`docs/compliance/`](../compliance/) (E9.6 / #1609).

## Agent-changes report (`GET /v0/reports/agent-changes[.md]`)

run range, with who approved / what reviewed / what validated each one. Like
the CSV endpoint it is a **projection** over the JSON export's shared
run-selection and per-run assembly layer (`resolveExportPage` +
`assembleRunData`) — never a parallel query path — so it inherits the SAME
run-level filters (`repo`/`from`/`to`/`limit`/`cursor`, `run_id` mutually
exclusive with the filter shape) and `created_at`-DESC keyset paging. It does
NOT offer the CSV-only `approver`/`category` entry filters. One report model
feeds both the JSON and the `.md` render (one-model-two-renders parity, locked
by a test).

- **Per-change evidence.** For each selected run that produced a change (a
  `pull_request_opened` entry): the PR (`pr_url`/`pr_number`/`branch`/
  `head_sha`); merge status (`merged`/`merger`/`head_sha`/`base_sha`) from
  `pr_merged`; approvals (subject/decision/timestamp) from
  `approval_submitted` + `pr_approved_on_github`; reviews (stage/reviewer
  model/verdict) from `plan_reviewed` + `implement_reviewed`; acceptance
  (verdict/failure_mode/evidence_hashes/content_hash) from
  `acceptance_outcome_recorded`; the audit-chain span (first/last sequence +
  entry hashes + entry count); and redacted-tier evidence links
  (run/audit/export/artifact API pointers per ADR-054, `ExternalURL`-prefixed
  when configured else relative — never inlined trace bundles).
- **Human-led reduced evidence.** Runs whose `workflow_id` is
  `human_led_change` render in a SEPARATE `human_led_changes` section with
  `reviews`/`acceptance` omitted (reduced evidence). Runs that produced no PR
  are excluded from both lists but counted in `totals.runs_without_change`.
- **Continuation is in-band here.** UNLIKE the verifier-strict `Export v1`
  body, this report is not strict-decoded by any verifier, so `complete` and
  `next_cursor` appear in BOTH the response headers
  (`X-Fishhawk-Export-Complete` / `X-Fishhawk-Export-Next-Cursor`) AND the
  report body. The markdown render carries an explicit `PARTIAL REPORT —
  resume with cursor …` banner on a partial page. These body continuation
  fields are report-only; they never appear on `GET /v0/audit/export`.
- **Markdown download.** `GET .../agent-changes.md` serves
  `text/markdown; charset=utf-8` as `Content-Disposition: attachment;
  filename="fishhawk-agent-changes-<UTC timestamp>.md"`.
- **Malformed payloads never fail the report.** A single corrupt audit
  payload is skipped with a logged warning, not a `500` — a compliance report
  must not black-hole on one bad row.
- Auth scope (`read:audit-export`) is deferred to E9.5/#1608 for all export
  surfaces, exactly as the export endpoints state.

## Resource model

The handlers in `backend/internal/server` map cleanly onto the persistent
state layered up through `backend/internal/run`, `backend/internal/audit`,
`backend/internal/signing`, and `backend/internal/tracestore`.

| Resource | Source of truth | Notes |
|---|---|---|
| `Run` | `runs` table; state machine in `internal/run/transition.go` | All state transitions validated against the table; FOR UPDATE lock per write. `cost_usd_total` + `resolved_model` (#649) are rolled from each stage bundle's signed manifest token counts — recorded once per bundle (raw trace variant only, #678); both always present in the response (0 / empty string for legacy rows). `lineage_complete` (#1137) is computed on `GET /v0/runs/{id}` ONLY — true when the lineage-root run and every decomposed child are terminal; the local runner's worktree sweep reads it to reclaim a terminal lineage's shared worktree. `fixup_model` (#1164) is distilled on `GET /v0/runs/{id}` ONLY from the newest `stage_fixup_triggered` entry's pinned `{model, source, pass_ordinal}`; omitted when the run has had no fix-up. `runner_kind_resolved` (#1346 / #1348) is always present (false for legacy/un-resolved rows); it flips true once the run's first signed runner self-report LOCKS `runner_kind`, and the host-dispatch mismatch guardrail (#1355) reads it alongside `runner_kind` to reject a local host dispatch against a `github_actions`-locked run before execution |
| `Stage` | `stages` table | One stage per workflow stage; `type` is one of `plan`, `implement`, `review`, `deploy`, `acceptance`; `executor` is `{kind: agent|human, ref}`. `resolved_model` (#1416) is the model id the approval gate resolved for this stage's agent spawn, surfaced from the per-stage `model_resolved` audit keyed by the stage's id; empty string when no resolution was recorded (the adapter-default spawn, and every stage of a run approved before per-stage model selection landed) |
| `Artifact` | `artifacts` table | `kind` is one of `plan`, `pull_request`, `deployment`, `acceptance`. Plans (`kind=plan`) validated against `docs/spec/plan-standard-v1.schema.json`; deployment records are stored as `kind=deployment` artifacts (`DeploymentArtifactBody` shape); acceptance evidence is stored as `kind=acceptance` artifacts (`AcceptanceArtifactBody` shape, E31.6 / #1534) |
| `AuditEntry` | `audit_entries` table; chain in `internal/audit/chain.go` | Append-only at three layers: API surface, static-analysis test, DB triggers |
| `SigningKey` | `signing_keys` table; ephemeral private half | One key per run; private half NEVER persisted, returned exactly once |
| `User` / `ApiToken` | `users`, `api_tokens` tables | Tokens stored hashed; plaintext returned exactly once |

## Caller flows

### Web UI: review → approve a stage

```
GET  /v0/auth/me                      → User
GET  /v0/runs?state=running           → recent runs
GET  /v0/runs/{id}                    → run detail
GET  /v0/runs/{id}/stages             → stage list
GET  /v0/stages/{stage_id}/artifacts  → fetch the plan
GET  /v0/artifacts/{plan_id}          → plan content (standard_v1)
POST /v0/stages/{stage_id}/approvals  → { decision: "approve" }
```

`GET /v0/stages/{stage_id}/checks` injects a self-derived
`fishhawk_audit_complete` row for review stages (published to GitHub as a
required Check Run, #229/#231). Its `missing[]` entries each carry a `kind`
the SPA branches on. Most kinds (`plan_missing`, `trace_missing`,
`pr_missing`, `chain_invalid`, `chain_unrecoverable`, `foreign_commit`) are
hard gaps → state `fail`. Two are **pending-flavored** → state `pending`
when they are the only gaps: `head_fetch_failed` (couldn't read the live PR
HEAD) and `review_pending` (#947 — a configured agent implement-review was
dispatched but has not reached a terminal verdict). `review_pending` is the
pre-merge **presence** gate: it holds the required check pending — not
failing — until the review lands, then auto-republishes green. ADR-027's
advisory verdict stays non-blocking; ANY terminal review kind clears it, and
a hard backstop (`ReviewBudget.Cap` × configured agents) clears a reviewer
that died emitting nothing.

### CLI: kick off a run, watch it finish

```
POST /v0/runs                              → { repo, workflow_id, workflow_sha, trigger_source: "cli", workflow_spec: "<YAML>" }
GET  /v0/runs/{run_id}                     → poll until state in {succeeded, failed, cancelled}
GET  /v0/runs/{run_id}/stages              → final stage breakdown
```

For issue-triggered runs the CLI (and MCP server) also ship an inline
`issue_context` — title, body, url, number, and the issue's `comments`
(see the `IssueContext` schema) — captured via `gh issue view` so the
plan agent reads comment-borne refinements without a backend-side
GitHub fetch (#415, #618).

`POST /v0/runs` also accepts an optional `budget_override` (boolean).
When the resolved workflow declares an `enforcement: blocking` periodic
cost budget whose current calendar-period spend has reached `limit_usd`,
a new run is refused with `402 budget_exhausted` (#688 / ADR-030);
`budget_override: true` forces the run past and records a
`run_admitted_budget_override` audit entry. The CLI exposes this as
`fishhawk run start --override-budget` and the MCP `fishhawk_start_run`
tool as `budget_override`. There is no override on the webhook-dispatch
path — a webhook-triggered run that crosses a blocking budget is simply
refused (a `run_rejected_budget` audit entry, no run row). In-flight runs
and CI-retry / decomposition-child continuations are never gated.

`POST /v0/runs` also accepts an optional `upstream_run_id` (uuid). It
names the upstream `feature_change` run whose `ci_green` / `review_merged`
a standalone deploy-only `release` run's deploy-stage `required_upstream`
pre-flight gate evaluates (E23.11 / #1417). A deploy-only run has no
implement/review stage of its own, so the gate evaluates the referenced
upstream instead; omitting it (the appended-deploy / non-deploy default)
leaves the gate evaluating the current run. It is DISTINCT from
`parent_run_id` (#216) — a deploy-gate safety reference, not a
follow-up/lineage link, so it carries none of the get_plan-resolution /
resume-retry / decomposition-provenance semantics `parent_run_id` does.
Because the referenced run directly controls a safety-critical deploy gate,
the gate validates its provenance at evaluation time: the upstream must be a
`feature_change` run IN THE SAME REPO as the deploy run. A reference to a
different repo or a non-`feature_change` workflow is treated as unresolved and
fails the gate closed, so a release run cannot be made to gate on an unrelated
run's CI state. Persisted on the run row and echoed back as `upstream_run_id`
on the Run resource. The MCP `fishhawk_start_run` tool exposes it as the
optional `upstream_run_id` input field, and the CLI `fishhawk run start`
exposes it as the `-upstream-run-id` flag; both validate the value is a
well-formed UUID locally before the round-trip.

`POST /v0/runs` also accepts an optional `drive` (boolean) — the per-run
drive-mode override (#1023 / #996 theme 1). When present it wins over the
workflow spec's `drive` default; when omitted the run inherits the spec
value (false when the spec doesn't set one or no spec is resolved). The
resolved flag is snapshotted onto the run row at create time — a spec
edit mid-run can't change an in-flight run — and echoed back as `drive`
on the Run resource (always present; `false` for legacy rows). Under
drive, the engine (`backend/internal/drive`) auto-advances the
mechanical transitions and records a `run_auto_advanced` audit entry per
advance; `GET /v0/runs/{run_id}` distills those entries into the
`auto_advanced` list, the current `next_action`, and the derived
`awaiting_merge` / `ci_failed` (#1045) presentation status (see the Runs
section above). A plan-gate revise that re-opens the plan stage is one
such mechanical transition (`revise_replan`, #1256): on a `github_actions`
run the orchestrator re-dispatches via `workflow_dispatch`; on a `local`
run it parks with a `run_plan_stage` `next_action` (the runner is
host-spawned per ADR-024, so the operator re-dispatches the plan stage),
surfaced on the authoritative run resource exactly as the
`plan_approved_dispatch` `run_implement_stage` park already is. The same
mechanical-park shape covers two more local re-open paths (#1271): a
`retry_stage` that re-opens a failed plan/implement stage to pending
(`retry_reopen`) and a `resume_run`/`recover` in-place decomposition-child
re-drive (`recover_redispatch`). On a `github_actions` run each
re-dispatches via `workflow_dispatch`; on a `local` run each parks with the
appropriate `run_<stage>_stage` `next_action` (`run_plan_stage` or
`run_implement_stage`) surfaced on the run resource, exactly as
`plan_approved_dispatch` and `revise_replan` already do. Judgment
points — gate approvals, concern routing, merge — always park.

### Version coordination

`GET /healthz` advertises three version-coordination fields alongside `status` and `version`:

| Field | Type | Notes |
|---|---|---|
| `git_sha` | string | Commit SHA the backend binary was built from. scripts/dev stamps the short HEAD SHA (`-dirty` suffix on a dirty tree) into dev builds; `"unknown"` only for unstamped builds |
| `min_runner_version` | string | Minimum `fishhawk-runner` version the backend accepts; `"dev"` means no constraint |
| `schemas` | object | Map of schema-name → sha256 hex hash of the embedded JSON schema. Keys: `plan-standard-v1`, `workflow-v0` |

`/healthz` also carries an optional `start_nonce` (string): a per-start opaque identity token echoed verbatim from the operator-supplied `--start-nonce` flag / `FISHHAWKD_START_NONCE` env var, and omitted entirely when unset. `scripts/dev` generates a nonce per spawn and requires it to round-trip through its readiness gate (and consults it in `down`'s port fallback), proving the listener on the port is the daemon it started even across OS pid reuse (#1018).

The runner receives `min_runner_version` in the `GET /v0/stages/{stage_id}/prompt` response body. When the field is present and the runner's own version is older, the runner must exit with code 3 (`version_skew`) rather than invoking the agent.

`fishhawk doctor` uses all three fields to surface drift warnings to the operator:
- backend `git_sha` vs local `HEAD` (warns if the local tree is ahead and the binary hasn't been rebuilt). The comparison is prefix-wise after stripping a `-dirty` suffix, so a stamped short dev SHA matches the full local HEAD — meaningful for dev builds, not just releases
- runner schema hash vs backend's `plan-standard-v1` hash (warns if runner and backend are built from different schema versions)
- CLI version vs `min_runner_version` (warns if the CLI is too old)

### Scope-bounded implement commit

On implement stages the `GET /v0/stages/{stage_id}/prompt` response carries
`scope_files` — the approved plan's `scope.files`, each an object of `path`
(repo-relative) + `operation` (`create` / `modify` / `delete`). The runner
stages exactly these declared paths at the commit boundary (and when computing
the policy diff) instead of `git add -A`, so stray dirty files — dev `.pid`
artifacts, editor scratch, unrelated local edits — cannot leak into a
Fishhawk-attributed commit (#581). Dirty-but-undeclared paths are excluded
(never staged) and surfaced as a `scope_drift` signal rather than blocking the
commit, matching the spec's flag-only scope-drift treatment. When `scope_files`
is absent (no approved plan, i.e. `plan_missing_for_implement`) the runner
falls back to `git add -A`.

On a recovery run the same prompt response additionally carries
`scope_exemptions` — the operator's `exempt_scope_files` (#1229), each a
`{path, reason}` marking a DECLARED `scope.files` path
operator-justified-unchanged. The runner subtracts these from its
`MissingScopeFiles` shortfall gate, so a category-B run that failed SOLELY on a
declared-but-unchanged file recovers without a replan. It is the inverse of
`add_scope_files`: it subtracts from the gate, it does not widen scope (no
amendment row). Resolved from the run's own `plan_reused_from` provenance;
absent on every non-recovery run.

On an implement stage the prompt response also carries `implement_model` — the
backend-resolved implement model id (#1013). When present, the runner pins the
agent spawn to it (the claudecode adapter appends `--model <implement_model>`).
The backend resolves it through the implement-model ladder: operator gate
decision < plan `model_recommendation.implement_model` < spec `executor.model`
< deployment default. When `implement_model` is absent (no rung supplied a
model — the common case), the runner leaves the model unset and the agent
spawns on the adapter's built-in default, byte-identical to today.

### App bot commit identity

The same prompt response also carries `commit_author_name` and
`commit_author_email` — the GitHub App bot account's git author identity,
resolved backend-side from the App itself (slug via `GET /app`, bot user-id
via `GET /users/<slug>[bot]`, both App-JWT authed and cached for the process
lifetime). The runner sets these as the commit author so App-backed commits
attribute to the App's bot account (`<slug>[bot]`, e.g.
`<bot-user-id>+<slug>[bot]@users.noreply.github.com`) rather than a hardcoded
default (#722). Both fields are absent when the backend can't resolve the
identity (no App JWT wired — dev / CLI / local-runner); the runner then falls
back to its default bot identity (`fishhawk-runner[bot]`).

### Decomposed child slice-branch routing

When a run is a decomposed child the `GET /v0/stages/{stage_id}/prompt`
response carries `decomposed_from_run_id` (the parent run) and `slice_index`
— the child's 0-based `sub_plan` position (ADR-041 / #1141). The runner routes
the child's commit onto its own **sole-writer slice branch**
`fishhawk/run-<shortID(decomposed_from_run_id)>/slice-<slice_index>`, cut fresh
from the declared base, rather than the pre-ADR-041 single shared
`fishhawk/run-<parent>` branch every sibling force-pushed onto. Because each
slice branch is minted once and owned by one child, the runner drops the
shared-branch push coupling (no `--force-with-lease`, no remote-rebase, no
tracking-ref materialization) and bounds each child's policy diff against the
base. `slice_index` is omitted (treated as `0`) for standalone runs and for
slice 0. Fan-in of the slice branches onto the consolidated branch is
ADR-041 / E24.2.

When the child's plan declares a decomposition, the implement prompt narrows
`scope_files` to the child's own slice, matched by the persisted `slice_index`
(not a fragile string match on the issue body). If that slice cannot be
resolved — the `slice_index` is out of range for the parent plan's `sub_plans`,
or the matched sub-plan declares no scope files — both `GET
/v0/stages/{stage_id}/prompt` and `GET /v0/stages/{stage_id}/prompt-render`
**fail closed** with `409 decomposed_scope_unresolved` rather than silently
serving the parent's full scope (which reopened #1669 when campaign-minted
parents fanned out children with a nil cached issue context, #1721). The error
`details` carry `run_id` (the child), `slice_index` (its 0-based index, or `-1`
when the row carries none), and `parent_run_id` (the `decomposed_from` run). A
run whose loaded plan carries no decomposition is unaffected — it keeps the
plan's top-level scope.

### Acceptance-stage containment inputs

On acceptance stages both prompt responses carry two E31.7 (#1535) fields and
one E31.18 (#1569) field, absent on every other stage type:

- `egress_target_hosts` — the acceptance stage's FULL spec-declared egress
  target-host list (the E31.4/#1532 grammar), each a host or host:port. The
  runner feeds it into the ADR-050 egress proxy's allow-list before spawning
  the acceptance agent. Distinct from the prompt TEXT's target-instance line,
  which renders only the first host. Absent when the spec declares no egress
  block — the proxy then admits only model + backend hosts (fail-closed: the
  agent simply cannot reach any target).
- `acceptance_criteria_ids` — the approved plan's
  `verification.acceptance_criteria` id list (the E31.1 join keys) in plan
  order. The runner validates the shipped verdict's `criteria[].id` join keys
  against this set, failing closed on an unknown id. Absent when the approved
  plan carries no criteria.
- `acceptance_expected_head_sha` — the run's merge-candidate identity: the
  newest `head_sha` across the run's reported-head ledger entries
  (`pull_request_opened` / `child_pushed` / `fixup_pushed`, the same ADR-035
  lineage source `fixup_expected_head_sha` resolves from). The runner's
  pre-spawn target-identity gate compares the declared acceptance target's
  `/healthz` `git_sha` against it before spawning the acceptance agent, so
  acceptance validates the merge candidate rather than whatever build answers
  at the declared host (E31.18 / #1569). Absent when backend-side ledger
  resolution fails (empty ledger or read error, WARN-and-omit) — the runner
  then treats the target as unverifiable and warns-and-proceeds rather than
  blocking the stage.

### Fix-up re-dispatch routing

When an implement stage is an operator-triggered implement-review fix-up pass
(`fishhawk_fixup_stage`, #762), the prompt response carries `fixup: true` and
`fixup_branch` — the existing PR branch the fix-up commit must land on. The
runner takes its same-branch `RebaseFromRemote` path (fetch + checkout + pull
--rebase) and pushes onto `fixup_branch`, advancing the open PR's head rather
than opening a new one. `fixup_branch` is derived to match the runner's branch
formula exactly: non-decomposed
`fishhawk/run-<shortID(run_id)>/stage-<shortID(stage_id)>`, decomposed
`fishhawk/run-<shortID(decomposed_from_run_id)>` (shortID = first 8 chars of the
UUID). A backend-side mismatch would re-create the `checkout -b <existing
branch>` already-exists failure (#784). Both fields are absent on a normal
implement dispatch — branch routing falls through to the per-stage branch.

A fix-up dispatch also carries `fixup_expected_head_sha` — the run's recorded
head: the newest `head_sha` across the run's reported-head ledger entries
(`pull_request_opened` / `child_pushed` / `fixup_pushed`), the same audit
source the ADR-035 branch-lineage verifier treats as the run's legitimate
tips. Before invoking the agent the runner establishes its own git base
(fetch + checkout of `fixup_branch`) and fails fast when the fetched tip
differs from this value, so a fix-up never runs against the operator's
incidental checkout (#967). The field is absent when resolution fails
backend-side (or from an older backend); the runner then proceeds with the
checkout only, skipping the comparison.

### Verify gate fields

When the stage's `executor.verify` block is set, both prompt responses carry
`verify_command` (shell command run as the in-band test gate) and
`verify_timeout_seconds` (wall-clock cap; 0 or absent means the runner's
10-minute fallback). They also carry `verify_max_iterations` — the verify-fix
loop budget from `executor.verify.max_iterations`: `0` or absent preserves the
single-shot demote-on-failure gate; `>0` caps the total verify-fix agent
invocations across the stage. The runner's `--verify-cmd` / `--verify-timeout` /
`--verify-max-iterations` flags override the spec-sourced values. All three
fields are omitted when the spec declares no verify block.

### Operator-agent: drive a campaign from an epic

```
POST /v0/campaigns                         → { repo, epic_ref: "issue:1439" }   (201 → Campaign)
GET  /v0/campaigns/{campaign_id}/status    → poll: { campaign, items, rollup, next_action }
   next_action.action == "start_run"       → POST /v0/campaigns/{id}/runs for next_action.issue_ref (E26.2 —
                                              campaign-aware, DAG-gated, links the run; pass runner_kind:local
                                              to drive it through the local loop), then re-poll. The status
                                              re-poll settles each run as it reaches terminal (reconcile-on-read).
   next_action.action == "attention"       → a failed item needs retry/abandon (FAILED wins over eligible)
   next_action.action == "resume"          → the auto-driver paged a human at a gate (E25.7); handle the
                                              gate, then POST /v0/campaigns/{id}/resume to continue
   next_action.action == "attend_human_led" → the only deps-satisfied item(s) are autonomy:low (human-led);
                                              fires only when eligible is empty (start_run wins otherwise).
                                              A human must lead it — do NOT start an agent run; handle it
                                              out-of-band, then re-poll
   next_action.action == "wait"            → items running/blocked; re-poll later
   next_action.action == "complete"        → every item terminal; the campaign is done
```

The create is **runless**: with no run row to carry it, the handler resolves
the repo's GitHub App installation directly (the same path `POST /v0/runs`
uses) so the provider can query the epic's children. **Idempotency-Key is
honored** (E25.13): a retried `POST` with the same `(repo, key)` returns the
existing campaign (200, `idempotent: true`) instead of minting a second one,
backed by an `idempotency_key` column + partial unique index, mirroring `POST
/v0/runs`. `next_action` precedence is
strictly FAILED-wins: a failed item surfaces `attention` regardless of whether
eligible items also exist.

An empty/unconfigured allow-list fails OPEN (any model accepted, byte-identical to today). |
| `agent_review_pending` | 409 | `POST /v0/stages/{id}/approvals` (approve, plan stage only) — ADR-036 / #875: the plan stage's workflow declares `reviewers.agent` > 0, a review was dispatched (`plan_review_started` present), and fewer than the configured agent count of TERMINAL verdicts (`plan_reviewed` / `plan_review_failed` / `plan_review_skipped`) have landed. The lint config disables `operation-2xx-response` (the OAuth redirect
endpoints intentionally have no 2XX response) and `no-server-example.com`
(the localhost server URL is intentional for local-loop testing).

## Where this lives in code

The eventual handler implementations land in `backend/internal/server/`. Each
operation in the OpenAPI document corresponds to one handler function whose
name matches the `operationId`. The router is plain `net/http` `ServeMux`
(per `docs/ARCHITECTURE.md` §3 — Go 1.22+ is required).
