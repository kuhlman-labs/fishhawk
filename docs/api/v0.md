# Fishhawk Backend API — v0

This is the human-readable companion to [`v0.openapi.yaml`](./v0.openapi.yaml).
The OpenAPI document is the source of truth; this file exists to give agents
and humans a faster way to grok the surface than reading 900 lines of YAML.
If the two disagree, the OpenAPI document wins.

## Why `/v0`

`/v0` is the prefix for everything the Web UI, CLI, and runner call. It is
**unstable by design** — fields may be renamed, endpoints may move, request
bodies may grow required fields — until the v1 cut at the end of phase 4
(MVP_SPEC §13). Once `/v1` ships, `/v0` and `/v1` coexist for one minor
version cycle, then `/v0` is retired.

The webhook receiver (`/webhooks/github`) is **unversioned** because GitHub
controls the request shape; the backend adapts as GitHub evolves.

## Auth at a glance — ADR-005 (#69)

Three security schemes for three caller classes; no other auth is supported
in v0.

| Scheme | Used by | Carrier |
|---|---|---|
| `sessionCookie` | Web UI (browser) | `Set-Cookie: fishhawk_session=…; HttpOnly; Secure; SameSite=Lax`; CSRF token via `X-CSRF-Token` matching `__Host-csrf` |
| `bearerToken` | CLI, programmatic | `Authorization: Bearer <opaque>` — server-issued, scope-bound, immediately revocable |
| `githubOIDC` | Runner action only | `Authorization: Bearer <github-oidc-jwt>` — verified against GitHub's JWKS per request, no server state |

Default for all UI/CLI endpoints is `sessionCookie OR bearerToken`. Runner-
facing endpoints override this individually; webhook ingestion uses neither
(GitHub HMAC over the body).

## Conventions

- **JSON everywhere** except `POST /v0/runs/{run_id}/trace`, which uploads
  `application/gzip` bytes and signs them via `X-Fishhawk-Signature`.
- **Cursor pagination** for every list endpoint. Response: `{ items, next_cursor }`.
  `next_cursor` is `null` at the end. Cursors are opaque; clients pass them back
  unmodified.
- **Error envelope** is `{ error: { code, message, details? } }`.
  - `code` is snake_case, stable, machine-readable. Clients switch on it.
  - `message` is human, may change between versions.
  - `details` is structured per-code; the keys ARE part of the contract for
    that code.
- **Timestamps** are RFC 3339, always UTC, always with `Z`.
- **IDs** are UUIDv7 strings; `format: uuid` in the schema.
- **Idempotency** — only `POST /v0/runs` is non-idempotent in v0. The cancel
  endpoint and approval submission are idempotent on equal inputs.

## Resource model

The handlers in `backend/internal/server` map cleanly onto the persistent
state layered up through `backend/internal/run`, `backend/internal/audit`,
`backend/internal/signing`, and `backend/internal/tracestore`.

| Resource | Source of truth | Notes |
|---|---|---|
| `Run` | `runs` table; state machine in `internal/run/transition.go` | All state transitions validated against the table; FOR UPDATE lock per write. `cost_usd_total` + `resolved_model` (#649) are rolled from each stage bundle's signed manifest token counts — recorded once per bundle (raw trace variant only, #678); both always present in the response (0 / empty string for legacy rows) |
| `Stage` | `stages` table | One stage per workflow stage; `executor` is `{kind: agent|human, ref}` |
| `Artifact` | `artifacts` table | Plans (`kind=plan`) validated against `docs/spec/plan-standard-v1.schema.json` |
| `AuditEntry` | `audit_entries` table; chain in `internal/audit/chain.go` | Append-only at three layers: API surface, static-analysis test, DB triggers |
| `SigningKey` | `signing_keys` table; ephemeral private half | One key per run; private half NEVER persisted, returned exactly once |
| `User` / `ApiToken` | `users`, `api_tokens` tables | Tokens stored hashed; plaintext returned exactly once |

## Caller flows

### Web UI: review → approve a stage

```
GET  /v0/auth/me                      → User
GET  /v0/runs?state=running           → recent runs
GET  /v0/runs/{id}                    → run detail
GET  /v0/runs/{id}/stages             → stage list
GET  /v0/stages/{stage_id}/artifacts  → fetch the plan
GET  /v0/artifacts/{plan_id}          → plan content (standard_v1)
POST /v0/stages/{stage_id}/approvals  → { decision: "approve" }
```

`GET /v0/stages/{stage_id}/checks` injects a self-derived
`fishhawk_audit_complete` row for review stages (published to GitHub as a
required Check Run, #229/#231). Its `missing[]` entries each carry a `kind`
the SPA branches on. Most kinds (`plan_missing`, `trace_missing`,
`pr_missing`, `chain_invalid`, `chain_unrecoverable`, `foreign_commit`) are
hard gaps → state `fail`. Two are **pending-flavored** → state `pending`
when they are the only gaps: `head_fetch_failed` (couldn't read the live PR
HEAD) and `review_pending` (#947 — a configured agent implement-review was
dispatched but has not reached a terminal verdict). `review_pending` is the
pre-merge **presence** gate: it holds the required check pending — not
failing — until the review lands, then auto-republishes green. ADR-027's
advisory verdict stays non-blocking; ANY terminal review kind clears it, and
a hard backstop (`ReviewBudget.Cap` × configured agents) clears a reviewer
that died emitting nothing.

### CLI: kick off a run, watch it finish

```
POST /v0/runs                              → { repo, workflow_id, workflow_sha, trigger_source: "cli", workflow_spec: "<YAML>" }
GET  /v0/runs/{run_id}                     → poll until state in {succeeded, failed, cancelled}
GET  /v0/runs/{run_id}/stages              → final stage breakdown
```

For issue-triggered runs the CLI (and MCP server) also ship an inline
`issue_context` — title, body, url, number, and the issue's `comments`
(see the `IssueContext` schema) — captured via `gh issue view` so the
plan agent reads comment-borne refinements without a backend-side
GitHub fetch (#415, #618).

`POST /v0/runs` also accepts an optional `budget_override` (boolean).
When the resolved workflow declares an `enforcement: blocking` periodic
cost budget whose current calendar-period spend has reached `limit_usd`,
a new run is refused with `402 budget_exhausted` (#688 / ADR-030);
`budget_override: true` forces the run past and records a
`run_admitted_budget_override` audit entry. The CLI exposes this as
`fishhawk run start --override-budget` and the MCP `fishhawk_start_run`
tool as `budget_override`. There is no override on the webhook-dispatch
path — a webhook-triggered run that crosses a blocking budget is simply
refused (a `run_rejected_budget` audit entry, no run row). In-flight runs
and CI-retry / decomposition-child continuations are never gated.

### Runner: ship a signed trace

```
POST /v0/runs/{run_id}/signing-key         (githubOIDC)        → { public_key, private_key, ... }
# ... runner executes the agent, produces a trace bundle ...
POST /v0/runs/{run_id}/trace?stage_id=&variant=raw              → 202
   X-Fishhawk-Signature: <hex>                                  body: gzipped JSONL
POST /v0/runs/{run_id}/trace?stage_id=&variant=redacted         → 202
```

The signing key's private half lives only in the runner's process memory and
is discarded after upload. ADR-008 (#72) is the governing decision.

### Version coordination

`GET /healthz` advertises three version-coordination fields alongside `status` and `version`:

| Field | Type | Notes |
|---|---|---|
| `git_sha` | string | Hex commit SHA the backend binary was built from; `"unknown"` in dev builds |
| `min_runner_version` | string | Minimum `fishhawk-runner` version the backend accepts; `"dev"` means no constraint |
| `schemas` | object | Map of schema-name → sha256 hex hash of the embedded JSON schema. Keys: `plan-standard-v1`, `workflow-v0` |

The runner receives `min_runner_version` in the `GET /v0/stages/{stage_id}/prompt` response body. When the field is present and the runner's own version is older, the runner must exit with code 3 (`version_skew`) rather than invoking the agent.

`fishhawk doctor` uses all three fields to surface drift warnings to the operator:
- backend `git_sha` vs local `HEAD` (warns if the local tree is ahead and the binary hasn't been rebuilt)
- runner schema hash vs backend's `plan-standard-v1` hash (warns if runner and backend are built from different schema versions)
- CLI version vs `min_runner_version` (warns if the CLI is too old)

### Scope-bounded implement commit

On implement stages the `GET /v0/stages/{stage_id}/prompt` response carries
`scope_files` — the approved plan's `scope.files`, each an object of `path`
(repo-relative) + `operation` (`create` / `modify` / `delete`). The runner
stages exactly these declared paths at the commit boundary (and when computing
the policy diff) instead of `git add -A`, so stray dirty files — dev `.pid`
artifacts, editor scratch, unrelated local edits — cannot leak into a
Fishhawk-attributed commit (#581). Dirty-but-undeclared paths are excluded
(never staged) and surfaced as a `scope_drift` signal rather than blocking the
commit, matching the spec's flag-only scope-drift treatment. When `scope_files`
is absent (no approved plan, i.e. `plan_missing_for_implement`) the runner
falls back to `git add -A`.

### App bot commit identity

The same prompt response also carries `commit_author_name` and
`commit_author_email` — the GitHub App bot account's git author identity,
resolved backend-side from the App itself (slug via `GET /app`, bot user-id
via `GET /users/<slug>[bot]`, both App-JWT authed and cached for the process
lifetime). The runner sets these as the commit author so App-backed commits
attribute to the App's bot account (`<slug>[bot]`, e.g.
`<bot-user-id>+<slug>[bot]@users.noreply.github.com`) rather than a hardcoded
default (#722). Both fields are absent when the backend can't resolve the
identity (no App JWT wired — dev / CLI / local-runner); the runner then falls
back to its default bot identity (`fishhawk-runner[bot]`).

### Fix-up re-dispatch routing

When an implement stage is an operator-triggered implement-review fix-up pass
(`fishhawk_fixup_stage`, #762), the prompt response carries `fixup: true` and
`fixup_branch` — the existing PR branch the fix-up commit must land on. The
runner takes its same-branch `RebaseFromRemote` path (fetch + checkout + pull
--rebase) and pushes onto `fixup_branch`, advancing the open PR's head rather
than opening a new one. `fixup_branch` is derived to match the runner's branch
formula exactly: non-decomposed
`fishhawk/run-<shortID(run_id)>/stage-<shortID(stage_id)>`, decomposed
`fishhawk/run-<shortID(decomposed_from_run_id)>` (shortID = first 8 chars of the
UUID). A backend-side mismatch would re-create the `checkout -b <existing
branch>` already-exists failure (#784). Both fields are absent on a normal
implement dispatch — branch routing falls through to the per-stage branch.

A fix-up dispatch also carries `fixup_expected_head_sha` — the run's recorded
head: the newest `head_sha` across the run's reported-head ledger entries
(`pull_request_opened` / `child_pushed` / `fixup_pushed`), the same audit
source the ADR-035 branch-lineage verifier treats as the run's legitimate
tips. Before invoking the agent the runner establishes its own git base
(fetch + checkout of `fixup_branch`) and fails fast when the fetched tip
differs from this value, so a fix-up never runs against the operator's
incidental checkout (#967). The field is absent when resolution fails
backend-side (or from an older backend); the runner then proceeds with the
checkout only, skipping the comparison.

### Verify gate fields

When the stage's `executor.verify` block is set, both prompt responses carry
`verify_command` (shell command run as the in-band test gate) and
`verify_timeout_seconds` (wall-clock cap; 0 or absent means the runner's
10-minute fallback). They also carry `verify_max_iterations` — the verify-fix
loop budget from `executor.verify.max_iterations`: `0` or absent preserves the
single-shot demote-on-failure gate; `>0` caps the total verify-fix agent
invocations across the stage. The runner's `--verify-cmd` / `--verify-timeout` /
`--verify-max-iterations` flags override the spec-sourced values. All three
fields are omitted when the spec declares no verify block.

### GitHub: webhook fires, run starts

```
POST /webhooks/github
   X-GitHub-Event: issues, X-GitHub-Delivery: <uuid>, X-Hub-Signature-256: <hmac>
   body: GitHub event JSON
→ backend dedupes by delivery, dispatches a run if the event matches a
  trigger configured in .fishhawk/workflows.yaml
```

## Common error codes

These are the codes the backend currently emits. The list grows as endpoints
are implemented; codes are additive and never repurposed.

| Code | Status | Meaning |
|---|---|---|
| `unauthenticated` | 401 | No valid session, token, or OIDC token |
| `forbidden` | 403 | Authenticated, but not authorized for this resource |
| `csrf_required` | 403 | Session-cookie POST/PUT/PATCH/DELETE missing or with mismatched `X-CSRF-Token` header (E4.6 #152). Bearer-token clients bypass this check. |
| `budget_exhausted` | 402 | `POST /v0/runs` — the resolved workflow declares an `enforcement: blocking` periodic cost budget whose current calendar-period spend has reached `limit_usd` (#688 / ADR-030). No run row is created. `details` includes `workflow_id`, `period`, `limit_usd`, and `spent`. Clear it by raising `limit_usd`, switching the budget to `enforcement: advisory`, or retrying with `budget_override: true`. The same gate runs on the webhook-dispatch path (no override there). In-flight runs and CI-retry / decomposition-child continuations are never gated. |
| `retry_not_applicable` | 422 | `POST /v0/stages/{id}/retry` on a stage whose state or failure category isn't retriable (E8.3 #146). B and D-rejected; a fresh run is normally the right next step. A genuine category-B failure can instead be re-opened with an **audited override** — `{override: true, reason: "..."}` in the request body (#698): the stage re-runs and the policy gate re-evaluates the new diff (the gate is *not* bypassed), writing a distinct `stage_override_retried` audit entry. `reason` is required when `override` is true (`validation_failed` / 400 otherwise). |
| `fixup_not_applicable` | 422 | `POST /v0/stages/{id}/fixup` on a stage that can't take a fix-up pass: not an implement stage; not at the gate (neither parked at `awaiting_approval`, nor `succeeded` with the run's review stage still at `awaiting_approval` — the push_and_open_pr flow, #780); or with no recorded `approve_with_concerns` implement-review concerns to route back (#762). A fresh run or an operator hand-edit is the next step. |
| `fixup_budget_exhausted` | 422 | `POST /v0/stages/{id}/fixup` once the NORMAL bounded pass count (default 1) is spent and no override was requested — enforced by counting prior `stage_fixup_triggered` audit entries for the stage (#762). `details` includes `max_passes` and `used`. One more pass is available via `force_additional_pass: true` (#860); otherwise a fresh run is the next step. |
| `fixup_ceiling_reached` | 422 | `POST /v0/stages/{id}/fixup` once the hard ceiling of 3 total passes is reached — even a `force_additional_pass: true` attempt is refused (the override cannot push past the ceiling), distinct from `fixup_budget_exhausted` (#860). `details` includes `ceiling` and `used`. Merge with a follow-up, or start a fresh run. |
| `cross_run_fixup` | 403 | `POST /v0/stages/{id}/fixup` presented with an MCP run-bound token (subject `mcp:run:<uuid>`) whose run does not match the target stage's `run_id` (#762). MCP tokens may only fix up stages within their own run. |
| `redrive_not_applicable` | 422 | `POST /v0/runs/{id}/redrive` on a run that is not a failed decomposition child: `DecomposedFrom` is null, the run is not in `failed`, or it has no failed implement stage to re-open (#698). |
| `confirmation_required` | 400 | `POST /v0/runs/{id}/reset-branch` without `confirm: true`. The reset force-rewinds the PR head ref and is destructive, so it is never silent/auto (ADR-035 #867). |
| `reset_out_of_scope` | 422 | `POST /v0/runs/{id}/reset-branch` when the foreign commit is an ancestor of (or interleaved with) the run's commits, not strictly on top. A reset can't drop an ancestor — prevention (#861/#865) owns that case (#867). |
| `reset_not_applicable` | 422 | `POST /v0/runs/{id}/reset-branch` when the branch tip is already the last run-authored HEAD; there is no foreign commit on top to drop (#867). |
| `reset_not_determinable` | 422 | `POST /v0/runs/{id}/reset-branch` fail-closed refusal: the run branch's lineage could not be classified with certainty (unresolvable base ref, incomplete ledger, compare error, no identifiable run-authored HEAD), or the lease re-check found a concurrent push. The destructive action refuses rather than force-update on uncertainty (#867). |
| `cross_run_reset` | 403 | `POST /v0/runs/{id}/reset-branch` presented with an MCP run-bound token (subject `mcp:run:<uuid>`) whose run does not match the target run. MCP tokens may only reset their own run's branch (#867). |
| `reset_force_update_failed` | 502 | `POST /v0/runs/{id}/reset-branch` when the GitHub force-update of the PR head ref failed after the reset was authorized and classified (#867). |
| `agent_token_forbidden` | 403 | `POST /v0/runs/{id}/redrive` presented with an MCP/agent subject-bound token (subject `mcp:run:<uuid>`). Re-drive is an operator-only recovery action; an agent may not re-drive any run, including its own (#698). |
| `plan_violates_budget` | 422 | `POST /v0/stages/{id}/approvals` (approve, plan stage only) — the plan's `predicted_runtime_minutes` exceeds the spec-resolved implement-stage timeout and neither a `decomposition.sub_plans` block nor `--override-budget` in the comment is present. `details` includes `predicted_minutes`, `budget_minutes`, and `timeout_source`. Unblock by re-planning with a decomposition block, or re-submitting with `--override-budget` in the comment. |
| `agent_review_pending` | 409 | `POST /v0/stages/{id}/approvals` (approve, plan stage only) — ADR-036 / #875: the plan stage's workflow declares `reviewers.agent` > 0, a review was dispatched (`plan_review_started` present), and fewer than the configured agent count of TERMINAL verdicts (`plan_reviewed` / `plan_review_failed` / `plan_review_skipped`) have landed. The approval is refused before any row is recorded; poll until the review reaches a terminal state, then retry (idempotent). `details` includes `stage_id`, `configured_agents`, and `landed_terminal`. A hard backstop (`ReviewBudget.Cap` × configured agents, from the earliest `plan_review_started`) allows the approval once elapsed even with no terminal verdict, recording a `plan_review_backstop_elapsed` audit entry. |
| `run_not_found` | 404 | `run_id` does not exist |
| `stage_not_found` | 404 | `stage_id` does not exist |
| `artifact_not_found` | 404 | `artifact_id` does not exist |
| `token_not_found` | 404 | `token_id` does not exist |
| `invalid_state_transition` | 409 | Run/stage cannot move to the requested state from its current state |
| `signing_key_already_issued` | 409 | A run may have only one signing key |
| `signature_invalid` | 401 | `X-Fishhawk-Signature` did not verify against the run's public key |
| `webhook_signature_invalid` | 401 | `X-Hub-Signature-256` did not verify |
| `validation_failed` | 400 | Request body or query parameter failed schema validation. `details.field` names the offending field |
| `cursor_invalid` | 400 | The pagination cursor is malformed or stale |
| `service_unavailable` | 503 | The auth middleware could not validate the presented credential because the database was unreachable (connection refused, closed pool) — distinct from a bad credential, which still returns 401 on a healthy DB (#764). Surfaced at the auth boundary on **any** authenticated endpoint, ahead of the per-handler logic. `message` is `database unavailable; retry shortly`; the condition is transient — retry with backoff. An error-classification signal only: no token loses access, and healthy-DB behavior is unchanged. |

## Where this lives in code

The eventual handler implementations land in `backend/internal/server/`. Each
operation in the OpenAPI document corresponds to one handler function whose
name matches the `operationId`. The router is plain `net/http` `ServeMux`
(per `docs/ARCHITECTURE.md` §3 — Go 1.22+ is required).
